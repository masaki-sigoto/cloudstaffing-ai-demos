# セミナー実演ガイド

4本のデモをセミナーで実演するためのチェックリストと進行手順。

> **演者向け：リスク表現ガイド（厳守）**
>
> 実演中・ナレーション・スライドのすべてで以下を守ること。NotebookLM で動画ナレーションを生成する場合の方針でもある。
>
> - ✅ 採用: 「AI が確認作業を支援」「不一致の可能性がある箇所を抽出」「最終確認は担当者が行う」「正確性が必要な部分はロジックで処理」
> - ❌ 避ける: 「AI が完全自動で正確に判定」「勤怠・請求ミスを完全になくす」「AI だけで完結」「人の確認は不要」「自動で確定」
>
> **共通スタンス**: 整形・抽出・診断は AI／ロジックで支援し、**最終判断は担当者**。

> **コマンド実行の前提 CWD**: 本ドキュメント中のコマンドはすべて `REPO_ROOT`（= `ai-demos/`）から実行する想定です。`./launch-*.sh` は `REPO_ROOT` 直叩き、各プロジェクトの `python3 src/main.py ...` は `REPO_ROOT` から `cd <プロジェクト>` で移動してから実行します（以下の手順内コマンドにも `cd` を明示しています）。

> **実演時の AI 利用**: 本デモは **実行時 LLM API 非依存（ルール／ロジック中心の実装）** です。整形・抽出・診断は AI 的な処理として動きますが、内部はルール／真理値表／決定表で確定的に動作します。ナレーションでも「AI 単独で確定」「担当者の確認は不要」といった誇張表現は使用しません。

## 事前準備チェックリスト

### 環境確認
- [ ] Python 3.10 以上が入っている（`python3 --version`）
- [ ] `REPO_ROOT`（= `ai-demos/`）直下で Claude Code・ターミナルを起動できる
- [ ] Chromeなど最新ブラウザが起動できる
- [ ] プロジェクタ/画面共有設定済み（ターミナルとブラウザを切替）

### 推奨: 統合ダッシュボードを起動しておく
セミナー本番は **`./launch-dashboard.sh`** を起動して `http://localhost:8765/` を表示しておくのが基本動線。
ブラウザから 4 本のデモを切り替えで実演できる。

```bash
# REPO_ROOT (= ai-demos/) で実行
./launch-dashboard.sh
```

個別起動でも実演可能（[LAUNCH.md](./LAUNCH.md) 参照）:

```bash
./launch-csv.sh
./launch-rounding.sh
./launch-attendance.sh
./launch-yesno.sh
./launch-all.sh   # メニュー形式
```

### プロジェクト別の事前実行
各プロジェクト初回のみ事前に1回実行しておくと、実演時の待ち時間を減らせる。

```bash
# すべて REPO_ROOT (= ai-demos/) を起点に実行

# 01: out/ に生成物がすでにある状態で実演開始
cd 01_csv-automation && python3 src/main.py convert --input samples/timesheet_202604_haken_a.csv && cd ..

# 02: PyYAML の有無を事前に確認（なければ内蔵パーサで動く）
cd 02_rounding-checker && python3 src/main.py validate --config samples/rules/employee_friendly.yml && cd ..

# 03: generate-samples を一度走らせておく
cd 03_attendance-check && python3 src/main.py generate-samples --count 12 && cd ..

# 04: ブラウザで一度開いて動作確認
open 04_yesno-diagnosis/src/index.html
```

## 実演フロー

### 01 CSV加工の整形支援（5分） — 全企業に効く汎用性の提示
1. **Before**（1分）: 派遣元から届くCSVを画面表示 `cat samples/timesheet_202604_haken_c.csv`

2. **After**（1.5分）: ダッシュボードの「01 CSV加工」を実行（または `/csv-convert samples/timesheet_202604_haken_c.csv`）

   - 入出力件数・要確認件数が可視化
   - 演者コメント例: 「整形は自動、**最終確認は担当者**が行います」
3. **テンプレート再利用**（1分）: `/csv-save-template` → 翌月CSVに適用
   - 「同じ派遣元からの来月のCSVは、設定ゼロで整形支援」
4. **締め**（1分）: 派遣元メリット（加工工数減）/派遣先メリット（差戻し減）に加えて、勤怠・スタッフ情報・契約データなど **派遣管理運用のあらゆる CSV に応用可能** を強調

### 02 端数処理（7分） — 説明責任の可視化
1. **導入**（1分）: 「端数処理は派遣元/派遣先で揃わないまま、請求ズレの温床」

2. **基準計算**（1分）: `python3 src/main.py simulate --config samples/rules/strict_1min.yml --punch 9:03,18:07 --hourly 1800`

## 質疑で想定される質問と回答の糸口

- **「AI で確定判断するのか？」** → **いいえ**。AI は確認作業の支援・要確認箇所の抽出・パターン整理を行うだけで、**最終判断は担当者** が行います。02 端数処理のような正確性が必要な計算はロジック処理（真理値表）で実装しており、AI の自由判断は入れていません。
- **「学習データはどこから？」** → 今回は AI API 不使用のルールベース実装。同義語辞書・スコアリング・決定表で「AI らしさ」を演出。実運用では LLM 統合も可能だが、まず現状の運用課題に効く粒度から。
- **「セキュリティは？」** → データはローカル完結、外部通信ゼロ、PII マスキング実装済。04 はブラウザ完結（localStorage もデフォルト OFF）、外部依存ゼロでオフライン動作。
- **「導入コストは？」** → 本日見せたのはモック（数時間レベル）。本番化は業務の擦り合わせが本体で、技術コストは軽め。
- **「派遣法の解釈は？」** → 本デモの診断は整備状況把握の補助ツール。法的判断は社労士等の専門家へ。

## トラブルシュート

### Python が `python3: command not found`
- macOS 標準は `python3`。Homebrew: `brew install python@3.11`

### PyYAML が無い警告（02）
- 内蔵 mini-parser で動くので続行可。気になれば `pip install --user pyyaml`

### 04 の HTML が開かない
- `open 04_yesno-diagnosis/src/index.html` または Finder でダブルクリック
- ブラウザが警告を出した場合は「開く」を選択

### 実行が遅い・時間がかかる
- 01 は 10,000 行で 5 秒以内を目標。サンプルは 5〜10 行なので即時
- 03 は `generate-samples --count 12` で 266 行、処理時間 0.02〜0.05 秒
