# Codex レビュー履歴: 要件定義書

## ラウンド1 (2026-04-23)

### 総合評価
**C**
デモとしての見栄え・流れは良い一方、**法対応ロジックと要件の厳密性が不足**しており、現状のままでは「それっぽいが危ない」設計。
特に **派遣法関連の判定条件・出力文言・セキュリティ記述** に致命的な誤解リスクあり。

### Critical 指摘
- **C1. 法令対応の誤解を招く記述**（§11 パターンA「抵触日管理OFF可」「3年超案件はマニュアル管理」, §10 Q12）
  - 修正方針: 法令に関わる断定を撤回し、「法令判定ではなく運用設計の補助」であることを明記。抵触日は原則トラッキング前提に変更し、例外条件は「要確認」に分離。
- **C2. 判定ロジックが実装不能レベルで未定義**（§10.3 3軸定義のみ、5パターンへの写像ルールなし）
  - 修正方針: `回答→軸スコア→パターン` の決定表（全条件優先順位付き）を追加。タイブレークと「該当なし時のフォールバック」を必須定義。
- **C3. セキュリティ要件の虚偽に近い断定**（§5「localStorageのみ。顧客情報漏洩リスクなし」）
  - 修正方針: 「漏洩リスクなし」を削除。共有PC/ブラウザ残存/印刷PDF持ち出しリスクを明示し、`保存しないモード` `保存クリア` `PII入力最小化` をMUST化。
- **C4. 法対応出力と質問設計の不整合**（§4 S-06, §11 パターンEで36協定監視を出すが、36協定を問う質問がない）
  - 修正方針: 36協定・同一労働同一賃金（方式未確定含む）・抵触日運用体制を直接問う質問を追加し、未回答時は「要ヒアリング」表示にする。
- **C5. 要件矛盾**（§5 ファイルサイズ500KB目標 vs §7 制約1MB以下）
  - 修正方針: どちらかに統一し、`目標`と`上限`を明確化（例: 目標500KB / 上限1MB）。

### Major 指摘
- **M1. 成功基準が主観的で検証不能**（§8「歓声」「ざわつき」「8割以上」）
  - 修正方針: 計測可能KPIに置換（完走率、平均所要時間、再利用希望率、誤判定指摘率）。
- **M2. 5分デモ設計がギリギリで破綻しやすい**（§6.1 11問+説明+印刷）
  - 修正方針: デモ用ショートルート（7〜8問）を別途定義し、フル版は導入支援用に分離。
- **M3. 3者整理が弱く、派遣先視点が勤怠承認に偏る**（全体）
  - 修正方針: 派遣元/派遣先/自社SaaS管理者の責務表を1ページ追加。質問にも派遣先側制約を反映。
- **M4. `file://`運用での機能互換リスク**（§4 S-04 クリップボード、§4 S-03 localStorage継続）
  - 修正方針: ブラウザ別フォールバック（コピー失敗時手動選択UI、保存不可時警告）を要件に明記。
- **M5. Q14の業務前提が不自然**（「派遣元社員が派遣先でも役職を持つ」）
  - 修正方針: 実務で頻出する兼務（拠点兼任、営業兼管理）に置換し、希少ケースは自由メモに逃がす。

### Minor 指摘
- **m1. MUST/SHOULD文体の混在**（「尚良」「目標」「OK」表現が散在）
  - 修正方針: RFC風に統一（MUST/SHOULD/MAY + 測定条件）。
- **m2. Q10/Q11の重複で保守性低下**（§10.2）
  - 修正方針: 質問IDは1つに統合し、分岐文脈は補足テキストで表現。
- **m3. 印刷1〜2枚要件と出力項目量が衝突しやすい**（§4 M-07, §11.1）
  - 修正方針: 印刷テンプレートを「要約版/詳細版」の2種に分ける。

### 良かった点
- 単一HTML・オフライン完結という制約が明確で、セミナー実演に強い。
- ユーザー像と利用シーン（営業/導入支援）が具体的で、運用導線が描けている。
- 質問ツリーとパターン出力の骨格は十分あり、モックとして短期実装可能。

### このラウンドでの対応
- **C1（法令文言）** → §11パターンA/Eの「抵触日管理OFF可」等の断定を削除し、「法令判定ではなく運用設計の補助」免責を§1.5・§7.1に明記。抵触日は原則トラッキング前提に統一し、例外は「要ヒアリング」扱いに変更。
- **C2（判定ロジック）** → §10.3に「回答→軸スコア→パターン決定表」を追加。タイブレーク優先順（法対応軸＞拠点軸＞承認軸）と「該当なし時は最近傍パターン＋要ヒアリング表示」のフォールバック規則を明文化。
- **C3（セキュリティ）** → §5セキュリティ行の「漏洩リスクなし」を削除。共有PC/印刷物持ち出しリスクを明示し、MUSTに M-11（保存しないモード）・M-12（保存クリアボタン）・M-13（PII入力任意化・デフォルトOFF）を追加。
- **C4（36協定等の質問追加）** → Q14を36協定運用質問に置換、新Q16（同一労働同一賃金の方式確定状況）、新Q17（抵触日管理の体制）を追加。未回答/不明は「要ヒアリング」タグで結果に表示するルールを§10.3に追記。
- **C5（サイズ要件矛盾）** → §5性能「目標300KB」、§7制約「上限1MB」と用語統一。
- **M1（成功基準）** → §8を計測可能KPI（完走率・平均所要時間・再利用希望率・誤判定指摘率）に置換。
- **M2（デモショート版）** → §6.1に「ショート版（7問）」を追加し、フル版は§6.2の導入支援用に明確化。
- **M3（3者責務）** → §3.3として派遣元/派遣先/SaaS管理者の責務表を追加。
- **M4（file://互換）** → §5の互換性行に「クリップボード不可時の手動選択UI」「localStorage不可時の警告表示」をSHOULDとして追記。S-03/S-04の要件欄にもフォールバック明記。
- **M5（Q14不自然）** → 兼務例を「拠点兼任・営業兼管理」に差し替え、希少ケースは§4 S-07メモ欄に委譲する旨を補足説明に追記。
- **m1（文体統一）** → 「尚良」「OK」「目標」などを MUST/SHOULD/MAY 表記に寄せて修正（主要箇所のみ、軽微な口語は残置）。
- **m2（Q10/Q11重複）** → Q11をQ10に統合し、分岐文脈は補足列で表現。
- **m3（印刷版分け）** → M-07を「要約版（1枚）/詳細版（2〜3枚）」の2種に分けて記述。

## ラウンド3 (2026-04-23) ※最終

### 総合評価
**B**（R1 の C → R2 の B+ → R3 の B）
R1/R2 Critical はすべて反映済み。ただしパターンE の意味が §10.3 系（未整備寄りで発火）と §11（整備済み寄りの説明）で逆転していた **重大な内部矛盾** が1件残留しており、この単一論点によって B 止まり。判定式の括弧明示・KPI起点の共通化・Q16/Q17 の逃げ道追加・プライバシー既定値の明文化など、機械実装可能性と運用安全性の観点で詰めの甘さが散見。単一HTML・オフライン・法対応免責の大枠は一貫して維持されており、R3 での Critical 解消後は **セミナーデモ用モックとして十分実用レベル**。

### Critical 指摘
- [x] **C-R3-1. パターンEの意味がセクション間で矛盾**
  - §10.3.1/§10.3.2 では E は「未整備寄りで発火」（Q16/Q17/Q6 が未整備の時に C=3 が立つ）なのに、§11 パターンE は「整備済み・法対応深化」の説明になっている。同一回答で逆の結果説明を出し得る致命的矛盾。
  - 修正方針: E を **「法対応要整備型」** に一本化し、§10.3.1 の C=3 定義、§10.3.2 優先度1の条件式、§11 パターンE の該当条件/推奨設定/注意点をすべて「未整備要素を含むケース向け」の表現に統一。

### Major 指摘
- [x] **M-R3-1. 判定条件の論理演算子優先順位が曖昧**
  - §10.3.2 優先度3「L=3 かつ Q5 NO または Q8 YES」は AND/OR の結合順が曖昧。
  - 修正方針: 全優先度を `(L = 3) AND ((Q5 = NO) OR (Q8 = YES))` 形式の括弧明示記法に書き換え。
- [x] **M-R3-2. KPI計測起点がショート/フルで不整合**
  - §8「Q1表示時刻起点」 vs §6.1/§10.3.5（ショート版はQ2開始）で食い違い。
  - 修正方針: 「最初に表示した質問の表示時刻」を起点に統一し、ショート版=Q2、フル版=Q1 と明記。ショート/フル共通の定義とする。
- [x] **M-R3-3. Q16/Q17 に「わからない」の逃げ道がない**
  - 3択のうち「わからない」相当が未整備のみで、回答困難時の回答不能リスクあり。
  - 修正方針: Q16/Q17 ともに「わからない」を選択肢に追加し、未整備と同等扱い＋「要ヒアリング」タグ付与ルールに明記。§10.3.4 付与ルールにも追記。
- [x] **M-R3-4. ショート版の法対応軸Cのデフォルト規則が曖昧**
  - §10.3.5「C=2（Q12 YESなら…）」で Q12=NO や わからない 時の扱いが不明。
  - 修正方針: `Q12=YES→C=2 / NO→C=1 / わからない→C=2+要ヒアリング` を明文化。併せて、ショート版では Q16/Q17 が未回答のため C=3（パターンE）は発火しないことを明記。承認Aの Q7 未回答時のデフォルト（Q3 回答から初期値算出）も併せて明文化。
- [x] **M-R3-5. プライバシー上の安全既定値が未定義**
  - M-11/M-13/§5 で「保存しないモード」はあるが初期選択が不明。
  - 修正方針: M-11 を「初期選択を『保存しない』に固定、保存は明示オプトイン」に変更。§5 セキュリティ行にもプライバシー既定値を明記。

### Minor 指摘
- [x] **m-R3-1. 免責文の配置要件が曖昧**
  - M-14「結果カード直下または画面ヘッダー」だと結果カードのみの配置で常時表示にならないケースあり。
  - 修正方針: 「全画面共通ヘッダー（固定表示）」に一本化。§1.6 免責記述も整合させる。
- [x] **m-R3-2. 付録の3年ルール説明が簡略化しすぎ**
  - 事業所単位と個人単位（組織単位）の違いが読み取れず誤解余地あり。
  - 修正方針: 事業所単位（意見聴取で延長可）と個人単位（原則延長不可）の違いを1文で追記。
- [x] **m-R3-3. §11.1 の優先度番号の画面表示はデモでノイズ**
  - 修正方針: 画面上は非表示、詳細版印刷のみ記載に変更。

### 良かった点（Codex R3 より引用要約）
- 単一HTML・オフライン・外部依存ゼロの制約が要件全体で一貫している。
- ショート版7問とフル版の二段構えは、5〜10分デモ設計として現実的。
- 「法令判定ではない」免責と、`localStorage` 残留リスクまで書いている点は実務的。
- 派遣元/派遣先/SaaS管理者の責務分離が明確で、導入会話に使いやすい。

### このラウンドでの対応
- **C-R3-1（パターンE矛盾）** → §10.3.1 法対応軸Cの定義を「C=3 は法対応要整備を意味する」に明記し、Q16「未整備」 OR Q17「未整備」 OR Q6 NO の単一条件でC=3が立つよう一本化。§10.3.2 優先度1を `(C = 3) AND ((Q16 = 未整備) OR (Q17 = 未整備) OR (Q17 = 一部のみ) OR (Q6 = NO))` に修正。§11 パターンEを「派遣法強化型」→「**法対応要整備型**」へ改名し、該当条件・想定企業・推奨設定・注意点をすべて "体制整備が必要なケース" 向けに書き換え。§10.3.3 副次マッチ条件の例文も改名に合わせて修正。
- **M-R3-1（論理演算子）** → §10.3.2 決定表の優先度1〜5 をすべて `( ) AND/OR ( )` 形式の括弧明示記法に統一。
- **M-R3-2（KPI起点）** → §8 平均所要時間の計測方法を「最初に表示した質問（ショート版=Q2、フル版=Q1）の表示時刻を起点、ショート/フル共通定義」に変更。
- **M-R3-3（Q16/Q17の逃げ道）** → §10.1 ツリー・§10.2 質問一覧・§10.3.4 タグ付与ルールに「わからない」を追加。未整備と同等扱い＋要ヒアリングタグを明記。
- **M-R3-4（ショート版C規則）** → §10.3.5 の法対応C/承認Aのデフォルト規則を箇条書きで明文化。Q12 の YES/NO/わからない 3分岐、C=3 はショート版では発火しないこと、Q7 未回答時の A 初期値算出ルールを追加。
- **M-R3-5（プライバシー既定値）** → M-11 を「保存しないモード（初期選択固定）」に改題し、起動時モーダルで「保存しない」を初期選択・保存は明示オプトインと明記。§5 セキュリティ行にもプライバシー既定値を追記。
- **m-R3-1（免責配置）** → M-14 を「全画面共通ヘッダー（固定表示）」に一本化。スタート/質問/結果/印刷プレビューすべてで表示される旨を明記。§1.6 の記述も整合させる。

## 最終残論点

R3 Critical/Major/Minor はすべて対応済み。承認可能状態と判断。以下は **本要件定義書のスコープ外または実装判断に委ねる残論点** として明示的に記録する。

1. **パターンCとEの境界の業務的自然さ**（R2 M-R2-4 見送り継続）
   - C（多拠点×拡張）と E（法対応要整備）は条件分離済みだが、推奨設定の一部は重複する。デモ用モックとしては許容範囲だが、実運用時は「主+副マッチ条件」併記（§10.3.3）で補完する想定。
2. **ショート版 Q6 の配置順の実務妥当性**（R2 M-R2-2 見送り継続）
   - 「代理承認は法対応確認後の補助情報」という設計意図は §10.3.5 に明記済み。デモ運用で違和感があれば R4 相当の実地検証で順序変更を再検討。
3. **§11.1 結果カードUI項目10個の画面収まり**（R2 m-R2-2 見送り継続）
   - CSS/レイアウト実装時の調整事項。要約版印刷1枚の制約に合わせて項目の折り畳み/詳細表示の出し分けを実装判断で決定。
4. **フォールバック（優先度6）の最近傍パターン算出アルゴリズムの具体定義**
   - 「スコア距離最小」の距離関数（ユークリッド/マンハッタン/重み付き）は本要件定義書では未定義。単一HTML・最小実装前提のもと、実装時に最も簡易なマンハッタン距離等で落とし込む想定。要件粒度を超えるため残論点として記録。
5. **markdown lint 警告（MD022/MD032）の残置**
   - 既存ファイル全体に継続的に発生している軽微な整形警告で、要件内容には影響しない。体裁整備は本要件承認後の別作業として扱う。

**承認判断**: R3 Critical（パターンE矛盾）解消により **内部整合性が担保**、Major 5件も機械実装可能な記法・運用安全既定値まで到達。単一HTML・オフライン・派遣法免責の設計原則は一貫して維持。**セミナーデモ用モックの要件定義書として承認可能状態** と判断する。
