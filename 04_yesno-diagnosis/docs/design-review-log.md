# Codex レビュー履歴: システム設計書

## ラウンド1 (2026-04-23)

### 総合評価
**B**
要件の主要ラインは広く拾えており、単一HTML制約下での分割設計も妥当。
一方で、**プライバシー既定値の実効性**と**要件反映の抜け（4択型・印刷詳細）**に実装事故リスク。Phase4前に仕様の曖昧点を潰せばデモ成立性は高い。

### Critical 指摘
- **CD-R1-1. M-11「保存しない」既定が設計上で未閉塞**（§9.1, §3.1, §4.3）
  - `save()` のガードはあるが `load()` 実行条件が未定義。起動時に過去データを自動復元すると共有PCで即漏えい。
  - 修正方針: 初期状態では `Persist.load()` 禁止、ユーザーが「保存する」＋「続きから再開」を明示選択した場合のみ復元する状態遷移を明記。
- **CD-R1-2. 実データ投入禁止の設計担保が不足**（§9.3, §5.4）
  - 任意入力＋注意書きのみで実データ入力を防止できない。
  - 修正方針: デモモードでPII入力欄をデフォルト非表示、表示時もマスキング／印刷時匿名化を標準化。`localStorage` 保存対象から PII を除外する方針を追加。

### Major 指摘
- **MD-R1-1. 質問スキーマが要件と不整合**（§5.1）
  - `type` が `YN|3CHOICE` のみで、Q16/Q17 の4択を型で表現できない。
  - 修正方針: `choiceCount` または `type: 'YN'|'MULTI'` に統一し、選択肢配列長で処理。
- **MD-R1-2. 進捗バー母数の定義不足**（§6.1, §7.1）
  - 分岐型フル版で「全質問数」をどう算出するか未記載。M-02の表示一貫性が崩れる。
  - 修正方針: フル版は「想定ルート長」を動的再計算するか、「回答済/到達予定」の定義を明文化。
- **MD-R1-3. M-07詳細版の内容定義が弱い**（§6.1, §5.2）
  - 「回答一覧＋設定項目チェックリスト」をどのデータからどう生成するか未固定。
  - 修正方針: `printDetailModel` を定義し、必須列（質問ID/回答/理由/チェック項目）を仕様化。
- **MD-R1-4. 要ヒアリングの色仕様が衝突**（§10.2 vs §8.2）
  - 要件は黄色タグだが、設計はフォールバック時に赤バッジ。
  - 修正方針: 「要ヒアリング」は全ケース黄色で統一し、重大度は文言で区別。
- **MD-R1-5. 観測可能性が弱い**（§10.1）
  - 永続ログなしはよいが、将来検証用のセッション監査情報が残らない。
  - 修正方針: 画面内で「診断サマリJSONをコピー」機能を追加し、外部送信なしで追跡可能性を確保。

### Minor 指摘
- **mD-R1-1. 画面定義の用語不整合**（§9.2）: 「設定画面」に保存クリア配置とあるが、画面体系に設定画面がない。用語を統一。
- **mD-R1-2. `answers` 値がコード値と日本語混在**（§4.2）: 比較ロジックのバグ温床。内部コード値に統一し、表示ラベルは別管理に分離。
- **mD-R1-3. `position: sticky` 記述が印刷常時表示要件と誤読されやすい**（§9.4）: 印刷時は別実装であることを明記して誤実装防止。

### 良かった点
- 要件§10.3の優先度ロジックを `Judge` へ落とし込む構成が明快。
- ショート版7問ルートとデモシナリオAの整合が高い。
- 単一HTML制約下でも `State/Flow/Judge/Render/Persist` 分離で保守性を確保している。

### このラウンドでの対応
- **CD-R1-1（保存モード load 閉塞）** → §9.1 に `Persist.load()` の実行条件を明記。初期状態では `load()` 禁止。スプラッシュで「保存する」選択かつ「続きから再開」を明示選択した場合のみ復元する状態遷移を追記。§3.1 Persist の公開IF説明にも「明示同意前は load しない」ガードを追加。
- **CD-R1-2（PII 設計担保）** → §9.3 を強化。デモモードで PII 入力欄をデフォルト非表示（「企業名等を入力する」トグル）、表示時はマスキング、印刷時匿名化オプション（デフォルトON）を標準化。§5.4 の localStorage 保存対象から `pii` を除外する方針を明記。
- **MD-R1-1（4択スキーマ）** → §5.1 の `type` を `'YN' | 'MULTI'` に統一し、`choices` 配列長で 2/3/4 を判別する旨に変更。Q16/Q17 の4択もこれで表現可能。
- **MD-R1-2（進捗母数）** → §6.1 進捗バーと §7.1 Flow に「想定ルート長（到達予定数）」の定義を追加。フル版は現在の回答を元に到達予定の未回答質問数を動的再計算、ショート版は固定7。
- **MD-R1-3（印刷詳細モデル）** → §5 に `printDetailModel`（質問ID／質問文／回答ラベル／採点への寄与／該当した決定表ルール／チェック項目）を追加。§6.1 詳細版の生成元として参照。
- **MD-R1-4（要ヒアリング色統一）** → §8.2/§10.2 の色仕様を「黄色統一」に修正。フォールバック時は赤バッジではなく「確定度: 中」と文言で区別する運用に変更。
- **MD-R1-5（診断サマリJSONコピー）** → §10.1 に「診断サマリJSONをコピー」機能（外部送信なし、画面内のクリップボード API 利用）を追加。
- **mD-R1-1（設定画面用語）** → §9.2 の「設定画面」を削除し、保存クリアは結果画面と画面右上の2箇所に統一。
- **mD-R1-2（answers 内部値統一）** → §4.2 / §5.3 の例示値を全て英字コード（例: `KYOTEI`, `KINTOU`, `PART`, `UNKNOWN`）に統一。日本語は表示ラベル専用と明記。
- **mD-R1-3（sticky 誤読防止）** → §9.4 に「印刷時は `position:sticky` ではなく `@media print` で各ページヘッダーに別実装として印字」の注記を追加。

### 見送り
- なし（Critical 2件／Major 5件／Minor 3件すべて反映。いずれも単一HTML／オフライン完結／最小実装の絶対条件と両立可能な範囲で処理）。

  - `PATTERNS.condition` と `PATTERN_RULES` が並存し、将来差分事故の温床。
  - 修正方針: `PATTERNS` は表示情報のみ、判定は `PATTERN_RULES` 単一ソースへ集約。起動時バリデーション（`AppCore.validate()`）で `patternId` 参照整合を確認。

- **MD-R2-3. C軸ルールの「通常／ショート例外」分断**（§7.2/§7.3）
  - 修正方針: `calcCFull` / `calcCShort` を関数分離。`calcAFull` / `calcAShort` も同様に分離。モード分岐は `evaluate` 冒頭のみに閉じ、`PATTERN_RULES` はモード非依存に維持。
- **MD-R2-4. 戻る操作の境界条件未定義（M-03）**（§7.1）
  - 修正方針: `Flow.canGoBack()` を追加、`history.length===0` で `back()` は `{noop}` 返却、UI上は戻るボタン disabled＋キーボード `←` も no-op。

## ラウンド3 (2026-04-23) ※最終

### 総合評価
**B（条件付きで次フェーズ進行可）**
R1→R2→R3 の推移は **B → B → B（最終）**。R1 はプライバシー既定値と要件反映の抜け、R2 は二重定義・責務境界、R3 は「PII抑止の最後の穴（salesMemo）」「IF仕様の単一ソース化」「実装者ブレ要因の明文化」が論点。R3 指摘は設計の穴埋めに近く、絶対条件（単一HTML／外部依存ゼロ／オフライン）への影響はない。全件反映により **実用十分** と判断し、R3 を最終ラウンドとして切り上げる。

### Critical 指摘
- **CD-R3-1. PII抑止の抜け漏れ（`salesMemo`）**（§5.2 printDetailModel / §9.3.1）
  - 匿名化トグルONでも `salesMemo` が平文印刷され得る不整合。
  - 修正方針: 匿名化トグルの適用範囲を `companyName` / 担当者名 / `salesMemo` まで拡張。デフォルトは salesMemo 非印字、明示的な「営業メモを印字する」サブトグルONのときのみ詳細版最終ブロックに印字。

### Major 指摘
- **MD-R3-1. IF定義の不整合**（§3.1 / §4.1 / §4.2 / §4.3）
  - §4.3 に `Flow.complete()` が登場するが §3.1 公開IFに存在しない。`result.reasonHighlights` も §4.1 と §4.2 で差分あり。
  - 修正方針: `Flow.complete()` 表記を `Flow.next()` 戻り値 `{done:true}` ＋ AppCore 側ガード表記へ修正。`result` スキーマを §4.1 / §4.2 で完全一致化し、`reasonHighlights` の生成元と構造を §4.2 に明記。
- **MD-R3-2. `predicateText` 生成元未定義**（§5.2.2 / §5.2 printDetailModel）
  - `PATTERN_RULES.predicate` が関数のみで、詳細印刷の説明文生成規則がない。
  - 修正方針: `PATTERN_RULES` に `ruleLabel`（必須・自然文）を追加。印刷は関数を文字列化せず `ruleLabel` を出力。起動時バリデーションで `ruleLabel` 必須チェックを追加。
- **MD-R3-3. 進捗母数推定（`estimateRemaining`）の実装仕様不足**（§7.1.1）
  - 未回答分岐の扱いが曖昧で実装者ごとに値が揺れる。
  - 修正方針: 未回答時は QUESTIONS 側の `branch(state)` が返す既定遷移に従う（`estimateRemaining` 側で仮定しない）。ループ検出 `MAX_DEPTH = QUESTIONS.length`。テストケース T1/T2/T3 を設計書に添付。
- **MD-R3-4. 保存モード切替時のデータ保全が弱い**（§9.1）
  - 「切替後から書き込み開始」仕様だと、切替直後離脱時に復元不能。
  - 修正方針: `none → local` 切替時は AppCore が `Persist.save(snapshot)` を**即時1回実行**。逆方向 `local → none` 切替時は `Persist.clear()` 実行を追加。

### Minor 指摘
- **mD-R3-1. 「純ロジック」表現の厳密性不足**（§3.1 / §7.1）
  - 修正方針: Flow は State を書き換えるため「DOM非依存ロジック層」に文言修正。
- **mD-R3-2. 将来拡張 Y-03 の先行記述**（§10.3）
  - 修正方針: 「Judge が全パターン距離を内部計算済み」は現行仕様に未接続のため「拡張時に追加予定」に下げ、現行実装責務から外す。

## 最終残論点

R3 を最終ラウンドとして切り上げるにあたり、**設計書としては実用十分**と判断した。以下は Phase 4（実装）以降で扱う課題として明記する。
