# 要件定義書: 運用整備状況 簡易診断アプリ（旧: YES/NO 組織登録診断アプリ）

> **本プロジェクトのポジション**: 4本のAI活用デモの中で唯一の **"特別枠"**。他3本がバックオフィス業務（CSV取込、端数計算、勤怠チェック）の効率化を狙うのに対し、本プロジェクトは **クラウドスタッフィング導入前ヒアリング／派遣管理運用の整備状況を YES/NO で素早く可視化する簡易診断ツール** であり、**営業フェーズ〜導入支援フェーズの初動設計** に直接効く。セミナーで最もビジュアル映えする目玉枠として、チーム内発信（営業・導入支援・運用）の共通フォーマットとしても機能することを狙う。
>
> **テーマの位置付け**: 「組織登録診断」という抽象テーマではなく、クラウドスタッフィング（派遣管理SaaS）の業務文脈に沿った **「法対応・業務整備・データ運用成熟度」の簡易診断** として定義し直す。具体的には **発注 / 契約 / 案件管理 / 勤怠管理 / 請求 / 評価** の整備状況、および **派遣元・派遣先・スタッフ** 3者の運用責任分担を押さえる。

---

## 1. プロジェクト概要

### 1.1 一言サマリ
クラウドスタッフィング（派遣管理SaaS）の **導入前ヒアリング** および **派遣管理運用の整備状況（発注／契約／案件管理／勤怠管理／請求／評価、および法対応・データ運用）** を、YES/NO 形式の質問に答えるだけで診断し、派遣会社ごとの **運用整備パターン** と **初期設定・整備の優先タスク案** を自動提示する簡易診断ツール。

### 1.2 プロダクト名（仮）
**「運用整備ナビゲーター」** （セミナー資料上の呼称。旧: 組織設定ナビゲーター）
内部呼称: `04_yesno-diagnosis`

### 1.3 ターゲット（二段構え）
- **一次ターゲット**: クラウドスタッフィング（以下 CS）の **営業担当者**
  - 商談時に顧客画面の前で「ちょっと5分だけ質問させてください」と切り出し、**導入前ヒアリング補助** として診断を回し、提案の初動を作る
- **二次ターゲット**: CS の **導入支援担当者（カスタマーサクセス）**
  - キックオフミーティング時のヒアリングシートとして、**派遣管理運用（契約／案件／勤怠／請求／承認フロー／法対応／データ運用）** の整備状況を網羅的に引き出す
- **間接ターゲット**: 派遣元企業の **情報システム担当 / 運用責任者 / 導入検討担当**
  - 自社の派遣管理運用の整備段階を整理し、SaaS側に「何を伝えればいいか」「何を先に整えるべきか」を明確化する

### 1.4 核となる価値提案（KEY MESSAGE）
> **営業と導入支援、両方に効く特別枠 — 質問に答えるだけで「導入前ヒアリング」と「派遣管理運用の整備状況の把握」の初動が決まる。**

### 1.5 成果物の形
**単一HTMLファイル 1本**（`index.html`）。JavaScript / CSS をすべてインラインで内包し、外部ライブラリ・CDN依存ゼロ。ダブルクリックでブラウザが開き、オフラインで完結。セミナー会場のネット事情に左右されない。

### 1.6 位置づけに関する免責（重要）
本アプリは **法令判定ツールではなく、導入時の運用設計ヒアリングを補助するモック** である。
- 派遣法・労基法等の法令適合性判定は行わない。画面上に表示される推奨設定・法対応フラグは **あくまで「要確認項目の洗い出し」** であり、最終判断は顧客側の法務/労務専門家によるレビューを前提とする。
- **全画面共通ヘッダー（固定表示）** に常時 **「本ツールは診断補助であり法令判定ではありません」** の注記を MUST で表示する（M-14）。

## 2. 背景と解決する課題

### 2.1 なぜ「派遣管理運用の整備状況」の把握が、導入前フェーズで最もつまずくのか

派遣業界特有の構造上、派遣管理SaaS の導入可否・設定方針は **「組織マスタを登録できるか」だけでは決まらない**。**派遣元 / 派遣先 / スタッフ** 3者間で発生する **発注 → 契約 → 案件管理 → 勤怠管理 → 請求 → 評価** の業務フロー全体と、**法対応・データ運用成熟度** が絡むため、導入前ヒアリングで押さえるべき論点が多重化する：

| # | つまずきポイント | 業務領域 | 具体例 |
|---|---|---|---|
| 1 | **承認階層の複雑さ** | 契約・勤怠・請求の承認フロー | 派遣元の営業所長 → 本社管理部 → 経理、かつ派遣先からの勤怠承認も別ルート。誰が何を承認するかを画面上で定義できず放置される |
| 2 | **案件と組織の紐付け** | 案件管理 | 「営業1課が受注した案件を、管理2課のスタッフが担当する」ような横串案件がSaaS上で表現できず、評価・請求データも按分できない |
| 3 | **権限設定の属人性** | データ運用 | 「営業所長は自分の所の売上だけ見える」「本社経理は全拠点見える」といった権限設計を、導入担当者が独自判断で設定 → 後で齟齬 |
| 4 | **拠点 × 職種の二軸管理** | 案件・請求 | 「東京オフィス・事務職」「大阪オフィス・製造職」のようにマトリクスで管理したいが、システム上は単一ツリーしか切れないケース |
| 5 | **派遣法対応項目の整備漏れ** | 法対応 | 抵触日管理、3年ルール、同一労働同一賃金、36協定の監視対象範囲 … 派遣管理運用に紐付く法対応フラグ・体制が初期整備段階で漏れる |
| 6 | **兼務・兼任の扱い** | 契約・勤怠 | 1人が複数部署に所属するケース、派遣元社員が派遣先でも役職を持つケースの設定方法がブレる |
| 7 | **派遣先の情報粒度** | 発注・契約 | 派遣先企業 → 事業所 → 就業場所 のどこまで分解して登録すべきか、派遣法上の「事業所単位/個人単位の期間制限」に直結するが判断がつかない |
| 8 | **勤怠・請求・評価データの整合性** | 勤怠・請求・評価 | 勤怠の締め単位、請求の按分単位、評価の粒度がバラバラで、同じ案件の数字が突き合わない（データ運用成熟度の問題） |

### 2.2 現状の運用課題（クラウドスタッフィング営業・導入支援の現場より）
- 導入支援担当が **毎回ゼロから** ヒアリングしている（**初回ヒアリングの標準化が未達**、属人化）
- 営業段階で「御社には●●パターンで入れます」と即答できず、持ち帰り案件になる
- 初期設定で妥協 → 数ヶ月後に権限・承認ルートの手戻り発生 → 顧客満足度の低下
- **法対応（抵触日／3年ルール／36協定／同一労働同一賃金）の整備状況が導入前に可視化されておらず**、導入後に「そもそも運用体制が整っていない」ことが判明

### 2.3 本アプリで解決すること（チーム内発信として位置付け）
- **導入前ヒアリングの標準化**（YES/NO 15問程度で、契約／案件／勤怠／請求／承認フロー／法対応／データ運用の主要論点を網羅）
- 判定ロジックの **形式知化**（どの分岐で何パターンになるかを設計書として残し、営業・導入支援・運用で共通利用）
- 診断結果の **可視化**（カードUI + 印刷用ヒアリングシート）により顧客が即納得、チーム内の引き継ぎも容易
- **営業 → 導入支援 → 運用開始までのリードタイム短縮** と、**属人化の解消**（よくある「誰に聞けば分かる」問題の解決）

## 3. 対象ユーザーと利用シーン

### 3.1 ユーザー別プロファイル

#### A. CS営業担当者（一次ターゲット）
- **シーン**: 新規商談の初回訪問 / オンライン商談（**導入前ヒアリング補助** として起動）
- **使い方**: 自分のPCで画面共有しながら顧客（派遣元企業の導入検討担当）に質問を投げ、回答を自分でポチポチ入力
- **狙い**: 商談の「掴み」に使い、派遣管理運用の整備状況を自分も把握しながら、当日中に「御社は運用整備●●パターンです」と結論を出す
- **必要な体験**: 5分以内で診断完了、結果が "それっぽく" 見える、印刷ボタン1つで手土産化

#### B. CS導入支援担当者（二次ターゲット）
- **シーン**: 契約後のキックオフMTG / 設定前ヒアリング（**運用整備状況の簡易診断** として起動）
- **使い方**: 顧客担当者と画面共有 or 同席、顧客自身に回答させるケースも
- **狙い**: ヒアリング漏れを防ぐ、顧客に「なぜこの運用パターン／整備優先順位なのか」を根拠をもって説明する
- **必要な体験**: 診断結果 → 運用整備・設定案の項目別リスト → 次アクション（設定画面の該当箇所 or 整備タスク）への動線

#### C. 顧客担当者（派遣元企業の導入検討担当 / 情シス / 運用責任者）
- **シーン**: 営業・導入支援と対面 or 事前に自社内で整理
- **使い方**: 質問に答えるだけ。専門知識不要。
- **狙い**: 自社の派遣管理運用（契約／案件／勤怠／請求／承認／法対応／データ運用）の整備状況を改めて整理、SaaS導入時に何を準備すればいいか理解
- **必要な体験**: 業界用語が最小限、YES/NO以外の判断を迫られない、回答途中で戻れる

### 3.2 利用シーン・フロー

```
[営業フェーズ]
  商談   ─→   営業がアプリ起動   ─→   顧客にYES/NO質問   ─→   診断結果提示
                                                                      │
                                                                      ▼
                                                          「印刷してお渡し」 or
                                                          「PDFで後日メール」
[導入支援フェーズ]
  契約後 ─→  CS担当と顧客で再度実施  ─→  結果 + 設定項目チェックリスト生成
                                                    │
                                                    ▼
                                         次アクション: 組織マスタ登録へ
```

### 3.3 派遣元 / 派遣先 / SaaS管理者の責務整理

本アプリで扱う "組織設定" は、3者の責務がそれぞれ異なる。質問設計とパターン出力は以下の責務分担を前提とする。

| 役割 | 本アプリで扱う対象 | 本アプリが出力する内容 | 本アプリが扱わない内容 |
|------|--------------------|------------------------|------------------------|
| **派遣元（顧客＝派遣会社）** | 組織ツリー・承認階層・給与/職種マスタ・抵触日管理体制・36協定運用・同一労働同一賃金方式 | 推奨される組織構造・権限ロール・法対応フラグの初期値 | 個別労働者の就業可否・実際の協定締結・賃金額の具体算定 |
| **派遣先（派遣会社の顧客企業）** | 事業所単位/個人単位の期間制限に関する登録粒度、勤怠承認の電子化要否、派遣先管理台帳連携の有無 | 派遣先企業・事業所・就業場所の登録粒度の推奨、派遣先側アカウント発行方針 | 派遣先企業の就業規則判定、派遣先の管理台帳そのもの |
| **SaaS管理者（CS側）** | 提案パターン・設定手順書の選択、導入プロジェクト計画 | 設定項目チェックリスト・工数目安・注意点 | 実際のクラウドスタッフィング環境への自動反映（スコープ外） |

質問文は、各質問がどの役割の情報を引き出すものかを補足列（§10.2）で識別可能にする。

## 4. 機能要件

### 4.1 MUST（必須）

| ID | 機能名 | 要件 |
|----|--------|------|
| M-01 | YES/NO 質問フロー | 最低10問以上の質問ツリーを用意。回答に応じて次の質問へ分岐（動的） |
| M-02 | 進捗バー表示 | 全質問数と現在位置を視覚化（例: 「3/12」） |
| M-03 | 1つ前に戻る機能 | 誤回答時に即座に前の質問へ戻れる |
| M-04 | 運用パターン診断 | 回答結果を集計し、事前定義した3〜5パターンのうち1つに分類 |
| M-05 | 診断結果カード表示 | パターン名、概要、推奨設定のサマリをカードUIで表示 |
| M-06 | 設定案項目リスト | 組織構造・承認階層・権限・案件紐付けなどの項目別に、推奨設定値を表示 |
| M-07 | ヒアリングシート出力（2種） | ブラウザの印刷機能（`window.print()`）で出力。**要約版（A4縦1枚）** = パターン名・該当理由・推奨設定サマリ5項目のみ。**詳細版（A4縦2〜3枚）** = 要約版 + 回答一覧 + 設定項目チェックリスト。切替ボタンを用意 |
| M-08 | やり直しボタン | 診断完了後、最初から再開できる |
| M-09 | 単一HTML完結 | 外部CDN・ライブラリ一切なし。ダブルクリックで起動 |
| M-10 | オフライン動作 | ネット切断状態でも全機能動作 |
| M-11 | 保存しないモード（初期選択固定） | 起動時に「保存しない（一時利用）」「保存する（続きから再開可能）」の2択モーダルを表示し、**初期選択は「保存しない」に固定**（安全側デフォルト）。保存は明示的オプトインのみ。保存しないモード選択時は `localStorage` に一切書き込みせずセッションメモリのみで動作 |
| M-12 | 保存クリアボタン | 画面上から明示的に `localStorage` の回答・入力履歴を削除できるボタンを常設（診断結果画面にも配置） |
| M-13 | PII入力の最小化 | 顧客企業名・担当者氏名等の入力欄（S-05）は **任意入力かつデフォルト空欄**。入力時は「この情報はブラウザ内のみに残ります」の注意書きを併記 |
| M-14 | 免責文の常時表示 | **全画面共通ヘッダー（固定）** に「本ツールは診断補助であり、法令判定ではありません。最終判断は専門家レビューを前提とします」の注記を常時表示。スタート画面・質問画面・結果画面・印刷プレビューのすべてで表示されることを担保（結果カードのみの表示は不可） |

### 4.2 SHOULD（推奨）

| ID | 機能名 | 要件 |
|----|--------|------|
| S-01 | 質問ごとの補足説明 | 「これはどういう意味？」をクリックで展開、派遣業界用語を解説 |
| S-02 | アニメーション演出 | 質問切替時のフェード、診断結果表示時のカードスライドイン（CSS transition で十分） |
| S-03 | 回答の自動保存 | `localStorage` にセッション保存、ブラウザ閉じても翌日続き可能。M-11「保存しないモード」選択時はこの機能は無効化。`localStorage` がブラウザ設定等で利用不可の場合は画面上部に警告バナーを表示し、セッションメモリのみで動作継続 |
| S-04 | 診断結果のシェア用コピー | 結果サマリをクリップボードにコピー（メール貼り付け用）。`file://` プロトコル等で `navigator.clipboard` が使えない場合は、テキストを選択済み状態のモーダルを表示し、手動コピーできるフォールバックUIを用意 |
| S-05 | 顧客企業名・日付の入力欄 | ヒアリングシート出力時にヘッダーに印字。M-13 に従い任意入力・デフォルト空欄 |
| S-06 | 派遣法関連フラグ表示 | 抵触日管理が必要か、36協定連携が必要か等、診断結果に法対応チェックを含める |
| S-07 | 営業担当者向けメモ欄 | 診断後、営業が補足メモを追記できるテキストエリア（印刷時に含まれる） |

### 4.3 MAY（あると良い）

| ID | 機能名 | 要件 |
|----|--------|------|
| Y-01 | 結果の URL共有 | 回答パラメータをURLにエンコードし、同じ結果を再現できるリンク生成 |
| Y-02 | ダークモード | セミナー会場の照明に応じた表示切替 |
| Y-03 | 複数パターン比較 | 「御社は A パターンが最適ですが B でも可」のような次善案表示 |
| Y-04 | 業界統計との比較 | 「御社と同規模の派遣会社の70%が A パターンを選択」といったベンチマーク表示（モックデータでOK） |
| Y-05 | QRコード表示 | ヒアリングシートに結果再現URLのQRを印字 |

## 5. 非機能要件

| 分類 | 要件 |
|------|------|
| **性能** | 質問切替は体感即時（100ms 以内）。ファイルサイズ **目標300KB / 上限1MB**（§7.2制約と整合） |
| **互換性** | Chrome / Edge / Safari の最新版で動作。IE非対応（MUST NOT対象）。`file://` 起動時の `navigator.clipboard`・`localStorage` 利用不可ケースを考慮し、S-03/S-04 に記載のフォールバックを SHOULD で実装 |
| **ポータビリティ** | 単一HTMLファイルのみ。USB経由、メール添付、Slack投稿どれでも配布可能 |
| **操作性** | マウス操作のみで完結（MUST）。キーボード操作（Y/N キーで回答、←で戻る）に対応（SHOULD） |
| **アクセシビリティ** | 色のコントラスト比 4.5:1 以上。文字サイズ最低14px |
| **印刷品質** | A4縦・余白10mm・モノクロ印刷でも判読可能。要約版は1枚、詳細版は2〜3枚に収まること（M-07） |
| **保守性** | 質問ツリー・判定ロジック・運用パターン定義を JavaScript 内のデータ構造として分離し、後から質問追加が容易 |

## 7. 前提・制約

### 7.1 前提
- セミナー参加者は **派遣業界の実務者**（派遣元または派遣先の運用担当）
- 診断ロジックは "ヒアリング補助レベル"（MUST）。**法令適合性判定は一切行わない**（§1.6 免責に準拠）
- 診断パターンの実設定への落とし込みは、実際のクラウドスタッフィング画面では別途必要（本アプリはあくまで「方向性を決める」までを担う）
- 派遣法・同一労働同一賃金・36協定等に関する最終判断は、顧客側の法務/労務専門家レビューを前提とする。本アプリの出力は「要確認項目の洗い出し」に留まる
- デモ中はプロジェクター投影 or 画面共有前提。スマホ表示の最適化は不要（PCのみ、SHOULD）

### 7.2 制約
- **単一HTMLファイル制約**（外部ファイル分割禁止）
- **外部通信なし**（API呼び出し、CDN読込含む）
- **ライブラリ禁止**（React / Vue / jQuery 等、すべてVanilla JSで実装）
- **ファイルサイズ: 目標300KB / 上限1MB**（§5性能と整合、メール添付想定）
- 画像を使う場合は SVG をインラインで、または CSS で描画

## 10. 質問フロー設計（サンプル）

> **質問設計の観点**: 派遣管理運用の **契約管理・案件管理・勤怠管理・請求処理・承認フロー・法対応・データ運用** の各領域の整備状況を、YES/NO を主軸に引き出すよう調整。各質問がどの業務領域（発注／契約／案件／勤怠／請求／評価／法対応／データ運用）に紐付くかは §10.2 の評価軸列に明記する。

### 10.1 質問ツリー全体像

```
[START]
  │
  ├─ Q1: 派遣スタッフの就業先は複数事業所にまたがりますか？
  │      ├─ YES → Q2
  │      └─ NO  → Q2（単一事業所フラグ立てて次へ）
  │
  ├─ Q2: 拠点（営業所・支社）は3箇所以上ありますか？
  │      ├─ YES → Q3（多拠点ルート）
  │      └─ NO  → Q4（単一or少拠点ルート）
  │
  ├─ Q3: 拠点ごとに営業所長など承認者がいますか？（多拠点ルート）
  │      ├─ YES → Q5
  │      └─ NO  → Q6
  │
  ├─ Q4: 本社以外に実質的な管理機能を持つ拠点はありますか？（少拠点ルート）
  │      ├─ YES → Q5
  │      └─ NO  → Q7（承認階層簡易ルート）
  │
  ├─ Q5: 本社で全拠点の売上・勤怠を一元管理したいですか？
  │      ├─ YES → Q6
  │      └─ NO  → Q8（分散管理型）
  │
  ├─ Q6: 承認者が不在時、代理承認のルートを決めていますか？
  │      ├─ YES → Q9
  │      └─ NO  → Q9（未整備フラグ立てて次へ）
  │
  ├─ Q7: 承認は1段階（直属上司のみ）で十分ですか？
  │      ├─ YES → Q10
  │      └─ NO  → Q9
  │
  ├─ Q8: 拠点間で給与・勤務ルールは異なりますか？
  │      ├─ YES → Q10（分散型 + 異ルール文脈）
  │      └─ NO  → Q10（分散型 + 統一ルール文脈）
  │
  ├─ Q9: 派遣先からの勤怠承認を電子化したいですか？
  │      ├─ YES → Q10
  │      └─ NO  → Q10
  │
  ├─ Q10: 職種別に給与テーブルを分けていますか？（※旧Q11統合。Q8 YES時は「職種×拠点マトリクス」文脈で表示）
  │       ├─ YES → Q12
  │       └─ NO  → Q12
  │       （※Q11 は §10.2 で Q10 に統合済みの欠番）
  │
  ├─ Q12: 3年超の派遣契約（抵触日管理が必要な案件）を扱っていますか／扱う予定ですか？

  │       ├─ NO         → Q13
  │       └─ わからない → Q13（要ヒアリング タグ付与）
  │
  ├─ Q13: 同一労働同一賃金対応の方式は確定していますか？（3択・例外UI）
  │       ├─ 労使協定方式 → Q14
  │       ├─ 均等均衡方式 → Q14
  │       └─ 未確定       → Q14（要ヒアリング タグ付与）
  │
  ├─ Q14: 同一人物が複数拠点・複数部署を兼任する運用はありますか？
  │       ├─ YES → Q15
  │       └─ NO  → Q15
  │
  ├─ Q15: 今後1年以内に、拠点追加・M&A・事業拡大の計画はありますか？
  │       ├─ YES → Q16（拡張フラグON）
  │       └─ NO  → Q16（拡張フラグOFF）
  │
  ├─ Q16: 36協定の運用はどの体制ですか？
  │       ├─ 自社運用       → Q17
  │       ├─ 顧問社労士委託 → Q17
  │       ├─ 未整備         → Q17（要ヒアリング タグ付与、C=3寄り）
  │       └─ わからない     → Q17（要ヒアリング タグ付与、未整備と同等に扱う）
  │
  ├─ Q17: 抵触日運用体制（3ヶ月前通知・法務承認等）は定まっていますか？
  │       ├─ 定めている → [判定へ]
  │       ├─ 一部のみ   → [判定へ（要ヒアリング タグ付与）]
  │       ├─ 未整備     → [判定へ（要ヒアリング タグ付与、C=3寄り）]
  │       └─ わからない → [判定へ（要ヒアリング タグ付与、未整備と同等に扱う）]
  │
[判定ロジック → §10.3 決定表に従い運用パターン確定]
```

### 10.2 質問一覧（詳細）

> 評価軸列の末尾に、対応する **業務領域タグ**（発注／契約／案件／勤怠／請求／評価／法対応／データ運用）を併記。質問文はクラウドスタッフィング業務文脈（派遣元 / 派遣先 / スタッフ の3者間の運用）に沿った表現に調整。

| No | 質問文 | 選択肢 | 評価軸（業務領域） | 補足説明（S-01機能用） |
|----|--------|--------|--------|------------------------|
| Q1 | 派遣スタッフの就業先は、複数の派遣先事業所にまたがりますか？ | YES/NO | 派遣先の粒度（発注／契約） | 派遣先の事業所単位で抵触日が管理されるため、複数事業所への派遣がある場合は発注・契約マスタの登録粒度を細かくする必要があります |

| Q3 | 拠点ごとに、契約・勤怠・請求を承認する営業所長などの承認者がいますか？ | YES/NO | 承認階層（契約／勤怠／請求） | 拠点単位の承認者がいる場合、承認ルートを「拠点→本社」の2段階以上で設計します |
| Q4 | 本社以外に、案件管理・請求処理などの実質的な管理機能を持つ拠点はありますか？ | YES/NO | 管理機能分散度（案件／請求） | 管理機能がある拠点は、マスタ編集権限やレポート閲覧範囲を個別設定します |
| Q5 | 本社で全拠点の案件・勤怠・請求・評価を一元管理したいですか？ | YES/NO | 集約型 vs 分散型（案件／勤怠／請求／評価） | 一元管理型の場合、本社管理部門に全拠点横断の閲覧権限を付与します |
| Q6 | 承認者が不在時、契約・勤怠・請求の代理承認ルートを決めていますか？ | YES/NO | 代理承認の有無（契約／勤怠／請求） | 未整備の場合、導入時に代理承認フローの設計が別途必要になります |

| Q9 | 派遣先からの勤怠承認を電子化したいですか？ | YES/NO | 電子承認範囲（勤怠） | 電子化する場合、派遣先担当者のアカウント発行と承認権限設定が必要です |

| Q12 | 同一派遣先での継続派遣が3年（抵触日）を超える可能性のある案件を扱っていますか／扱う予定ですか？ | YES/NO/わからない | 抵触日管理の有無（法対応／契約） | 派遣法上の抵触日（事業所単位・個人単位の期間制限、各3年）の管理対象となり得る案件の有無を確認します（契約期間そのものの長短を問うわけではない）。本アプリでは抵触日管理は **原則ONを前提** とし、この回答はあくまで運用上の優先度判定に使用します。「わからない」は結果に「要ヒアリング」タグで残します |

| Q14 | 同一人物が複数拠点・複数部署を兼任する運用はありますか？（例: 営業所長が本社役員を兼ねる、営業兼管理） | YES/NO | 兼務管理（契約／データ運用） | 兼務ユーザーは複数組織への所属許可と権限の論理和制御が必要です。「派遣元社員が派遣先で役職」のような希少ケースは S-07 メモ欄に記入ください |
| Q15 | 今後1年以内に、拠点追加・M&A・事業拡大の計画はありますか？ | YES/NO | 拡張性（データ運用） | 拡張計画がある場合、組織コード体系を拡張に耐える桁数で設計します |
| Q16 | 36協定（時間外労働上限）の運用はどの体制ですか？ | 自社運用/顧問社労士に委託/未整備/わからない | 36協定運用（法対応／勤怠） | 36協定の締結・届出自体は法令上の手続きで本アプリは関与しません。この回答は「36協定超過監視アラートをSaaS上で必要とするか」の運用要件判定に使います。「未整備」「わからない」は結果に「要ヒアリング」タグで残します（わからないは未整備と同等扱い） |
| Q17 | 抵触日の運用体制（3ヶ月前通知・法務承認等）は定まっていますか？ | 定めている/一部のみ/未整備/わからない | 抵触日運用体制（法対応／契約） | 抵触日管理機能自体はONを前提とし、通知間隔と承認ルート設計の詳細度をこの回答で判定します。「一部のみ」「未整備」「わからない」はパターンEの法対応要整備型に寄せつつ「要ヒアリング」タグを付与（わからないは未整備と同等扱い） |

### 10.3 分岐ロジック（決定表）

#### 10.3.1 スコアリング規則

各回答を **評価軸スコア** として集計する。

| 軸 | スコア定義 | 判定元の主な質問 |
|----|-----------|------------------|
| **拠点軸** L | L=1 単一 / L=2 少拠点 / L=3 多拠点 | Q2（3箇所以上YES→L=3）, Q4（少拠点でも管理機能分散YES→L=2）, デフォルト L=1 |
| **承認軸** A | A=1 1段階 / A=2 2段階 / A=3 3段階以上 | Q3 YES → A≥2（拠点承認者あり）, Q7 YES → A=1（1段階で十分）, Q5 YES かつ L≥2 → A≥2（本社最終承認で2段階以上）, Q9 YES → A に加算なし（派遣先承認は別フロー扱い）, Q6 NO → A値は変えず「代理承認未整備」フラグを立てる |
| **法対応軸** C | C=1 基本 / C=2 標準 / C=3 強化（= **法対応要整備**を意味する。Q16/Q17/Q6 のいずれかが未整備側で、運用体制整備が急務のケースに付与） | Q12, Q13, Q16, Q17, Q6 の回答により加算。Q12 YES または Q17「一部のみ」で C≥2。**Q16「未整備」 OR Q17「未整備」 OR Q6 NO（代理承認未整備）** のいずれか1つで C=3（未整備要素を含むため強化＝要整備扱い）。Q13「未確定」は C を上げず「要ヒアリング」タグのみ付与 |
| **派遣先粒度補助** P | P=細 / P=粗 （Lの補正と出力注記に使用、軸判定には直接寄与しない） | Q1 YES → P=細（派遣先の事業所単位を細かく登録推奨）、結果カードに「派遣先登録粒度: 事業所単位で分離」の注記を付与 |

#### 10.3.2 パターン決定表（上から順に評価し、最初にマッチしたものを採用）

| 優先度 | 条件 | 確定パターン |
|--------|------|--------------|
| 1 | `(C = 3) AND ((Q16 = 未整備) OR (Q17 = 未整備) OR (Q17 = 一部のみ) OR (Q6 = NO))` | **E: 法対応要整備型** |
| 2 | `(L = 3) AND (A ≥ 2) AND (Q15 = YES)`（拡張計画あり） | **C: 将来拡張対応型**（旧: 多拠点・階層承認・将来拡張型） |
| 3 | `(L = 3) AND ((Q5 = NO) OR (Q8 = YES))`（一元管理しない、または拠点別ルール異） | **D: 多事業所分散型**（旧: 分散管理型） |
| 4 | `(L = 2) AND (A = 2)` | **B: 複数拠点対応型**（旧: 階層承認型） |
| 5 | `(L = 1) AND (A = 1) AND (C = 1)` | **A: 基本整備型**（旧: シンプル単一型） |
| 6（フォールバック） | 上記いずれにも該当しない | **最近傍パターン（スコア距離最小）を提示し、カード上部に「要ヒアリング（確定度: 中）」バッジを表示** |

#### 10.3.3 タイブレーク優先順位（§10.3.2 決定表の補足）

§10.3.2 の決定表は **法対応軸 C ＞ 拠点軸 L ＞ 承認軸 A** のタイブレーク優先順位を既に優先度順に反映している（C=3 → E を最上位、次に L=3 系、最後に L=2・L=1 系）。実装上は「上から順に評価し最初にマッチしたものを採用」するだけで優先順位が成立する。
副次条件（優先度1で E が採用された場合でも、L=3 や Q15 YES など他の特徴がある場合）は、結果カード内の「該当理由」欄に **副次マッチ条件** として併記する（例: 「主: E 法対応要整備型 / 副: 多拠点＋拡張計画ありの特徴あり」）。

#### 10.3.4 「要ヒアリング」タグ付与ルール

以下のいずれかに該当する場合、結果カードに **黄色の「要ヒアリング」タグ** を付与し、該当論点を列挙する。

- Q12 が「わからない」
- Q13 が「未確定」
- Q16 が「未整備」または「わからない」
- Q17 が「未整備」または「一部のみ」または「わからない」
- Q6 が NO（代理承認ルート未整備）
- フォールバック（優先度6）による確定

#### 10.3.5 ショート版（デモ用7問）ルート

デモ専用に、以下の7問を **固定順・分岐なしで線形に実行** する短縮ルートを並行定義する（フル版の分岐ツリーとは独立）。
Q2 → Q3 → Q5 → Q12 → Q13 → Q6 → Q15。

- Q5=NO（本社一元管理しない）の場合も Q12 へそのまま進む（フル版のQ8分岐はスキップ）
- Q3=NO（拠点承認者なし）の場合も Q5 へそのまま進む（フル版のQ6分岐はスキップ）
- Q6 はショート版の6問目として **Q13 の後**（法対応確認の後）に実施。派遣法論点を先に扱い、代理承認は補助情報として扱う意図
- このルートでは Q1 / Q4 / Q7 / Q8 / Q9 / Q10 / Q14 / Q16 / Q17 が未回答となるため、結果カードに **自動的に「要ヒアリング（省略項目あり）」バッジ** を付与する
- 判定式は上記 10.3.2 をそのまま適用し、未回答軸のデフォルトは以下のとおり明文化する:
  - **拠点L**: 回答のみから算出（Q2 で分岐）
  - **承認A**: Q3・Q5 から算出（Q7 未回答時は Q3=YES なら A=2、Q3=NO なら A=1 を初期値とする）
  - **法対応C**: `Q12=YES → C=2 / Q12=NO → C=1 / Q12=わからない → C=2 かつ「要ヒアリング」タグ付与`。ショート版では Q16/Q17 が未回答なので、**C=3（パターンE）はショート版では発火しない**（代わりに Q6=NO が回答済みのため、C=2 かつ要ヒアリングタグで代理承認未整備を明示）
  - Q13「未確定」: C を変えず「要ヒアリング」タグのみ付与
- Q5=NO の場合でも §10.3.2 の優先度3（L=3 かつ Q5 NO → D:分散管理型）判定は有効

## 11. 運用パターン出力例

> **パターン名の位置付け**: 各パターンは「組織構造の類型」ではなく **派遣管理運用の整備段階・対応範囲** を表す名称に整理した。A=基本整備型（単一拠点・基本整備）／B=複数拠点対応型（階層承認・中規模整備）／C=将来拡張対応型（多拠点・階層承認・拡張計画）／D=多事業所分散型（拠点別独立運用）／E=法対応要整備型（派遣法対応体制の整備未完）。§10.3 の決定表（L/A/C軸）はそのまま維持。

### パターンA: 【基本整備型（旧: シンプル単一型）】
- **該当条件**: 単一事業所・承認1段階・法対応基本（L=1, A=1, C=1）
- **想定企業**: 従業員30名未満、単一オフィスの派遣会社、新規参入事業者
- **推奨設定**:
  - 組織ツリー: 本社1層のみ（部署マスタ3〜5件）
  - 承認ルート: 直属上司のみ（1段階）
  - 権限: 役職ベースの簡易ロール2種（管理者 / 一般）
  - 案件紐付け: 全案件が本社所属
  - 派遣法対応: **抵触日管理は原則ON**（派遣法上、派遣を行う事業者は期間制限の管理が必要なため、本アプリでは常時ON前提）。通知間隔・承認フローは「簡易（抵触日3ヶ月前アラートのみ）」で良い。3年超の案件の運用方式（労使協定・意見聴取等）は **「要ヒアリング」として法務・社労士と確認** し、本アプリはこの設計の判断をしない
- **設定工数の目安**: 2〜3営業日
- **注意点**:
  - 成長時の拡張に耐えないため、拠点追加計画がある場合はパターンBを推奨
  - 本パターン採用時も、抵触日・同一労働同一賃金・36協定の体制整備は派遣元として法令上必要。「小規模だから省略できる」という示唆ではない旨を結果カードに明記

### パターンB: 【複数拠点対応型（旧: 階層承認型）】
- **該当条件**: 少拠点(1〜2)・承認2段階・法対応標準
- **想定企業**: 本社+営業所1〜2、従業員30〜100名の中堅派遣会社
- **推奨設定**:
  - 組織ツリー: 本社 → 営業所の2層
  - 承認ルート: 営業所長 → 本社管理部（2段階）
  - 権限: 拠点別ロール（所長 / 所員 / 本社管理 / 経理）
  - 案件紐付け: 案件は受注拠点に所属、横串案件は「担当者」で解決
  - 派遣法対応: 抵触日管理ON、同一労働同一賃金は労使協定方式
- **設定工数の目安**: 5〜7営業日
- **注意点**: 代理承認ルールの未整備が多いため、導入時に必ず整理

### パターンC: 【将来拡張対応型（旧: 多拠点・階層承認・将来拡張型）】★デモの主役
- **該当条件**: 多拠点(3以上)・承認2〜3段階・法対応標準/強化・拡張計画あり
- **想定企業**: 全国展開の中堅〜大手派遣会社、M&A志向の事業者
- **推奨設定**:
  - 組織ツリー: 本社 → エリア統括 → 営業所 の3層
  - 承認ルート: 所長 → エリア長 → 本社（最大3段階、案件規模別に分岐）
  - 権限: 細粒度ロール（7〜10種）、拠点横断閲覧は本社職のみ
  - 案件紐付け: 受注拠点 × 担当拠点の二軸管理、横串案件専用フラグ
  - 派遣法対応: 抵触日管理ON、3年ルール通知ワークフロー、同一労働同一賃金は労使協定方式推奨
  - 拡張性: 拠点コード4桁、組織コード体系は階層別プレフィックス
- **設定工数の目安**: 10〜15営業日
- **注意点**: 導入プロジェクトとして3ヶ月計画推奨、段階リリースが望ましい

### パターンD: 【多事業所分散型（旧: 分散管理型）】
- **該当条件**: 多拠点・本社集約しない・拠点ごとにルール異なる
- **想定企業**: 地域色の強い派遣会社、フランチャイズ型、グループ会社横並び
- **推奨設定**:
  - 組織ツリー: 各拠点を独立事業体として登録（疑似マルチテナント）
  - 承認ルート: 拠点内完結（1〜2段階）、本社は参照のみ
  - 権限: 拠点長に強めの編集権限、本社は閲覧限定ロール
  - 案件紐付け: 拠点内完結、横串案件は原則なし
  - 給与テンプレート: 拠点ごとに分離（異ルール）
  - 派遣法対応: 拠点別に設定（抵触日も拠点内で独立）
- **設定工数の目安**: 拠点数 × 2営業日
- **注意点**: 横串の経営レポート作成にBIツール連携が別途必要

### パターンE: 【法対応要整備型】
- **該当条件**: C=3（**法対応要整備**）に該当し、かつ Q16「未整備」 OR Q17「未整備」 OR Q17「一部のみ」 OR Q6 NO（代理承認未整備） のいずれかで発火。すなわち **派遣法関連の運用体制に未整備要素を含むケース**
- **想定企業**: 36協定・抵触日運用・代理承認フロー等の体制整備が未完の派遣会社（新規参入・急成長・組織再編直後など）
- **推奨設定**:
  - 組織ツリー: パターンBまたはCをベース（拠点数・承認階層は他の軸で決定）
  - 承認ルート: 抵触日3ヶ月前アラート時の承認フロー整備を優先タスク化。代理承認ルート未整備（Q6 NO）の場合はその設計も必須タスクに追加
  - 権限: 法務/労務担当ロールの新設を推奨、抵触日情報への限定アクセス
  - 案件紐付け: 案件ごとに派遣法区分（3年ルール対象 / 除外）を明示フラグ化
  - 派遣法対応: 抵触日管理ON・通知間隔を3ヶ月/1ヶ月/1週間の3段階を推奨。**Q16「未整備」の場合は36協定超過監視の要件自体を「要ヒアリング」扱いで出力**（体制未整備のまま監視機能だけONにしない）
  - 同一労働同一賃金: Q13 の回答に応じて、労使協定方式・均等均衡方式・未確定（双方テンプレを併置＋要ヒアリング）のいずれかを提示
- **設定工数の目安**: 体制整備の前段調整を含め 8〜12営業日（体制整備完了後は B/C と同等の工数）
- **注意点**:
  - 本パターンは「法対応が強化されている型」ではなく「**法対応体制の整備が必要な型**」である点を結果カードに明記（C=3 は "要整備" の意味であることを明示）
  - 派遣法改正時の設定見直し運用プロセスを別途定義
  - 本アプリは36協定の締結自体や同一労働同一賃金の具体的賃金算定は扱わない（§1.6 免責）。表示される推奨はあくまでSaaS上の運用機能設定の方向性

### 11.1 診断結果カードUIの要素

各パターンカードには以下を表示：

1. **パターン名**（大見出し、色分け）
2. **該当理由**（回答のハイライト、例「拠点3以上 + 承認2段階」）。§10.3.2 優先度番号は画面上は非表示とし、**詳細版印刷にのみ記載**（デモ時のノイズ回避）

8. **次のアクション**（例: 「組織マスタ設定画面へ」ボタン、モックなのでダミー遷移）

## 付録: 参考となる派遣業界用語

| 用語 | 意味 |
|------|------|
| 抵触日 | 派遣期間の上限に達する日。事業所単位(3年)と個人単位(3年)の2種類 |
| 3年ルール | 派遣期間の上限に関する派遣法上のルールの通称。**事業所単位**（派遣先事業所が派遣を受け入れられる期間の上限＝過半数労働組合等の意見聴取で延長可）と **個人単位（組織単位）**（同一の派遣労働者が同一の組織単位＝課等で働ける期間の上限＝原則延長不可）の2つがあり、両者は適用対象も延長可否も異なる |
| 同一労働同一賃金 | 派遣労働者と派遣先正社員の待遇差を合理的にする制度（労使協定方式 / 均等均衡方式） |
| 36協定 | 時間外労働・休日労働に関する労使協定。月45時間・年360時間が原則上限 |
| 労使協定方式 | 派遣元が労使協定を結び、給与水準を一般の労働者の平均以上にする方式 |
| 均等均衡方式 | 派遣先の同種業務の正社員と均等・均衡な待遇を確保する方式 |
| 事業所単位の期間制限 | 派遣先事業所が派遣労働者を受け入れられる期間の上限（原則3年） |
| 個人単位の期間制限 | 同一の派遣労働者が同一組織単位で働ける期間の上限（3年） |
