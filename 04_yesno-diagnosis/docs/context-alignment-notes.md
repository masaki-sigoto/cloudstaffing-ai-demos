# クラウドスタッフィング文脈寄せ修正メモ

## 日時: 2026-04-23

## 寄せ方針

- 「組織登録診断」という抽象テーマから、クラウドスタッフィング（派遣管理SaaS）の業務文脈に沿った **「運用整備状況の簡易診断／導入前ヒアリング補助／法対応・業務整備・データ運用成熟度の簡易診断」** へ位置付けを変更。
- チーム内発信（営業・導入支援・運用・顧客導入検討担当）の共通フォーマットとして自然に馴染むテーマへ。
- 業務領域タグ（発注／契約／案件管理／勤怠管理／請求／評価／法対応／データ運用）と登場人物3者（派遣元／派遣先／スタッフ）をドキュメント全体に浸透させた。
- 実装本質（単一HTML、決定表、スコアリング、ショート版7問、派遣法免責）は一切変更していない。

## テーマ変更

- **旧タイトル**: 「要件定義書: YES/NO 組織登録診断アプリ」
- **新タイトル**: 「要件定義書: 運用整備状況 簡易診断アプリ（旧: YES/NO 組織登録診断アプリ）」 ※併記形式で後方互換を確保
- **旧プロダクト名（仮）**: 「組織設定ナビゲーター」
- **新プロダクト名（仮）**: 「運用整備ナビゲーター」（旧名併記）
- **位置付けの変化**:
  - 「組織マスタ登録の設定診断」 → 「派遣管理運用（契約／案件／勤怠／請求／評価／法対応／データ運用）の整備状況診断」
  - セミナーの目玉枠という側面はそのまま維持しつつ、**チーム内発信**（営業→導入支援→運用の共通フォーマット、属人化解消）としての位置付けを明示。

## 修正箇所と内容

### 1. 冒頭ポジション（文書ヘッダ）

- 変更前: 「営業〜導入支援の初動設計」のみを強調
- 変更後: 「クラウドスタッフィング導入前ヒアリング／派遣管理運用の整備状況を YES/NO で素早く可視化する簡易診断ツール」「法対応・業務整備・データ運用成熟度の簡易診断」としてテーマの位置付けを再定義。登場人物3者と業務領域を明記。

### 1. プロジェクト概要（§1.1〜§1.4）

- 変更前: 「導入時の『組織設定』を診断し、派遣会社ごとの運用パターン・初期設定案を提示」
- 変更後: 「導入前ヒアリング／派遣管理運用の整備状況（発注／契約／案件／勤怠／請求／評価、法対応・データ運用）を診断し、運用整備パターンと初期設定・整備の優先タスク案を提示」
- KEY MESSAGE を「導入前ヒアリング」「派遣管理運用の整備状況の把握」の双方に効く表現へ更新。

### 2. 背景と解決する課題（§2）

- §2.1: 「なぜ『組織設定』が派遣管理SaaSで最もつまずくのか」→ 「なぜ『派遣管理運用の整備状況』の把握が、導入前フェーズで最もつまずくのか」
  - 派遣元／派遣先／スタッフ3者、発注〜請求〜評価の業務フロー、法対応・データ運用成熟度の観点を導入部で明記。
  - つまずきポイント表に「業務領域」列を追加し、既存7項目を発注／契約／案件／勤怠／請求／法対応／データ運用に紐付け。
  - 新規項目として「勤怠・請求・評価データの整合性（データ運用成熟度）」を追加（※情報追加のみで既存要件には影響なし）。
- §2.2: 「初回ヒアリングの標準化が未達」「属人化」「導入前に法対応体制の整備状況が可視化されていない問題」をチーム内発信の文脈で明示。
- §2.3: 「導入前ヒアリングの標準化」「チーム内の引き継ぎ容易化」「属人化解消」を解決価値として明示。

### 3. 対象ユーザーと利用シーン（§3.1）

- ユーザー種別ラベルを「派遣元企業の導入検討担当 / 情シス / 運用責任者」に変更し、利用シーンを **「導入前ヒアリング補助」「運用整備状況の簡易診断」** として具体化。

### 10. 質問フロー設計（§10.1・§10.2）

- §10.1 冒頭に質問設計の観点を追記（契約管理・案件管理・勤怠管理・請求処理・承認フロー・法対応・データ運用の各領域を YES/NO で引き出す方針）。
- §10.2 の質問一覧では、評価軸列に **業務領域タグ**（発注／契約／案件／勤怠／請求／評価／法対応／データ運用）を付記。質問文も「契約・勤怠・請求の承認者」「案件管理・請求処理の管理機能」「案件・勤怠・請求・評価の一元管理」など、派遣管理運用の具体観点に寄せ直し。
- ショート版7問・フル版13〜15問（Q10/Q11統合後）、分岐ツリー、決定表の構造は一切変更していない。

### 11. 運用パターン出力例（§11）

- パターン名を「派遣管理運用の整備段階・対応範囲」を表す名称に整理（旧名併記）:
  - A: シンプル単一型 → **基本整備型**
  - B: 階層承認型 → **複数拠点対応型**
  - C: 多拠点・階層承認・将来拡張型 → **将来拡張対応型**
  - D: 分散管理型 → **多事業所分散型**
  - E: 法対応要整備型（そのまま維持）
- §10.3.2 の決定表内パターン名も新名称に更新（旧名併記）。§6.1 デモシナリオ内のパターンC表記も同様に更新。
- 決定表の判定条件式（C軸／L軸／A軸／Q15／Q5／Q8）は一切変更していない。

### 6. デモシナリオ（§6.1）

- 登壇者台詞を「組織設定のややこしさ」から「派遣管理運用（契約・勤怠・請求・法対応）の整備状況の棚卸し」に寄せ替え。
- 画面タイトル「YES/NO 組織設定ナビ」→「運用整備ナビ（YES/NO 簡易診断）」。

## 触らなかった箇所とその理由

- **単一HTML制約（M-09, §7.2）**: 本質的な実装制約。変更不要。
- **外部通信なし／ライブラリ禁止／ファイルサイズ上限1MB**: 同上。
- **決定表（§10.3.2）・スコアリング規則（§10.3.1）・タイブレーク（§10.3.3）・要ヒアリングタグ付与ルール（§10.3.4）**: 判定ロジックの本質。文言・判定式は変更せず、パターン名の別名追加のみ。
- **ショート版7問ルート（§10.3.5, §6.1, KPI §8）**: デモ尺の核心で、Codexレビューで承認済みの設計。順序・軸初期値・バッジ付与ルールは一切変更しない。
- **派遣法免責（§1.6, M-14, §7.1）**: 法令判定ではなくヒアリング補助という位置付けは維持必須。免責文の常時表示要件もそのまま。
- **PII最小化（M-13）・保存しないモード（M-11）・保存クリア（M-12）・プライバシー既定値**: プライバシー要件の本質。変更不要。
- **Q1〜Q17 の分岐構造・選択肢・Q13の3択UI例外**: 質問文の寄せ替えは実施したが、選択肢数・分岐先・評価軸への寄与は一切変更していない。

## クラウドスタッフィング文脈として反映できた観点

- **運用整備状況診断**（発注／契約／案件管理／勤怠管理／請求／評価／法対応／データ運用の8領域での整備段階を可視化）
- **導入前ヒアリング補助**（営業商談・キックオフMTGの初回ヒアリングを標準化）
- **営業・導入支援の標準化**（個人の経験に依存せず、5分〜10分で同質の初動アウトプットを作れる）
- **属人化解消**（誰が担当しても同じ論点を網羅。チーム内引き継ぎが容易）
- **派遣元・派遣先・スタッフ3者の責務整理**（既存§3.3の枠組みを踏襲しつつ質問文・業務領域タグで強化）
- **法対応・業務整備・データ運用成熟度**（パターンEが"法対応要整備型"として機能、Q12/Q13/Q16/Q17/Q6 で法対応軸Cを評価）
- **よくある導入前課題の把握**（「承認フロー未整備」「代理承認ルート未定義」「抵触日運用未整備」「36協定体制未整備」「同一労働同一賃金方式未確定」を要ヒアリングタグで可視化）

## Phase3着手可否

本寄せ修正は **文言・位置付けの調整に限定** し、実装制約・判定ロジック・ショート版7問構成を一切変更していないため、**Phase3（実装フェーズ）着手に支障なし**。
