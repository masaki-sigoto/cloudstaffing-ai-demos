# クラウドスタッフィング文脈寄せ修正メモ

## 日時: 2026-04-23

## 寄せ方針

- 本プロジェクト（勤怠チェック自動化）は、**4本のAIデモの中で最もクラウドスタッフィング文脈に近い題材** のため、用語・シナリオ・登場人物・業務フロー記述を **強めに** 派遣管理寄りに書き直した
- クラウドスタッフィング（人材派遣管理SaaS）の中核業務である **発注／契約／案件管理／勤怠管理／請求／評価** のうち、本ツールは **勤怠管理〜月次締め（請求／給与の直前工程）** に明確に位置付けた
- 登場人物 **派遣元担当者 / 派遣先承認者 / スタッフ** の3者ワークフローを明示化し、各所での **確認・差戻し・承認** の往復を題材化
- 本質的な機能仕様（異常検知ロジック、判定閾値、重要度判定の3軸、MUST/SHOULD/MAY区分、入力CSVスキーマ）は変更していない

## 修正箇所と内容

### 1. プロジェクト概要（§1）

- 変更前: 「クラウドスタッフィング上に蓄積される勤怠データおよび各種申請データを、月次締め処理の前にルールエンジンが自動で突合・異常検知し、担当者が本当に確認すべきレコードだけをピックアップして提示するデモツール」
- 変更後: 「**スタッフ打刻データ** および **休暇・残業・振替等の申請データ／派遣先承認ステータス** を、**月次締め処理の直前** にルールエンジンが突合・異常検知。派遣元・派遣先・スタッフの **3者ワークフロー** が関わる勤怠確定プロセスの『締め前アラート』として機能させる」旨を明記
- 追加: `主対象業務: 派遣管理SaaSの勤怠管理〜月次締めフェーズ（請求・給与計算の直前工程）` を記述
- KEY MESSAGE補足で「数百人規模のスタッフ勤怠を月末に全件突合する派遣元担当者」を具体化

### 2. 背景と解決する課題（§2.1 / §2.2）

- 変更前: 派遣元事務担当者の工数圧迫／打刻漏れ／派遣先承認者の放置／「どこを見ればよいか分からない」
- 変更後:
  - 冒頭に「派遣元 / 派遣先承認者 / スタッフの **3者ワークフロー** で勤怠を確定させる」構造を明示
  - 差戻し・再承認の往復経路を具体的に記述（派遣元 → 派遣先承認依頼 → スタッフ打刻訂正 → 派遣元再確認）
  - 「どの案件の誰を、どの観点で見るべきか分からない」に具体化
  - 解決アプローチで「派遣先承認ステータスもルールエンジン突合対象」と明示、「この案件のこのスタッフだけ見ればOK」という利用イメージに

### 3. 対象ユーザーと利用シーン（§3）

- 変更前: 3ロール（派遣元締め担当／コーディネーター／派遣先現場責任者）の表
- 変更後:
  - **4ロール（派遣元締め担当、派遣元コーディネーター、派遣先承認者、スタッフ）** を明示した責任境界表に拡張
  - スタッフを主役級の1ロールとして追加（打刻者・申請者・差戻しを受けて修正する者）
  - 利用シーンに「派遣先未承認」「承認待ち滞留」「締め前アラート」を織り込み
  - 一次対応フロー図を3者ワークフロー版に書き直し（派遣先承認・スタッフ打刻訂正の2経路を明示）

### 4. 機能要件（§4）

- 異常検知パターン命名の調整:
  - A-01 「退勤打刻なし」→「**退勤打刻漏れ**（出勤のみ）」

### 6. デモシナリオ（§6）

- 標準出力例:

## 触らなかった箇所とその理由

| 箇所 | 理由 |
|---|---|
| §1.1 用語規定（ルールエンジン／AI／LLM） | 実装上の重要な規定であり、文脈変更と無関係 |
| §4.1 M-1 入力ファイル仕様・月別ディレクトリ規約 | CSV構造・パス規約はロジック本体。触るとテスト・実装前提が崩れる |
| §4.1 M-2 異常検知ロジック・判定表（閾値・A-06 status 判定表） | 「本質的な機能仕様は変えない」制約に該当 |
| §4.1 M-2.1 検知対象外リスト（36協定、抵触日、同一労働同一賃金など） | 法令観点のスコープ外定義であり、これ自体が派遣業務に則した記述として既に成立 |
| §4.1 M-3 重要度の数値スコア・例外条件 | 判定閾値・重要度定義の本質部分 |
| §5 非機能要件 / §5.1 実データ防波堤 | ロジック外だがセキュリティ・運用規定として固定 |
| §7.3 入力CSVスキーマ | データモデル本体。列追加は実装影響が大きいため、表示用として JSON 出力例にのみ `client_site`／`approver_status` を追加 |
| §8 成功基準 | 定量・定性基準は変更対象外 |

## クラウドスタッフィング文脈として反映できた観点

- **3者ワークフロー**（派遣元 / 派遣先承認者 / スタッフ）を §3 に明示、一次対応フロー図でも2経路（打刻訂正／承認依頼）を可視化
- **派遣先承認待ち滞留**（A-07 改名／説明強化）
- **月次締め前アラート** としての位置付け（§1, §2, §4.M-6, §6, §10）
- **申請×実績不整合**（A-06 改名）＝ 休暇申請と打刻の矛盾・残業申請漏れ
- **派遣先事業所別** チェックリスト（§4 M-4）
- **派遣元コーディネーター別** 通知（§4 M-5、§10.3）
- **給与影響／請求影響／法令影響** の3軸に派遣管理的な補足（スタッフ給与 / 派遣先への請求）
- **差戻し・再承認の往復** を課題として言語化（§2.1）
- **派遣先3社（ACME商事・BETA工業・GAMMA物流）** と事業所名（本社事業所／関西工場／東京DC）でリアリティ付与
- **派遣管理SaaSのモジュール境界**（勤怠管理／請求・支払／ロール認証）を §9 スコープ外で明示

## 自己採点

- **文脈寄せ度: 高（他3本より明確に強く寄せた）**
- 題材そのもの（勤怠×申請×承認）が派遣管理の中核業務と直結しているため、用語レベルで自然に置換できた
- 登場人物を4ロールに拡張しスタッフを明示した点、派遣先承認ステータスを検知軸として格上げした点、3社×事業所での具体化が特に寄せ度に効いている
- 本質ロジック（判定閾値・重要度定義・M-1〜M-6の機能区分）は無変更。Codex 3往復レビュー承認済みの骨格は保持

## Phase3着手可否

- **着手可能**。要件定義書の本質要素は無変更のため、Phase3（実装着手）に影響なし。新たに表示用として追加した `client_site`／`approver_status` は JSON 出力の任意フィールドとして扱えばよく、CSV スキーマ変更は不要。
