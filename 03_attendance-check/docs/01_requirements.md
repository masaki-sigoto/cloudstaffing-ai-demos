# 要件定義書: 勤怠チェック自動化

## 1. プロジェクト概要

本プロジェクトは、クラウドスタッフィング（人材派遣管理SaaS：発注／契約／案件管理／勤怠管理／請求／評価を一気通貫で扱う）上に蓄積される **スタッフ打刻データ** および **休暇・残業・振替等の申請データ／派遣先承認ステータス** を、**月次締め処理の直前** にルールエンジンが自動で突合・異常検知し、**派遣元の勤怠チェック担当者** が本当に確認・差戻しすべきレコードだけをピックアップして提示するデモツールである。派遣元担当者・派遣先承認者・スタッフの **3者ワークフロー** が関わる勤怠確定プロセスにおいて、「締め前アラート」として機能させることを想定する。

- プロジェクト名: 勤怠チェック自動化（Attendance Check Automation）
- KEY MESSAGE: **「確認地獄からの解放」** — 数百人規模のスタッフ勤怠を月末に全件突合する派遣元担当者を、見るべき箇所に集中させる
- 位置づけ: セミナー実演用モック（「動けば正義」方針）
- 主対象業務: 派遣管理SaaSの **勤怠管理〜月次締め** フェーズ（請求・給与計算の直前工程）
- 技術スタック: Python + CLI（標準ライブラリ優先、依存最小）
- 実行形式: Claude Code のスラッシュコマンドからターミナル起動

### 1.1 用語規定

- **ルールエンジン**: 実装上の検知ロジック主体。本要件のM-2はすべてルールベースで記述する。
- **AI**: セミナー演者口上（「AIが拾った12件」等）における対外表現。実装上はルールエンジンの出力を指す。
- **LLM**: MAY-1で扱う外部LLM連携機能を指す限定的な用語。
- 要件本文の機能記述では「ルールエンジン」を用い、デモシナリオ・サマリ表示など訴求文脈のみ「AI」を許容する。

## 2. 背景と解決する課題

### 2.1 現状の課題

派遣管理業務では **派遣元担当者 / 派遣先承認者 / スタッフ** の3者ワークフローで勤怠を確定させるが、各所に確認・差戻しの往復が発生し、月末月初が「確認地獄」になりがちである。

- 派遣元の勤怠チェック担当者は、月次締め前に **担当スタッフ数百〜数千人分の勤怠レコードを人力で全件確認** する必要があり、慢性的に工数を圧迫している
- スタッフの **打刻漏れ・休憩未入力・申請（休暇／残業／振替）と実績の不整合** は、目視では見落としやすいにもかかわらず、見逃すと **派遣先への請求エラー・スタッフへの給与ミス** に直結する
- 派遣先の承認者側でも、**承認依頼が放置（pending のまま滞留）** されたり、打刻不備の指摘を月末に受けて慌てて対応するケースが多い
- 結果として、派遣元 → 派遣先（承認依頼）→ スタッフ（打刻訂正依頼）→ 派遣元（再確認）という **差戻し・再承認の往復** が締め直前に集中する
- 「どの案件の誰を、どの観点で見るべきか分からない」状態が、締め担当者の心理的負荷を増大させている

### 2.2 解決のアプローチ

- スタッフ打刻データ・申請データ（休暇／残業）・派遣先承認ステータスをルールエンジンが突合し、**締め前に要対応となる異常パターン** に該当するレコードだけを抽出
- **重要度（高・中・低）** を「給与影響・請求影響・法令影響」の3軸で自動判定し、対応優先度を明示（請求への直撃＝高リスクを優先可視化）
- 派遣元担当者（コーディネーター）別・派遣先案件別の **チェックリスト** を自動生成し、「この案件のこのスタッフだけ見ればOK」な状態を提供
- 248件中12件だけにフォーカスできる → **月次締め前の確認工数を1桁減らす**

### 2.3 Before/After（デモ固定シナリオ）

| 観点 | Before（従来運用） | After（本ツール） |
|---|---|---|
| 対象件数 | 248件 全件目視 | 248件中12件のみ確認 |
| 1件あたり確認時間 | 平均15秒（人力突合含む） | 平均30秒（推奨アクション付きで即判断） |
| 合計確認時間 | 約62分 | 約6分 |
| 短縮率 | — | **約10倍短縮** |

※ 数値はデモ用の固定サンプル `samples/202604/` を前提とする。実運用ではデータ量・不備率により変動する。
※ **分母定義**: 「248件」は `timesheet.csv` の **行数（timesheet row count）** を指す。1行=1出退勤ペアであり、同一スタッフ・同一日に複数行が存在し得る（§7.3参照）。参考指標として「人日数（staff-day）」を別途表示する場合があるが、本Before/Afterの主指標は行数ベースとする。

## 3. 対象ユーザーと利用シーン

### 3.1 想定ユーザーと責任境界

派遣管理業務の **3者（派遣元 / 派遣先 / スタッフ）** を前提とした責任境界を整理する。本ツールの主ユーザーは派遣元側だが、出力されるチェックリスト・通知は派遣先承認者やスタッフへの差戻し起票資料としても使われる。

| ロール | 所属 | 役割 | 主な利用シーン | 修正依頼の起票 | 最終承認 |
|---|---|---|---|---|---|
| 派遣元 勤怠チェック担当者（締め担当） | 派遣元企業 | 月次締め・請求処理の実務担当 | 月末月初に **全担当スタッフの勤怠** をチェックし、不備を派遣先承認者／スタッフに差し戻す | ○（一次起票） | — |
| 派遣元 コーディネーター | 派遣元企業 | 案件・スタッフ管理（担当スタッフのフォロー） | 担当案件について異常レコードがあれば、スタッフや派遣先へ連絡する | △（補助的に起票・スタッフ・現場確認） | — |
| 派遣先 承認者（現場責任者） | 派遣先企業 | 受け入れ企業側の勤怠承認者 | 自社事業所に入っているスタッフの **打刻承認（approve/reject）** を行う | — | ○（打刻・申請の最終承認） |
| スタッフ | 派遣スタッフ（個人） | 勤務実施者 | 自ら打刻を行い、休暇／残業／振替を申請する | —（差戻しを受けて本人が修正） | — |

> 凡例: **○** = 主担当／**△** = 補助的に関与（一次起票は締め担当者、コーディネーターはスタッフ・派遣先確認に応じて起票支援）／**—** = 当該責務を持たない

### 3.2 利用シーン（デモ想定）

- **シーン1**: 派遣元の勤怠チェック担当者が、**月次締め日の3日前** に `/attendance-check` を実行し、要対応レコード（打刻漏れ・申請不備・派遣先未承認・承認待ち滞留など）の一覧を取得
- **シーン2**: 重要度「高」のレコード（請求直撃・給与直撃）から順に対応し、派遣先承認者や該当スタッフへ連絡・差戻し
- **シーン3**: コーディネーター別チェックリストを各担当に配布し、自身の担当スタッフ・案件の **締め前アラート** として消化させる（通知として標準出力またはテキストファイル）

### 3.3 一次対応フロー（3者ワークフロー）

```
[ルールエンジン] 締め前に異常検知（打刻漏れ・申請×実績不整合・承認待ち滞留 等）
        ↓
[派遣元 勤怠チェック担当者] 通知ファイル受領 → 内容確認・起票（差戻し／修正依頼）
        ↓
[派遣元 コーディネーター] 担当スタッフ・派遣先承認者へ確認依頼
        ↓  （打刻訂正）　　　　　　　　↓  （承認依頼）
[スタッフ] 打刻・申請を修正　　  [派遣先 承認者] pending 申請を approve/reject
        ↓
[派遣元 勤怠チェック担当者] 修正反映・承認完了を確認し月次締めに進む
```

本ツールのスコープは「**検知 → 差戻し起票情報の整形**」までであり、実際の修正・承認は派遣管理SaaS本体（本ツール外）で行う。

## 4. 機能要件

### 4.1 MUST（必須機能）

#### M-1. 勤怠×申請データ突合

- 入力は3つのCSVファイル（`timesheet.csv` / `applications.csv` / `shifts.csv`）を月別ディレクトリ `samples/{YYYYMM}/` から読み込む
- 実行時の指定方法は `--month YYYY-MM` 指定を第一級とし、該当ディレクトリを自動探索する
- スタッフID・日付をキーに突合する
- 3ファイルのうち `applications.csv` `shifts.csv` は存在しない場合 WARN 出力のうえ該当パターンをスキップ（突合対象外として処理継続）

#### M-2. 異常検知（ルールベース）

以下の異常パターンをルールエンジンで検知する（本MUSTにLLM等の非決定性要素は含まない）。検知は完全に決定的で、同一入力に対して同一出力を保証する。

| パターンID | 内容 | 検知ロジック |
|---|---|---|
| A-01 | 退勤打刻漏れ（出勤のみ） | 当該日のスタッフ打刻レコードに `clock_in` はあるが `clock_out` が NULL |
| A-02 | 出勤打刻漏れ（退勤のみ） | `clock_in` が NULL で `clock_out` のみ存在 |
| A-03 | 休憩未入力 | 実働6時間超かつ8時間以下で休憩<45分／実働8時間超で休憩<60分（労基法34条の最低基準を参照） |
| A-04 | 連続24時間以上の勤務 | 単一出退勤ペア（シフト跨ぎ含む）で `clock_out - clock_in` ≥ 24時間 |
| A-05 | 1日複数回の出退勤 | 同一スタッフ・同一日（JST 00:00-23:59）に2組以上の完結した出退勤ペアが存在 |
| A-06 | 申請×実績不整合（休暇申請と打刻の矛盾／残業申請漏れ） | 休暇申請`approved`済だが打刻あり／残業申請なしで **所定労働時間＋30分** を超過。所定労働時間は当該日の `shifts.csv` の `scheduled_end - scheduled_start - 休憩相当1h` を参照。`shifts.csv` が欠損している場合は **8時間** を固定値として用いる。**「残業申請なし」の判定は下記 A-06 判定表に従う** |

##### A-06 残業申請 status 別判定表

同一スタッフ・同一日について、`applications.csv` の `type=overtime` 行を全件走査し、以下に従って「申請あり扱い／なし扱い」を決定する。

| status | 扱い |
|---|---|
| `approved` | 申請あり扱い（A-06 不整合とはしない） |
| `pending` | 申請あり扱い（A-06対象外。ただし A-07 の対象になる） |
| `rejected` | 申請なし扱い（A-06 検知対象） |
| 行なし | 申請なし扱い（A-06 検知対象） |

※ 同一スタッフ・同一日に複数の `overtime` 申請行が存在する場合、`approved` > `pending` > `rejected` の優先順位で最上位の status を採用する。
| A-07 | 派遣先承認待ち滞留（承認遅延） | 申請ステータスが `pending` のまま、`applied_at` から **判定基準日（§7.1 参照）基準で過去3営業日以上** 経過。派遣先承認者の承認が詰まり、締め前アラート対象となる。`applied_at` が対象月の前月に入る場合も同じ算出式で判定する。営業日定義は §7.1 参照。**デモ再現性のため、判定基準日は `--as-of-date YYYY-MM-DD` で明示固定可能（未指定時は `--month` 月末締め日と CLI 実行 JST 日付の早い方）** |
| A-08 | 深夜打刻（シフト外） | 22:00〜翌5:00帯の打刻が `shifts.csv` の予定時間帯外で発生（労基法37条の深夜帯を判定基準として使用） |
| A-09 | シフトとの大幅乖離 | シフト予定の開始／終了と実打刻の差が±60分超 |
| A-10 | 重複打刻 | 同一スタッフ・同一種別（in/out）の打刻が5分以内に複数件 |

##### M-2.1 検知対象外（スコープ外の業務ルール）

以下は本ツールでは検知・判定しない。デモの範囲外である旨を明示する。

- **36協定の月間時間外上限**（月45時間／年360時間、特別条項）
- **派遣法の抵触日管理**（3年ルール）
- **同一労働同一賃金の賃金差分析**
- **年次有給休暇の付与・消化管理**
- **社会保険加入要件の判定**
- **複数事業所通算の労働時間集計**

#### M-3. 重要度別表示（高・中・低）

**評価3軸**: 各パターンを「給与影響」「請求影響」「法令影響」の3軸で評価し、最大スコアを採用する（スコア: 高=3／中=2／低=1／影響なし=0）。

| パターンID | 給与影響（スタッフ給与） | 請求影響（派遣先への請求） | 法令影響（労基法等） | 採用 重要度 |
|---|---|---|---|---|
| A-01 退勤打刻漏れ | 高 | 高 | 中 | **高** |
| A-02 出勤打刻漏れ | 高 | 高 | 中 | **高** |
| A-03 休憩未入力 | 中 | 低 | 中 | **中** |
| A-04 連続24h超 | 中 | 中 | 高 | **高** |
| A-05 複数出退勤 | 低 | 低 | 低 | **低** |
| A-06 申請×実績不整合 | 高 | 高 | 中 | **高** |
| A-07 派遣先承認待ち滞留 | 低 | 中 | 低 | **中** |
| A-08 深夜打刻（シフト外） | 中 | 低 | 中 | **中** |
| A-09 シフト乖離 | 中 | 中 | 低 | **中** |
| A-10 重複打刻 | 高 | 高 | 低 | **高** |

**例外条件**:

- 同一レコードに複数パターンが該当する場合は **最も高い重要度** を採用する。
- A-08が`shifts.csv`の予定時間内に収まる場合は検知対象外（スコープ外扱い）。
- A-04はシフト跨ぎの通し勤務が明示的に設定されている場合（`shifts.csv` で予定が24h以上）は「中」に降格。
- A-05は「同一 `client_id` 内で合計実働が所定労働時間以内」の場合は重要度「低」のまま据え置き（分割勤務が通常運用の現場での誤検知抑制）。所定労働時間の算出は A-06 と同じ規則に従う。

#### M-4. チェックリスト自動生成

- **派遣元コーディネーター別** にグルーピングしたチェックリストを出力（担当スタッフ単位の差戻し資料）
- **派遣先案件別（派遣先事業所別）** にグルーピングしたチェックリストも出力（承認依頼・打刻確認依頼の資料）
- 各項目には「スタッフ名 / 日付 / 派遣先事業所名 / 異常パターン / 重要度 / 推奨アクション / 派遣先承認ステータス」を記載

#### M-5. 派遣元コーディネーターへの通知（軽実装）

- 標準出力へのサマリ表示 + `output/notifications/{assignee_id}_{assignee_slug}.txt` 形式でのテキストファイル書き出し（派遣元担当者ごとに1ファイル）
  - ファイル名は担当者IDベースとし、`assignee_slug` は担当者名を `[A-Za-z0-9-]` に制限してサニタイズ（日本語・記号は不可）
  - サニタイズ結果が空文字になった場合は `unknown` を用いる（例: `output/notifications/U-999_unknown.txt`）
  - 例: `output/notifications/U-001_sato.txt`
- 通知内容: 重要度「高」件数、対応推奨レコードのリスト、**月次締め対応期限**
- **対応期限の算出ルール**: 「その月の締め日の2営業日前」を一律設定（例: 2026年4月締めで締め日=2026-04-30なら対応期限=2026-04-28）。営業日は §7.1 の定義に従う。曜日表記は年依存のため本文には含めない。

#### M-6. セミナー映えするサマリ表示

- 実行結果のヘッダーに「**全 N 件中、要確認 M 件（高: X / 中: Y / 低: Z）**」を強調表示
- 例: `スタッフ勤怠 248件 → AIが拾った 12件（高:4 / 中:5 / 低:3）/ 月次締め前アラート` （※「AIが拾った」は演者口上表現。§1.1 用語規定参照）

### 4.2 SHOULD（推奨機能）

- **S-1. JSON形式の詳細出力**: チェックリストをJSONでも書き出し、後工程で扱えるようにする
- **S-2. 担当者・案件フィルタ**: `--assignee 山田` `--client ACME` で絞り込み
- **S-3. サンプルデータ生成スクリプト**: 意図的に異常を混入したダミーCSVを自動生成
- **S-4. カラー付き標準出力**: 重要度に応じてANSIカラー（赤／黄／青）で強調

> 注: `--month YYYY-MM` 指定は M-1 で MUST 化済のため SHOULD からは削除。

### 4.3 MAY（あれば嬉しい）

- **MAY-1. 簡易LLM連携（推奨アクション文生成）**: 異常レコードに対し「推奨アクション文」をLLMで自動生成
  - 入力: `{pattern_id, staff_name, date, client_name, severity}`
  - 出力: 単一行テキスト（最大80字）
  - 失敗時挙動: LLM呼出失敗時はルール辞書によるデフォルト文にフォールバックし、MUST機能は全て成立すること
  - 再現性: LLMの非決定性は許容（温度固定推奨）。デモ再現用にデフォルト文のみでも完結する構成を維持
- **MAY-2. HTMLレポート出力**: ブラウザで見やすい静的HTML
- **MAY-3. 前月比の異常件数トレンド表示**
- **MAY-4. Slack/メール通知モック**（標準出力へのダミー表示）

## 5. 非機能要件

| 項目 | 要件 |
|---|---|
| 実行環境 | Python 3.10以上、macOS/Linux |
| 依存 | 標準ライブラリ優先。追加する場合も最小限（例: `rich` 等のCLI装飾のみ許容） |
| 実行時間 | 1,000件規模のCSVを3秒以内に処理（計測条件: Apple Silicon M2・メモリ16GB・CSV読込から出力書込まで込み、LLM連携は除外） |
| 保守性 | 異常検知ルールはモジュール化し、ルール追加が容易な構造とする |
| ログ | 標準出力で処理進捗を可視化（セミナー映え重視） |
| エラー耐性 | CSVの列欠損・日付不正等はWARNログ出力のうえスキップし、処理継続。**スキップ件数・理由を実行末尾のサマリに集計表示し、`output/skipped_records.csv` へ書き出す（監査性担保）** |
| セキュリティ | デモ目的のため個人情報は全てダミー。実データは扱わない。通知ファイル名は担当者IDベース＋サニタイズ済みslugとし、生の個人名はパスに含めない。**実データ投入時の防波堤は §5.1 参照（必須）** |

### 5.1 実データ投入時の防波堤（必須仕様）

実データ／ダミーデータの判別を人間の推測に委ねず、**明示フラグによる二段ガード**で機械的に判定する。

- (1) 起動時に「ダミーデータ前提である」旨の注意表示を標準出力へ出す
- (2) `--data-class {dummy|real}` を **必須フラグ** とし、未指定時は処理中断（エラー終了）
  - `--data-class real` を指定した場合は `--allow-real-data` の併記も必須（二段ガード）。片方のみ指定時は処理中断
  - `--data-class dummy` 指定時は `--allow-real-data` が付いていても無視（dummy が優先）
- (3) `--data-class real` 指定時は `--mask-names` を **既定ON**（`--no-mask-names` で明示解除可）。`staff_name` / `assignee_name` をイニシャル化する
- (4) 実データ／ダミーの内容判定ロジックは本ツールに持たせない（機械判定不能のため、運用者の `--data-class` 指定を唯一の真実とする）

## 7. 前提・制約

### 7.1 前提

- スタッフ打刻データ・申請データ（休暇／残業）・シフトデータはCSVで提供される（固定スキーマ、§7.3参照）。派遣先承認ステータスは `applications.csv` の `status` 列で代表させる
- スタッフID・派遣先案件ID（client_id）・派遣元コーディネーターID（assignee_id）は派遣管理SaaS側マスタで既に紐づいている
- **ファイル配置規約**: `samples/{YYYYMM}/timesheet.csv` `samples/{YYYYMM}/applications.csv` `samples/{YYYYMM}/shifts.csv` の3ファイルを月別ディレクトリに配置する
- **タイムゾーン**: JST（UTC+9）固定。`date` 列はJSTの暦日を意味する
- **締め時刻**: 対象月末日の 23:59（JST）を締めとみなす
- **営業日定義**: 月〜金のうち、`samples/{YYYYMM}/holidays.csv`（列: `date`）に列挙された日を除く日。`holidays.csv` が存在しない場合は土日のみを非営業日とする
- **判定基準日（as-of-date）**: A-07 等の時間経過判定で用いる「現在日」。`--as-of-date YYYY-MM-DD` 指定時はその日付を採用。未指定時は「`--month` 月末締め日と CLI 実行時の JST 日付のうち早い方」を採用。デモ時は再現性担保のため **常に `--as-of-date` を明示固定する**（推奨値: `2026-04-28`）
- **A-07 の「3営業日」はこの営業日定義と判定基準日に従って算出する**

### 7.2 制約
- 本ツールは **セミナー実演用のモック** であり、実運用品質は担保しない
- データはすべてダミー生成、個人情報や実案件情報は含まない
- LLM API連携は MAY 項目とし、デモではオフラインでも完結する構成を優先
- 通知はメール等の実送信ではなくファイル出力／標準出力で代替

### 7.3 入力CSVスキーマ（想定）

#### timesheet.csv

**データモデル**: 1行 = 1組の出退勤ペア（打刻イベント行モデル）。**同一スタッフ・同一日に複数行が存在することを正規仕様として許容** する（A-05 複数出退勤・A-10 重複打刻の表現に必要）。重複打刻の判定は `clock_in`／`clock_out` それぞれの時刻を種別ごとに時系列で比較する。

| 列名 | 型 | 許容値・備考 |
|---|---|---|
| record_id | string | 例: `T-2026-04-15-0042`（行の一意キー、重複打刻判定・突合で使用） |
| staff_id | string | 例: `S-1011` |
| staff_name | string | 日本語可（出力のみに使用） |
| client_id | string | 例: `C-001` |
| client_name | string | 例: `ACME商事`（出力例 §10.3／§10.5 で使用。欠損時は `client_id` をフォールバック表示） |
| date | date | `YYYY-MM-DD`（JST） |
| clock_in | datetime? | `YYYY-MM-DD HH:MM` または空（NULL） |
| clock_out | datetime? | 同上 |
| break_minutes | int | 0以上の整数（分） |
| assignee_id | string | 例: `U-001`（担当コーディネーターID） |
| assignee_name | string | 日本語可（表示のみ） |

#### applications.csv

| 列名 | 型 | 許容値・備考 |
|---|---|---|
| application_id | string | 例: `AP-0001` |
| staff_id | string | timesheet と結合 |
| date | date | `YYYY-MM-DD` |
| type | enum | `leave` / `overtime` のいずれか |
| status | enum | `pending` / `approved` / `rejected` のいずれか |
| applied_at | datetime | `YYYY-MM-DD HH:MM` |
| approved_at | datetime? | 同上または空 |

#### shifts.csv

| 列名 | 型 | 許容値・備考 |
|---|---|---|
| staff_id | string | |
| date | date | `YYYY-MM-DD` |
| scheduled_start | datetime | `YYYY-MM-DD HH:MM`（JST） |
| scheduled_end | datetime | 同上（跨ぎ時は翌日日付） |

#### holidays.csv（任意）

| 列名 | 型 | 許容値・備考 |
|---|---|---|
| date | date | `YYYY-MM-DD`（JST）。営業日判定の除外日に使用 |

## 9. スコープ外

- 勤怠データの **修正・承認機能**（検知のみ、打刻訂正／申請承認は派遣管理SaaS本体で人が行う）
- 派遣先承認者／スタッフへの **自動差戻し起票**（本ツールは起票情報の整形まで）
- 給与計算・請求書発行との連携（請求／支払モジュールは別プロジェクト）
- モバイルアプリ / Webフロントエンド
- 実メール・Slack送信
- 多国語対応、タイムゾーン跨ぎ対応
- 実SaaS（クラウドスタッフィング本体）とのAPI接続
- 認証・権限管理（派遣元／派遣先／スタッフのロール認証）
- 監査ログ、データ永続化（DB利用）
- 大量データ（10,000件超）でのパフォーマンスチューニング
