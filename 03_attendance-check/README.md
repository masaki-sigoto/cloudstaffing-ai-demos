# 勤怠チェック自動化（Attendance Check Automation）

派遣管理SaaS「クラウドスタッフィング」上に蓄積されるスタッフ打刻・申請・シフトを、**月次締め処理の直前**にルールエンジンが自動突合し、派遣元の勤怠チェック担当者が確認すべき候補レコードだけを抽出・提示するデモツール。**抽出主体は A-01〜A-10 のルールエンジン。AI 活用は説明・可視化補助の位置付け**で、担当者が最終確認を行うワークフロー前提です。

**KEY MESSAGE:** 「確認地獄からの解放」— 数百人規模のスタッフ勤怠を月末に全件突合する派遣元担当者を、見るべき箇所に集中させる

## 統合ダッシュボード経由の起動

## 3者ワークフロー（派遣元／派遣先／スタッフ）

```
[ルールエンジン] 締め前に異常検知（打刻漏れ・申請×実績不整合・承認待ち滞留 等）
        |
        v
[派遣元 勤怠チェック担当者] 通知ファイル受領 → 内容確認・起票（差戻し／修正依頼）
        |
        v
[派遣元 コーディネーター] 担当スタッフ・派遣先承認者へ確認依頼
        |  （打刻訂正）              |  （承認依頼）
        v                            v
[スタッフ] 打刻・申請を修正     [派遣先 承認者] pending 申請を approve/reject
        |
        v
[派遣元 勤怠チェック担当者] 修正反映・承認完了を確認し月次締めに進む
```

本ツールのスコープは「**検知 → 差戻し起票情報の整形**」まで。
実際の修正・承認は派遣管理SaaS本体で行う。

## クイックスタート

### 前提
- Python 3.10以上
- 標準ライブラリのみ（外部依存なし）

### 実行

```bash
cd 03_attendance-check

# 1. ダミーサンプル生成（10異常パターン混入）
python3 src/main.py generate-samples --month 2026-04 --data-class dummy --count 12 --overwrite

# 2. 勤怠チェック実行
python3 src/main.py check --month 2026-04 --as-of-date 2026-04-28 --data-class dummy
```

### 出力先

| パス | 内容 |
|---|---|
| `out/notifications/U-xxx_slug.txt` | 派遣元コーディネーター別通知（1担当1ファイル） |
| `out/checklist/by_coordinator_YYYYMM.txt` | コーディネーター別チェックリスト |
| `out/checklist/by_client_site_YYYYMM.txt` | 派遣先事業所別チェックリスト |
| `out/result_YYYYMM.json` | Finding 全件 JSON |
| `out/run_summary_YYYYMM.json` | 実行サマリ |
| `out/skipped_records.csv` | 読込スキップ監査 |

## 10 異常パターン一覧

| ID | 内容 | 重要度 | 影響 |
|---|---|---|---|
| A-01 | 退勤打刻漏れ（出勤のみ） | 高 | 給与・請求直撃 |
| A-02 | 出勤打刻漏れ（退勤のみ） | 高 | 給与・請求直撃 |
| A-03 | 休憩未入力（労基法34条） | 中 | 法令 |
| A-04 | 連続24時間以上の勤務 | 高 | 労務リスク |
| A-05 | 1日複数回の出退勤 | 低 | 要確認 |
| A-06 | 申請×実績不整合（休暇×打刻矛盾／残業申請漏れ） | 高 | 給与・請求直撃 |
| A-07 | 派遣先承認待ち滞留（3営業日以上） | 中 | 締め前アラート |
| A-08 | 深夜打刻（シフト外） | 中 | 法令・労務 |
| A-09 | シフトとの大幅乖離（±60分超） | 中 | 現場確認 |
| A-10 | 重複打刻（5分以内） | 高 | 給与・請求 |

**重要度評価**: 給与影響／請求影響／法令影響の3軸で最大値を採用。同一レコードで複数パターン該当時は最高重要度。

## デモシナリオ（セミナー演出、7〜10分）

### 1. 状況設定（1分）
派遣元「クロスリンクスタッフ」の田中さん（勤怠チェック担当）が、4月度の月次締め日（2026-04-30）の2営業日前に、担当スタッフ全件勤怠を確認する必要がある。派遣先3社（ABC商事・XYZ物流・DEF製造）の承認状態と、スタッフからの休暇・残業申請の整合性を目視で全件チェックしている。

### 2. 実行（30秒）
```
python3 src/main.py check --month 2026-04 --as-of-date 2026-04-28 --data-class dummy
```

### 3. サマリ強調（30秒）
```
================================================
  勤怠チェック結果サマリ (2026-04 / 締め前)
================================================
  全 266 件 → 要確認 20 件
    [高] 11 件 / [中] 8 件 / [低] 1 件
================================================
```
**演者コメント：**「266件全部を目視でチェックしていたのが、ルールエンジンが要確認候補として抽出したこの20件に集中できる（AI は説明・可視化の補助）。最終的な差戻し可否は担当者が判断します。高優先度の11件から先に見れば月次締めに進めます」

### 4. 3者ワークフロー具体性（2分）
- コーディネーター別通知：担当ごとに`U-001_sato.txt` `U-002_takahashi.txt` `U-003_watanabe.txt`を生成
- 派遣先事業所別チェックリスト：派遣先承認者への差戻し資料として即使える
- 各Findingに推奨アクション付き：誰に何を照会すべきか明確

### 5. 10異常パターンの網羅性デモ（2分）
- 高：打刻漏れ(A-01/A-02) / 連続24h超(A-04) / 申請×実績不整合(A-06) / 重複打刻(A-10)
- 中：休憩未入力(A-03) / 承認待ち滞留(A-07) / 深夜打刻(A-08) / シフト乖離(A-09)
- 低：1日複数回(A-05)

### 6. 締め（1分）
「派遣元→派遣先→スタッフの3者間差戻しで最も負荷が高いのが月末。締め日の作業がこれだけ楽になる、というのが今日一番お伝えしたいことです」

## CLI 仕様

### `check` サブコマンド
```
python3 src/main.py check \
  --month 2026-04 \
  --as-of-date 2026-04-28 \
  --data-class dummy \
  [--assignee U-001] \
  [--client CL-001] \
  [--no-color]
```

### `generate-samples` サブコマンド
```
python3 src/main.py generate-samples \
  --month 2026-04 \
  --data-class dummy \
  --count 12 \
  [--seed 42] \
  [--overwrite]
```

### 実データ防波堤（二段ガード）
- `--data-class` 未指定 → exit 2
- `--data-class real` 指定時は `--allow-real-data` 必須
- `--data-class real` + `--no-mask-names` 解除は `--confirm-unmask-real` 必須

### 終了コード
- `0`: 正常終了（検知0件でも0）
- `1`: 致命的エラー（samples不在など）
- `2`: 引数エラー（data-class 未指定など）

## 想定QA（3問）

- **Q. 検知ロジックは AI が判断しているのですか？**
  A. 違います。**抽出主体は A-01〜A-10 のルールエンジン**による自動突合で、AI（生成 AI）の自由判断は介在しません。AI 活用は説明・可視化補助の位置付けです。差戻し可否の最終確認は派遣元の勤怠チェック担当者が行います。
- **Q. 266 件→20 件の数字は毎回同じですか？**
  A. 同梱のダミーサンプル（`--data-class dummy --count 12 --month 2026-04`）で再現できる固定値です。実データは件数も内訳（高 11／中 8／低 1）も変動します。
- **Q. このツールが派遣管理 SaaS の打刻訂正・承認も実行するのですか？**
  A. 行いません。本ツールのスコープは「検知 → 差戻し起票情報の整形」までで、実際の打刻訂正はスタッフ、承認は派遣先承認者が派遣管理 SaaS 本体で行います。
