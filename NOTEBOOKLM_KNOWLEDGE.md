### 1-2. 何のためのリポジトリか

- 「クラウドスタッフィング」は人材派遣管理 SaaS（派遣元・派遣先・派遣スタッフの 3 者間業務をデジタル化）
- その周辺業務に AI を当てた場合、どこに効果が出るかを「実際に動くモック」として 4 本提示
- 派遣会社向け無料セミナーの実演ネタ。本番運用ではなくセミナーの**体感教材**として位置づけ

### 1-3. 全体スタンス（重要、ナレーションでも踏襲）

> **AI は「確認作業を支援」する役割。整形・抽出・診断は AI／ロジックが行い、最終確認・最終判断は担当者が行います。**

- 「AI が完全自動で確定」「人の確認は不要」といった表現は採用しない
- 正確性が必要な領域（給与計算・勤怠判定）は決定論的ロジックで処理
- 判断支援が必要な領域（CSV 整形、要確認行抽出、組織登録パターン推奨）は AI 的補助で処理
- 最終確認は常に人が担う設計

## 2. なぜこのテーマか（背景）

### 2-1. 派遣管理業務の現場の痛み

- **紙運用 / Excel 運用が残る**: タイムシート、シフト表、契約書類が PDF や紙、Excel に散らばる
- **差戻しの往復が多い**: 派遣元と派遣先の間でフォーマット差・計算差・打刻差で月初月末に確認が往復する
- **月初月末の事務負荷ピーク**: 月次締め直前に勤怠・請求・給与の確認が集中
- **属人化**: 同じ派遣元・同じ派遣先でも担当コーディネーターによって運用が変わる
- **派遣法・36 協定・同一労働同一賃金などの周辺法規**: 専門知識が要るが、現場担当者は法律家ではない

### 2-2. 議事録から抽出された 4 テーマ

クラウドスタッフィング社との打ち合わせ議事録から、AI 活用の余地が高いテーマとして抽出:

1. **CSV 加工**（派遣元から届く非標準フォーマットの整形）
2. **端数処理**（同じ打刻でも丸めルール違いで給与/請求が変わる説明責任）
3. **勤怠チェック**（月次締め前の確認地獄、3 者ワークフロー固有の異常パターン）
4. **組織登録判断**（クラウドスタッフィング導入時の組織登録設計をパターン化）

### 2-3. AI 適用方針

- **AI 単体で確定判断はしない**: 計算・判定はロジック、整形・抽出・推奨は AI 的補助
- **担当者の最終確認を必須**: 要確認行は明示的にハイライト（`__needs_review` 列など）
- **再現性の確保**: 同じ入力 + 同じ as-of-date で同じ出力（時間バイアスを排除）
- **データガード**: ダミーデータと実データを CLI 引数で明示的に分離

## 3. 4 つのデモ

### Demo 1: CSV 加工の整形支援（`01_csv-automation`）

#### 価値
- 派遣元から届く崩れた CSV（列順バラバラ・ヘッダー揺れ・全角半角混在・令和表記・文字コード混在・エラー行混入）を 1 コマンドで標準形式に整形
- 整形は自動、最終確認は担当者（要確認行は `__needs_review` 列でハイライト）
- 対象は勤怠・スタッフ情報・契約データなど派遣管理運用の各種 CSV

#### よくある崩れパターン（実機で再現済）
- 列順がファイルごとにバラバラ（社員番号と氏名が左右逆など）
- ヘッダー揺れ: 「社員番号」「従業員ID」「スタッフコード」が混在
- 全角半角混在: 「９：００」と「9:00」、「３２時間」と「32時間」
- 令和表記: 「R6.4.1」と「2024-04-01」が同じ列に共存
- 文字コード混在: BOM 付き UTF-8 / CP932 / UTF-8 が月によって変わる
- エラー行混入: 集計行・空行・凡例行が混ざる

#### セミナー演出
- Before / After を画面で並べる
- 「Excel 手作業を大幅削減」を体感させる
- テンプレート保存（`save-template`）で「翌月以降は設定ゼロで自動整形」を見せる

#### 技術
- Python 標準ライブラリのみ（外部依存ゼロ）
- 文字コード自動判定（BOM → UTF-8 → CP932 の順でフォールバック）
- src 配下のモジュール構成（記載数値は当時のスナップショット値、現行リポジトリ実数で更新の可能性あり）
- 1 万行の CSV を 5 秒以内で処理する設計目標

#### 主な CLI
- `python3 src/main.py convert --input samples/timesheet_202604_haken_c.csv` — 1 ファイル変換
- `python3 src/main.py save-template --input ... --name haken_a` — テンプレート保存
- `python3 src/main.py convert --input ... --template haken_a` — テンプレート適用

#### 出力
- `out/<basename>.csv`（整形済み標準フォーマット）
- `out/<basename>_report.md`（処理レポート）
- `out/<basename>_needs_review.csv`（要確認行のみ抽出）

#### サンプル
- `samples/timesheet_202604_haken_a.csv` ほか派遣元 4 社分の崩れ CSV

### Demo 2: 端数処理チェッカー（`02_rounding-checker`）

#### 価値
- 同じ打刻でも丸めルール違いで給与・請求が変わる現実を可視化
- 「同じ打刻で 1 日 900 円差、月 20 日換算で 18,000 円差」という具体数字でインパクトを出す
- 計算は真理値表ベースの決定論的ロジック処理（AI の自由判断ではない）

#### 5 つの丸めルール
1. **strict_1min（フェア）**: 1 分単位で素計算
2. **5min_round**: 5 分丸め
3. **10min_round**: 10 分丸め
4. **employee_friendly（15 分スタッフ有利）**: 出勤は切り捨て、退勤は切り上げ
5. **company_friendly（15 分会社有利）**: 出勤は切り上げ、退勤は切り捨て

#### 用途
- 月次締め前の説明責任資料
- 派遣先との契約前のルール設定検証
- 派遣スタッフへの給与説明、派遣先への請求金額説明

#### セミナー演出
- 同じ打刻 `9:03 出勤 / 18:07 退勤 / 時給 1,800 円 / 休憩 60 分` を 3 ルールで並べる
- スタッフ有利 vs 会社有利で 1 日 900 円、月 20 日で 18,000 円の差が出ることを画面で見せる
- `explain --demo` で「なぜこの結果か」を 3 行で出力する説明責任機能を見せる

#### 技術
- Python 標準ライブラリ + PyYAML（無ければ内蔵 mini パーサにフォールバック）
- ルール定義は YAML（`samples/rules/*.yml`）
- 単一 `src/main.py` 実装

#### 主な CLI
- `python3 src/main.py simulate --config samples/rules/strict_1min.yml --punch 9:03,18:07 --hourly 1800` — 単発計算
- `python3 src/main.py compare --config A.yml --config B.yml --config C.yml --punch ... --hourly ... --break ...` — 複数ルール比較
- `python3 src/main.py explain --config ... --punch ... --hourly ... --demo` — 説明文出力
- `python3 src/main.py validate --config ...` — ルール定義の妥当性チェック

#### 主要数字
- 月 18,000 円差（同じ打刻で employee_friendly vs company_friendly、20 日換算）
- 1 日 900 円差（同じ打刻で employee_friendly vs company_friendly）

### Demo 3: 勤怠チェック自動化（`03_attendance-check`）

> **このデモは 4 本の中で最も派遣管理業務に深く結合**

#### 価値
- 月次締め前の確認地獄からの解放
- 全 266 件の勤怠データから要確認 20 件に絞り込み（高 11 / 中 8 / 低 1）
- 担当コーディネーター別 / 派遣先事業所別の通知資料を自動生成

#### 10 異常パターン
| ID | 異常パターン | 内容 |
|---|---|---|
| A-01 | 退勤打刻漏れ | 出勤打刻のみで退勤打刻なし |
| A-02 | 出勤打刻漏れ | 退勤打刻のみで出勤打刻なし |
| A-03 | 休憩未入力 | 6 時間超勤務で休憩記録なし |
| A-04 | 連続 24 時間超勤務 | 1 勤務レコード内で連続 24 時間以上の勤務 |
| A-05 | 1 日複数回出退勤 | 同日に複数の出退勤打刻 |
| A-06 | 申請×実績不整合 | 残業申請と実績が一致しない |
| A-07 | 派遣先承認待ち滞留 | 派遣先承認が長期間滞留 |
| A-08 | 深夜打刻 | 深夜帯の打刻（要確認対象） |
| A-09 | シフト乖離 | シフトと実績の大きな乖離 |
| A-10 | 重複打刻 | 同じ時刻に複数の打刻 |

#### 3 者ワークフロー
```
スタッフ打刻 → 派遣先承認者 → 派遣元集計 → 月次締め
```

派遣会社固有の 3 者構造を前提にした設計。派遣先承認者の承認待ち滞留（A-07）は派遣管理 SaaS ならではの異常パターン。

#### 出力
- `out/notifications/U-001_sato.txt` — 担当コーディネーター別の通知ファイル
- `out/checklist/by_client_site_202604.txt` — 派遣先事業所別チェックリスト
- `out/run_summary_202604.json` — 実行サマリ JSON（3 軸スコアリング: 給与影響 / 請求影響 / 法令影響）

#### データガード機構（実運用への配慮）
- `--data-class {dummy|real}` の二段ガード
- `real` 指定時は `--allow-real-data` フラグの追加が必須
- さらに `--no-mask-names` で氏名マスキングを解除する場合のみ `--confirm-unmask-real` の追加が必要
- ダミーデータと実データを誤って混ぜない CLI 設計

#### 再現性
- `--as-of-date YYYY-MM-DD` で「いつ時点のチェック結果か」を固定
- 同じ入力 + 同じ as-of-date で必ず同じ出力

#### 技術
- Python 標準ライブラリのみ
- src 配下のファイル構成（記載数値は当時のスナップショット値、現行リポジトリ実数で更新の可能性あり）
- 10 異常ルールはルールエンジン的にプラガブル
- 3 軸スコアリング（給与・請求・法令）で優先度付け

#### 主な CLI
- `python3 src/main.py generate-samples --month 2026-04 --data-class dummy --count 12 --overwrite` — サンプル生成
- `python3 src/main.py check --month 2026-04 --as-of-date 2026-04-28 --data-class dummy` — メイン実行

#### 主要数字
- 全 266 件 → 要確認 20 件
- 内訳: 高 11 件 / 中 8 件 / 低 1 件
- 処理時間: 0.02 〜 0.05 秒

### Demo 4: 運用整備状況 簡易診断（`04_yesno-diagnosis`）

#### 価値
- クラウドスタッフィング導入時の組織登録設計を、YES/NO 中心の質問（一部は Q12・Q13 など選択式）で判断支援
- 営業の初回商談（5 分）／導入支援の初回ヒアリング（10 分）で使える
- 属人化していた組織登録パターン判定をパターン化

#### 5 つの推奨パターン
- **パターン A（基本整備型）**: 単一拠点・運用シンプル
- **パターン B（複数拠点対応型）**: 複数拠点ありで組織階層が必要
- **パターン C（将来拡張対応型）**: 現状単一だが拠点拡張予定
- **パターン D（多事業所分散型）**: 多事業所で承認権限分散
- **パターン E（法対応要整備型）**: 36 協定・労使協定の整備が要件

#### 質問版
- **ショート版 7 問**: 営業商談 5 分用（固定 7 問）
- **フル版 最大 15 問（分岐あり）**: 導入支援 10 分用（YES/NO 中心、一部選択式。回答経路に応じて設問数が変動）

#### 結果画面の構成
- 推奨パターン提示（最終的な組織登録は担当者が確認のうえ決定）
- 推奨事項
- 要ヒアリング項目（追加確認が必要な点）
- 営業メモ欄
- 印刷対応（要約版／詳細版）

#### 技術
- 単一 HTML（約 81 KB）
- 外部依存ゼロ（CDN すら不使用）
- オフライン動作
- localStorage はデフォルト OFF（PII 漏洩リスク回避）
- 派遣法免責を全画面ヘッダーに固定
- PII 入力欄はデフォルト非表示

#### セミナー演出
- 想定動線: Q2=Y → Q3=Y → Q5=Y → Q12=Y → Q13=労使協定 → Q6=N → Q15=Y
- → パターン C（将来拡張対応型）に着地
- 印刷（Ctrl+P）で要約版／詳細版を派遣元担当者に渡せる体験

## 4. 統合ダッシュボード

### 4-1. 概要
- URL: `http://localhost:8765/`（デフォルト、環境変数 `PORT` で変更可）
- 起動: `./launch-dashboard.sh`（リポジトリルートで実行）
- アーキ: Python `http.server` ベースのカスタムサーバ + 各デモ専用 HTML 詳細画面

### 4-2. 機能
- Demo 01〜03 はボタン押下で実 CLI を `subprocess` 実行し、結果を整形して画面表示
- Demo 04 は静的 HTML（`04_yesno-diagnosis/src/index.html`）へのリンク遷移（API 呼び出しなし、ブラウザ完結）
- 4 デモ間をカード UI で切り替え

### 4-3. ファイル構成
- `dashboard/index.html` — ルートダッシュボード（カード UI）
- `dashboard/01.html` / `02.html` / `03.html` — 各デモ詳細画面
- `dashboard/server.py` — `http.server` ベースの API サーバ

### 4-4. ポート 8765 を採用した理由
- 主要開発ポートと競合しない番号
- 避けたメジャーポート: 3000 / 3001（Node・React）/ 4000 / 4200（Angular・Phoenix）/ 5000（Flask）/ 5173（Vite）/ 5500 / 5501（VSCode Live Server）/ 8000（Django・http）/ 8080（Apache）/ 8081（Jenkins）/ 8888（Jupyter）/ 9000（PHP-FPM・SonarQube）
- 8765 は memorable（昇順）で競合しにくい

### 4-5. セミナー本番での推奨運用
- 統合ダッシュボード経由で 4 デモを画面切り替え
- ダブルクリックで開く `file://` よりも、`http://` 経由のほうが印刷挙動が安定（Demo 4 で重要）

```
Phase 3   システム設計書 × 4 本 → Codex 3 往復 × 4 本
   ↓
Phase 4   技術仕様書 × 4 本 → Codex 3 往復 × 4 本
   ↓
Phase 5   実装 × 4 本（並列） + E2E 動作確認
   ↓
Phase 6   統合ダッシュボード制作
   ↓
Phase 7   議事録レビュー反映（リスク表現軟化、対象拡大注記、丸め単位追加）
   ↓
🎉 動くモック 4 本 + 統合ダッシュボード 完成
```

### 5-2. Codex 3 往復レビューの中身

各ドキュメント（要件・設計・仕様）に対して以下を実施:

1. Codex に厳しめレビュー依頼（Critical / Major / Minor で分類）
2. Claude Code が指摘を Edit で反映
3. Codex が再レビュー → 残指摘
4. Claude Code が再修正
5. Codex が 3 回目レビュー → 残論点メモ
6. 次フェーズへ

ドキュメントの評価推移例（01 CSV 加工）: **C → B → B**
プロジェクトごとの合計指摘件数: **20 〜 30 件を全対応**

### 5-3. Phase 2.5「文脈寄せ」とは
- 要件定義書が「汎用的すぎる」という Codex 指摘を受け、業界用語・ユースケース・登場人物（派遣元 / 派遣先 / 派遣スタッフ / コーディネーター）を全プロジェクトで軽修正

### 5-5. 人 vs AI の役割分担

#### 人の仕事
- 最初の「こういうモック作りたい」のアイデア提示

- クラウドスタッフィング文脈寄せの方針出し

#### Claude Code の仕事

- サンプルデータ生成
- 動作確認
- README とセミナーガイド作成
- 全ドキュメントの整合性担保

#### Codex の仕事
- 厳しめ技術レビュー（設計矛盾、過剰仕様、実装不能点、セキュリティ欠陥）
- 派遣業界論点の妥当性チェック
- 各ラウンドで総合評価（S / A / B / C）を付与

### 5-6. このプロセスの再現可能性

特徴:
1. **単独 AI の自己レビュー限界を超える**: Claude と Codex の相互レビューで忖度のない指摘
2. **並列化で時間圧縮**: 4 本同時進行で直列の 1/4 程度に
3. **フェーズ承認制で暴走防止**: 各フェーズ後にユーザー承認で方向修正可能

適用できる場面:
- 新規プロジェクトの MVP 検証

- 仕様が曖昧なときの段階的明確化

## 8. 質疑応答 想定

### Q1: 学習データはどこから？
**A**: 今回は AI API を使わずルールベース実装です。同義語辞書・スコアリング・決定表で「AI らしさ」を演出しています。実運用では LLM 統合も可能ですが、まず現状の運用課題に効く粒度から始めるアプローチを採用しました。

### Q2: セキュリティは？
**A**: データはローカル完結、外部通信ゼロです。Demo 3 では PII マスキングを実装、Demo 4 はブラウザ完結で localStorage もデフォルト OFF にしています。Demo 3 ではダミーデータと実データを CLI 引数で明示的に分離するデータガード機構を設けています。

### Q3: 派遣法の解釈は？
**A**: 本デモは整備状況把握の補助ツールです。法的判断は社労士等の専門家にご相談ください。Demo 4 では派遣法免責を画面ヘッダーに固定しています。

### Q4: 導入コストは？
**A**: 本日見せたのはモックで、制作は数時間レベルです。本番化は業務の擦り合わせが本体で、技術コストは比較的軽めだと考えています。

### Q5: なぜ AI API を使わなかった？
**A**: 派遣管理業務の AI 化は「一気通貫より点で効く小さい AI を積む」方が現実的だと考えたためです。正確性が必要な領域（給与計算）はロジック、判断支援が必要な領域（要確認行抽出）はルール / 同義語辞書、最終確認は人、という設計方針です。

### Q6: 本番運用するには？
**A**: 業務擦り合わせ（運用フローと整合させる）、本番データでの動作検証、セキュリティ審査、PII 運用設計、運用保守体制が必要です。本リポジトリはセミナー実演用モックで、本番運用はサポート外です。

### Q7: Codex とは？
**A**: OpenAI の Codex CLI。Claude Code と Codex で「忖度のない相互レビュー」を回しました。3 往復 × 12 ドキュメント = 36 往復のレビューを経て品質担保しています。

### Q8: 人間は何をしたの？
**A**: 「こういうモック作りたい」のアイデア提示、各フェーズの承認、文脈寄せの方針出し、最終ラインの明示の 4 つです。コードは 1 行も書いていません。指示回数は 10 回未満です。

### Q9: モックを社内で動かしたい
**A**: GitHub の公開リポジトリ（`github.com/masaki-sigoto/cloudstaffing-ai-demos`）から clone して、`./launch-dashboard.sh` で動きます。Python 3.10+ が必要です。

### Q10: 4 つのうち最初に試すなら？
**A**: 派遣管理業務との結合度が最も高い Demo 3（勤怠チェック）を推奨します。3 者ワークフローを前提とした 10 異常パターンと担当者別通知ファイル生成が、派遣会社固有の業務に最も近い形です。
