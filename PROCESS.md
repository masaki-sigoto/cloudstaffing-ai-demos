## 各フェーズの中身

### Phase 1: 要件定義書作成（4本並列）
- 1エージェント = 1プロジェクト
- 既定テンプレート（10セクション）に沿って書く
- 派遣業界の論点（派遣法・同一労働同一賃金・抵触日・36協定）を織り込む

### Phase 2: Codex 3往復レビュー（4本 × 3往復 = 12往復）
1. Codex に厳しめレビュー依頼 → Critical/Major/Minor 指摘
2. Claude Code が指摘を Edit で反映
3. Codex が再レビュー → 残指摘
4. Claude Code が再修正
5. Codex が3回目レビュー
6. 残論点をメモして次フェーズへ

**各ラウンドの評価推移例（01 CSV加工）**: C → B → B  
**各プロジェクトで合計指摘件数**: 20〜30件を全対応

### Phase 2.5: クラウドスタッフィング文脈寄せ
要件定義が「汎用的すぎる」という指摘を受け、業界用語・ユースケース・登場人物を全プロジェクトで軽修正。  
→ 「社内誰が見ても派遣管理SaaSの話とわかる」水準に。

### Phase 3・Phase 4: 設計書・技術仕様書も同じ3往復を繰り返し
設計レベル → 実装直前レベル へと粒度を段階的に下げる。

### Phase 5: 実装（4本並列）
- 1エージェント = 1プロジェクトを src/・samples/・README・スラッシュコマンドまで一気通貫で実装
- 動作確認（smoke test）もエージェント内で実行
- 完了時点で `python3 src/main.py ...` が動く状態

### Phase 6: 統合ダッシュボード制作
- 4 本のデモを 1 画面から切り替え実行できる統合 UI を作成
- `dashboard/server.py`（Python `http.server` ベースの API サーバ）でブラウザから本物の CLI を実行
- 起動コマンド `./launch-dashboard.sh` 一発で `http://localhost:8765/` が立ち上がる
- 個別起動スクリプト（`launch-csv.sh` / `launch-rounding.sh` / `launch-attendance.sh` / `launch-yesno.sh`）とメニュー版 `launch-all.sh` も整備
- ポート 8765 はメジャー開発ポート（3000/5000/8000/8080 等）を回避するため採用

### Phase 7: 議事録レビュー反映
- 議事録レビューを受け、ドキュメント・UI 文言を全体軟化
- **リスク表現の軟化**: 過度な自動化表現は不使用とし、「AI が確認作業を支援」「最終確認は担当者」に統一（具体的な NG 文言の一覧は AGENTS.md および SEMINAR_GUIDE.md のリスク表現ガイド参照）
- **対象拡大注記**: 01 CSV 加工は「勤怠・スタッフ情報・契約データなど派遣管理運用のあらゆる CSV に応用可能」を明示
- **丸め単位追加**: 02 端数処理に `5min_round.yml` / `10min_round.yml` を追加し、計算は **真理値表ベースのロジック処理** で AI 自由判断を排することを明記
- **04 のフォーカス再強調**: 「導入時 組織登録 判断支援」というブランドバッジで用途を明示

## 人が何をしたか / AI が何をしたか

### 人の仕事
- 最初の「こういうモック作りたい」のアイデア提示
- 各フェーズの承認
- **クラウドスタッフィング文脈寄せ** の方針出し（「もっと派遣管理SaaS寄りに」）
- 「セミナーは無料、モックさえ動けばOK」という最終ラインの明示

### AI（Claude Code）の仕事
- 要件・設計・仕様の文書化
- Codex とのレビュー往復の実行・反映
- 並列サブエージェントの起動と結果統合
- 実装（Python 3本 + HTML 1本）
- サンプルデータ生成
- 動作確認
- README とセミナーガイド作成
- 全ドキュメントの整合性担保

### AI（Codex）の仕事
- 厳しめ技術レビュー（設計矛盾、過剰仕様、実装不能点、セキュリティ欠陥の指摘）
- 派遣業界論点の妥当性チェック
- 各ラウンドで総合評価（S/A/B/C）を付与

## 再現可能な方法論

このプロセスは **"AIエージェント2種の相互レビュー" + "フェーズ承認制"** の組み合わせ。

### 特徴
1. **単独AIの自己レビュー限界を超える**: Claude と Codex の相互レビューで、忖度のない指摘が出る
2. **並列化で時間圧縮**: 4本同時進行で、直列なら 4 倍かかるところを並列で処理
3. **フェーズ承認制で暴走防止**: 要件 → 設計 → 仕様 → 実装、各フェーズ後にユーザー承認で方向修正可能
4. **ドキュメント駆動でトレーサビリティ**: 要件 ID を設計・仕様まで追跡可能
5. **レビュー履歴が成果物として残る**: `review-log.md` で「どこが議論されたか」を可視化

### 適用できる場面
- 新規プロジェクトのMVP検証
- 複数案の並行検証（A案・B案を同時に作って比較）
- 仕様が曖昧なときの段階的明確化
- 社内ツール・業務補助ツールの内製

## セミナーでの使い方

### このモックそのものを見せる（主役）
`./launch-dashboard.sh` で **実際に動く統合ダッシュボード** を立ち上げ、デモを順に切り替えて実演する（[SEMINAR_GUIDE.md](./SEMINAR_GUIDE.md) 参照）。
個別の `launch-*.sh` を使えば、ダッシュボードを介さず単独実行も可能。

### 作られ方を見せる（隠し玉）
「ちなみにこの4本、**全部Claude CodeとCodexが数時間で作りました**」  
→ `docs/*-review-log.md` を開いて、実際に Codex が指摘してClaude が直していく往復を画面共有

### 結論
**「AI は実行だけでなく、**計画・レビュー・品質管理・実装**まで一気通貫で担える」** が伝わる。
