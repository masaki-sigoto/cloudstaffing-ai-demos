## 各フェーズの中身

### Phase 1: 要件定義書作成（4本並列）
- 1エージェント = 1プロジェクト
- 既定テンプレート（10セクション）に沿って書く
- 派遣業界の論点（派遣法・同一労働同一賃金・抵触日・36協定）を織り込む

### Phase 2: Codex 3往復レビュー（4本 × 3往復 = 12往復）
1. Codex に厳しめレビュー依頼 → Critical/Major/Minor 指摘
2. Claude Code が指摘を Edit で反映
3. Codex が再レビュー → 残指摘
4. Claude Code が再修正
5. Codex が3回目レビュー
6. 残論点をメモして次フェーズへ

**各ラウンドの評価推移例（01 CSV加工）**: C → B → B  
**各プロジェクトで合計指摘件数**: 20〜30件を全対応

### Phase 2.5: クラウドスタッフィング文脈寄せ
要件定義が「汎用的すぎる」という指摘を受け、業界用語・ユースケース・登場人物を全プロジェクトで軽修正。  
→ 「社内誰が見ても派遣管理SaaSの話とわかる」水準に。

### Phase 3・Phase 4: 設計書・技術仕様書も同じ3往復を繰り返し
設計レベル → 実装直前レベル へと粒度を段階的に下げる。

### Phase 5: 実装（4本並列）
- 1エージェント = 1プロジェクトを src/・samples/・README・スラッシュコマンドまで一気通貫で実装
- 動作確認（smoke test）もエージェント内で実行
- 完了時点で `python3 src/main.py ...` が動く状態

## 再現可能な方法論

このプロセスは **"AIエージェント2種の相互レビュー" + "フェーズ承認制"** の組み合わせ。

### 特徴
1. **単独AIの自己レビュー限界を超える**: Claude と Codex の相互レビューで、忖度のない指摘が出る
2. **並列化で時間圧縮**: 4本同時進行で、直列なら 4 倍かかるところを並列で処理
3. **フェーズ承認制で暴走防止**: 要件 → 設計 → 仕様 → 実装、各フェーズ後にユーザー承認で方向修正可能
4. **ドキュメント駆動でトレーサビリティ**: 要件 ID を設計・仕様まで追跡可能
5. **レビュー履歴が成果物として残る**: `review-log.md` で「どこが議論されたか」を可視化

### 適用できる場面
- 新規プロジェクトのMVP検証
- 複数案の並行検証（A案・B案を同時に作って比較）
- 仕様が曖昧なときの段階的明確化
- 社内ツール・業務補助ツールの内製

## セミナーでの使い方

### このモックそのものを見せる（主役）
4本のデモを順に実演（[SEMINAR_GUIDE.md](./SEMINAR_GUIDE.md) 参照）

### 作られ方を見せる（隠し玉）
「ちなみにこの4本、**全部Claude CodeとCodexが数時間で作りました**」  
→ `docs/*-review-log.md` を開いて、実際に Codex が指摘してClaude が直していく往復を画面共有

### 結論
**「AI は実行だけでなく、**計画・レビュー・品質管理・実装**まで一気通貫で担える」** が伝わる。
