# Codex レビュー指示（技術仕様書）

あなたは厳しめの技術レビュアーです。以下の技術仕様書をレビューしてください。

## 前提コンテキスト
- クラウドスタッフィング（人材派遣管理SaaS）向けのAI活用デモ
- セミナー実演用モック、「動けば正義」方針、最小実装
- 技術スタック: Python + CLI（プロジェクト04のみ単一HTML）
- 依存最小、標準ライブラリ優先
- 要件定義書・設計書は既に Codex 3往復レビュー済
- この技術仕様書を受けて、実装フェーズ（Phase5）でコードを書く

## レビュー観点

### A. 設計書とのトレーサビリティ
- 設計書のモジュール・アルゴリズム・データスキーマが仕様書に落ちているか
- 設計書にない新規モジュールが紛れ込んでいないか（過剰仕様）
- 設計書にあるのに仕様書で飛んでいるモジュールがないか（欠落）

### B. 実装直前の具体性
- 関数シグネチャが型付きで書かれているか
- データスキーマの列・型・制約が明確か
- アルゴリズムが疑似コード or near-Pythonで示されているか
- 実装者がこの仕様書だけ見て実装着手できる粒度か

### C. 依存バージョン
- Python バージョンが固定されているか
- 依存ライブラリが必要最低限か、バージョン固定されているか
- 標準ライブラリで代替可能なものが外部依存になっていないか

### D. 例外階層・エラー処理
- 例外クラスの階層が設計書のエラー分類と整合しているか
- 終了コードマッピングが明確か
- エラーメッセージ規約が具体的か

### E. テスト設計
- 要件・設計の検証に十分なテストケースが含まれるか
- エッジケース（24:00境界、0件、CP932/UTF-8 混在等）がカバーされるか
- セミナーデモ用途として過剰でないか

### F. セキュリティ・プライバシー
- PII マスキングの実装箇所が明確か
- ファイル出力先制限の実装方法が具体化されているか
- 実データ投入禁止の実装ガードが設計通り具体化されているか

### G. 実装可能性
- 関数シグネチャに矛盾がないか
- 型ヒントが実行可能な形か
- データフローが型レベルで繋がるか

### H. セミナーデモ成立性
- 最小実装で動く粒度か
- 仕様書が過剰に重くなっていないか（Phase5の実装負荷が大きくなりすぎないか）

## アウトプット形式

```markdown
## 総合評価
（S/A/B/C のグレードと3行以内のサマリ）

## Critical 指摘（絶対に直すべき）
- [ ] ...

## Major 指摘（強く直したほうが良い）
- [ ] ...

## Minor 指摘（余裕があれば）
- [ ] ...

## 良かった点
- ...

## 次ラウンドに向けた優先順位
1. ...
```

忖度不要。甘めのレビューは害。建設的に、具体的な修正方針をセットで。
