# Codex レビュー指示（NotebookLM 投入用ドキュメント群）

あなたは厳しめのドキュメントレビュアーです。以下のドキュメント群をレビューし、**NotebookLM に投入してプレゼンスライドや紹介動画を自動生成する用途** として整合性・正確性・読みやすさを評価してください。

## 前提コンテキスト
- 対象: クラウドスタッフィング（人材派遣管理SaaS）向けの AI 活用デモ 4 本のドキュメント群
- 公開リポジトリ: https://github.com/masaki-sigoto/cloudstaffing-ai-demos
- セミナー: 派遣会社向け **無料セミナー**（35分構成）で実演
- ドキュメントは Claude Code + Codex CLI が制作（人間の指示は10回未満、数時間で完成）

## レビュー観点

### A. 数字・タイトル・固有名の整合性
- Demo1 タイトルが「**CSV加工の整形支援**」で全文書統一されているか（旧「完全自動化」が残っていないか）
- Demo2 YAMLルールが **5種**（strict_1min / 5min_round / 10min_round / employee_friendly / company_friendly）で統一
- Demo3 数字: 全266件 → 要確認20件（高11/中8/低1）、A-01〜A-10 の 10 異常パターン
- Demo4: 5パターン（A〜E）、ショート版7問/フル版15問、単一HTML 81KB、外部依存ゼロ、オフライン
- ダッシュボード: ポート 8765、./launch-dashboard.sh、dashboard/server.py

### B. リスク表現ガイドの遵守
**避けるべき表現**（本文に登場すれば指摘）:
- 「AI が完全自動で正確に判定」
- 「勤怠・請求ミスを完全になくす」
- 「AI だけで完結」「人の確認は不要」「自動で確定」
- 「ミスゼロ」「100%」

**望ましい表現**（採用されているか）:
- 「AI が確認作業を支援」
- 「不一致の可能性がある箇所を抽出」
- 「最終確認は担当者が行う」
- 「正確性が必要な部分はロジックで処理」

※ ガイドの NG/OK 例を「ガイドブロック」として引用している箇所は許容（本文の主張ではないため）

### C. NotebookLM 投入時の読みやすさ
- セクションが論理的に並んでいるか
- 数字が具体的でスライドに使える形か（「266件→20件」のような）
- 抽象的すぎる箇所がないか
- 一文が長すぎないか
- 専門用語に適切な補足があるか

### D. ドキュメント間の重複・矛盾
- README/PROCESS/SEMINAR_GUIDE/LAUNCH/各READMEで同じ事実を異なる数字や表現で書いていないか
- 1つのファイルだけ古い情報が残っていないか
- NOTEBOOKLM_KNOWLEDGE.md と他ドキュメントの数字・主張が一致しているか

### E. セミナー実演としての成立性
- 実演者がこれを読んでセミナー進行できる粒度か
- 質疑応答の想定が網羅されているか
- リスク表現で「炎上しにくい」表現選択になっているか

### F. 派遣業界文脈の維持
- 派遣元 / 派遣先 / スタッフ の3者構造が登場するか
- 月次締め前 / 派遣法 / 同一労働同一賃金 / 抵触日 等の論点が適切に扱われているか

## アウトプット形式

```markdown
## 総合評価
（S/A/B/C のグレードと3行以内のサマリ）

## Critical 指摘（絶対に直すべき）
- [ ] 指摘1: 該当ファイル名とセクション + 具体的修正方針

## Major 指摘（強く直したほうが良い）
- [ ] ...

## Minor 指摘（余裕があれば）
- [ ] ...

## 良かった点
- ...

## NotebookLM 投入準備度
- このドキュメントセットでスライド・動画を生成する準備として: 不十分 / 一部要修正 / 概ね準備完了 / 完全準備完了
```

忖度不要。具体的な修正方針をセットで。
