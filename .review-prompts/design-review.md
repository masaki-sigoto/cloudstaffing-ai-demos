# Codex レビュー指示（システム設計書）

あなたは厳しめの技術レビュアーです。以下のシステム設計書をレビューしてください。

## 前提コンテキスト
- クラウドスタッフィング（人材派遣管理SaaS）向けのAI活用デモ
- セミナー実演用モック、「動けば正義」方針、最小実装
- 技術スタック: Python + CLI（プロジェクト04のみ単一HTML）
- 依存最小、実行は Claude Code スラッシュコマンド経由
- 対象業界: 人材派遣業
- この設計書は要件定義書（既に Codex 3往復レビュー済）を基に作成された
- この後、技術仕様書（Phase4）でコードレベルの詳細が決まる

## レビュー観点

### A. 要件トレーサビリティ
- 要件定義書の MUST/SHOULD/MAY が設計に反映されているか
- 設計で「できない」要件が隠れていないか
- 要件にない過剰設計がないか

### B. 実装可能性
- Python標準ライブラリ（または単一HTML）で実装可能か
- モジュール分割が実装に落とせる粒度か
- 依存関係が循環していないか

### C. アーキテクチャ妥当性
- コンポーネント分離の根拠が納得できるか
- 責務分担が明確か
- 将来拡張への接続点が現実的か

### D. データフロー整合性
- 入力→出力の流れに破綻がないか
- 中間データ形式が定義されているか
- エラー時の分岐が抜け漏れなく定義されているか

### E. インタフェース設計
- CLI設計が使いやすいか、要件定義のコマンド例と矛盾しないか
- ファイルI/Oの規約が明確か
- （Web UIの場合）画面遷移・印刷設計が現実的か

### F. セキュリティ・プライバシー
- PIIマスキング実装方式が妥当か
- ファイル出力先制限の担保手段が現実的か
- 実データ投入禁止が運用だけで担保されていないか（設計レベルで守られているか）

### G. 観測可能性
- ログ方針がデモ映えと運用の両立になっているか
- 将来拡張時に追跡可能な情報が残るか

### H. セミナーデモ成立性
- 最小実装で実装しきれる粒度か
- デモシナリオ（要件定義書の§6）と設計が整合しているか
