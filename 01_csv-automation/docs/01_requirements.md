# 要件定義書: CSV加工の完全自動化

## 1. プロジェクト概要

本デモは、人材派遣管理SaaS「クラウドスタッフィング」の導入運用における、派遣元企業から毎月提出される**勤怠CSV**の自動整形を題材とする。派遣業務は発注・契約・案件管理・勤怠管理・請求・評価と多段階にわたり、派遣元ごとに勤怠CSVの列順・ヘッダー名・日付形式・文字コードがバラバラで、派遣先企業・クラウドスタッフィング運用側が毎回Excel/CSV手作業で「標準スキーマ（canonical schema）」に整形している実態がある。本デモは、Claude Code のスラッシュコマンドから起動する Python CLI として、崩れた勤怠CSVを1コマンドで標準スキーマに自動整形するAIアシスト機能を実演する。セミナーでは「Before（崩れた生データ）→ After（標準スキーマに整った勤怠CSV）」のコントラストを軸に、**月次締め処理の手作業削減と差戻し削減**、および **汎用性（＝多様な派遣元の勤怠CSVに適用しやすい）** を訴求する。

> 備考: 本デモの対象は **勤怠CSVのみ**。契約CSV・請求用集計CSV・評価CSV・スタッフ情報CSV等は将来プロジェクトの接続点として想定しつつ、本回のスコープ外とする（9章参照）。

## 2. 背景と解決する課題

**現状の痛み:**
- 派遣元ごとに勤怠CSVフォーマットが違う（列順・ヘッダー名・日付表記・改行コード・文字コード）
- 紙・Excelベースの運用が残り、毎月「人の目」で修正してから取り込んでおり、担当者依存・属人化が強い
- 月初月末の締め事務が集中し、確認・承認フローの煩雑さと加工ミスによる「不備差戻し」「再送」が両社の工数を圧迫
- 新規派遣元を追加するたび、取り込みロジックをExcel関数やマクロで手動で作り直している
- 派遣特有の法令対応（36協定・抵触日等）に伴うチェック負荷が、手作業のままでは確認漏れの温床になっている

**理想像:**
- 提出CSVをそのまま投入すると、AI的推論で列名・型・日付形式を自動マッピングし、標準スキーマに変換
- 一度マッピングした派遣元は「テンプレート」として保存し、次回以降はワンクリック適用（新規受け入れのリードタイム短縮）
- 変換できなかったセル・怪しいセルは**自動ハイライト**して人間に判断を委ね、**確認漏れ・差戻しを抑止**
- 派遣先企業・クラウドスタッフィング運用チームは月次締めの取り込み工数削減の恩恵を、派遣元企業は差戻し減少と再送削減の恩恵を受ける

## 3. 対象ユーザーと利用シーン

CLI 実行主体は **派遣先企業担当者または クラウドスタッフィング CS チーム担当者** のいずれか1名。派遣元企業は「入力CSVの提供者」に限定し、本CLIは操作しない。テンプレート管理者はクラウドスタッフィング CS チーム（導入支援/運用支援）を想定。

- **派遣元企業担当者（人を出す側の経理・労務／勤怠担当／請求担当）** ＝ **入力提供者**: 自社の基幹システムやExcelから落とした勤怠CSVをそのまま提出する。本CLIは操作しない。
- **派遣先企業担当者（人を受け入れる側の総務・経理・現場管理者）** ＝ **CLI実行主体（候補1）**: 複数の派遣元から届くバラバラCSVを標準スキーマに寄せてクラウドスタッフィングに取り込む。月次の差戻し・確認漏れを減らす。
- **クラウドスタッフィング CS チーム担当者（SaaS運用側／導入支援・運用支援）** ＝ **CLI実行主体（候補2）＋テンプレート管理者**: 顧客（派遣元企業／派遣先企業）ごとの変換ルールをテンプレートとして資産化し、新規導入・受け入れのリードタイムを短縮する。

利用シーン: 月初月末の勤怠締めタイミングで、当月分の勤怠CSV（数十〜数百ファイル）を一括整形してクラウドスタッフィングに投入する。契約・請求・評価といった他業務工程への連結は将来拡張で想定する。

## 4. 機能要件

本プロジェクトのエントリポイントは Python CLI（`src/main.py`）。Claude Code のスラッシュコマンド（`.claude/commands/` 配下）経由で、ユーザーは自然言語に近い形でCLIを呼び出せる。

### 4.1 MUST（必須）

- **M1. 元データ投入**: ユーザーが任意のCSVファイルパスを指定してCLIに渡す。
  - CLI例: `python src/main.py convert --input samples/broken_01.csv --output out/clean_01.csv`
  - Claude Codeコマンド例: `/csv-convert samples/broken_01.csv`
- **M2. 自動フォーマット検出**: 投入されたCSVの文字コード（UTF-8/UTF-8 BOM/Shift_JIS）、改行コード、区切り文字を自動判定する。
- **M3. ヘッダー表記揺れの自動マッピング**: 標準スキーマ（例: `employee_id`, `name`, `work_date`, `start_time`, `end_time`, `break_minutes`, `hourly_wage`）に対し、派遣元CSVの表記揺れヘッダー（"従業員ID"/"社員コード"/"従業員番号"/"スタッフNo." 等）をAI的ルール（同義語辞書＋類似度スコア）で推論してマッピングする。
  - **決定規則**: 同義語辞書ヒット時は信頼度=1.0 として確定。辞書外は正規化（全角半角統一・記号除去）後の編集距離ベース類似度（0.0〜1.0）を算出。
  - **閾値**: 類似度 **0.80以上で自動確定**、**0.60以上0.80未満は候補として要確認フラグ**（M7）、**0.60未満は未マッピング扱い**で該当列は出力しない（レポートに「未対応ヘッダー」として記録）。
  - **タイブレーク**: 同一候補に複数スコアが並んだ場合、優先度は ①同義語辞書ヒット > ②編集距離の小さい側 > ③元CSVの左側（列インデックスが小さい方）。複数候補が閾値以上で並んだ場合は要確認化し、最終採用は **S1 `save-template` サブコマンド**（既定=自動採用、`--interactive` で対話確認、`--mapping-file` で外部指定）の採用方式に従う。
- **M4. 日付・時刻フォーマットの正規化**: 「2026/4/23」「2026-04-23」「令和8年4月23日」「R8.4.23」「20260423」などを `YYYY-MM-DD` に統一。時刻は `HH:MM` に統一（例: `9:0` → `09:00`）。
  - `24:00` は勤怠上ありうる終業値として受け付けるが、翌日 `00:00` への自動繰越は行わず「要確認」フラグを立てる。
  - 終業時刻 < 始業時刻 の場合、日またぎとして自動判定はせず「要確認」行として記録する。
- **M5. 全角半角・空白・記号の正規化**: 全角数字→半角、全角英字→半角、前後空白除去、`¥`/`￥`、`:`/`：` の混在を解消。
- **M6. 整形済CSVの出力**: 標準スキーマ順・UTF-8（BOMなし）・LF改行で出力。
  - **例外**: `--error-policy keep` の場合のみ、M8 に従い末尾に監査列 `__needs_review` を1列追加する（標準スキーマ列の順序は保持し、末尾に追記するのみ）。
- **M7. エラー箇所の自動ハイライト＋件数照合**: 変換不能・欠損・型違反のセルを **レポートファイル（既定 `<入力ファイル名>_report.md`、`--report-format csv` 指定時は `<入力ファイル名>_report.csv`）** に列挙し、**ファイル行番号（ヘッダー行=1、データ1行目=2）＋列名＋元の値＋推定理由** を出力。レポート形式は `--report-format {md,csv}`（既定 `md`）で切替可能。CLI 標準出力には件数サマリ（`input_rows / output_rows / dropped_rows / review_rows`）を必ず表示する。
  - **指標の定義**:
    - `input_rows`: 入力CSVのデータ行数（ヘッダー除く）。
    - `output_rows`: 整形済CSVに書き出されたデータ行数。
    - `dropped_rows`: 整形済CSVから除外された行数。
    - `review_rows`: 要確認フラグが立った行数（ポリシーによって出力側・除外側のどちらに属するかが変わるため、独立指標）。
  - **ポリシー別の関係式**（成立しなければCLIは **fail-fast（非ゼロ終了・ファイル未出力）**）:
    - `drop` 既定: `input_rows = output_rows + dropped_rows`、`dropped_rows ≧ review_rows`（要確認行は dropped に内包）。
    - `keep`: `input_rows = output_rows`、`dropped_rows = 0`、`review_rows ≧ 0`。
    - `fail`: `review_rows > 0` の時点で整形済CSVを書き出さず非ゼロ終了。
- **M8. エラー行ポリシー**: `--error-policy {drop,keep,fail}`（既定 `drop`）で制御する。
  - `drop`: 要確認行を整形済CSVから除外し、レポートに記録（既定）。
  - `keep`: 要確認行も出力に含め、該当セルは原値のまま。**出力CSVには `__needs_review` 列（値: `1`=要確認 / `0`=クリーン）を末尾に必ず追加** し、後段取込側が機械的にスキップ可能な形にする。
  - `fail`: 要確認行が1件でもあればCLIは非ゼロ終了かつ整形済CSVを出力しない（再提出導線用）。
  - いずれのモードでも監査項目（`input_rows / output_rows / dropped_rows / review_rows`）を標準出力およびレポート先頭に記載する。
- **M9. PII・賃金情報のマスキング**: 標準出力（S2 差分プレビュー・進捗ログ・件数サマリ含む）では、氏名は**文字列先頭1文字のみ残し以降を `***` に置換**（空白・非日本語名も機械的に同ルール。例: `山田 太郎` → `山***`、`Alice` → `A***`、`田中` → `田***`）、時給は**全桁を `*` に置換**（例: `1500` → `****`）する。**生値はローカルの出力ファイル内でのみ保持**し、標準出力・ログに生値を書き出さない。

**M7/M8/S3 の挙動マトリクス**（`dry-run` × `error-policy` × `report-format` × `batch`）:

| error-policy | dry-run | 整形済CSV出力 | レポートファイル出力 | 標準出力サマリ | 終了コード |
|---|---|---|---|---|---|
| drop | off | あり（要確認行は除外） | あり（既定 md） | 必須 | 0（関係式OK時） |
| drop | on | なし | なし（サマリ＋After先頭3行のみ） | 必須 | 0 |
| keep | off | あり（`__needs_review`列付き） | あり | 必須 | 0 |
| keep | on | なし | なし | 必須 | 0 |
| fail | off | `review_rows=0` なら出力／`>0` なら未出力 | `review_rows>0` 時は要確認一覧を出力 | 必須 | `review_rows=0` で 0／`>0` で非ゼロ |
| fail | on | なし | なし | 必須 | `review_rows=0` で 0／`>0` で非ゼロ |

バッチ（`--input-dir`）時は各ファイルに対し上記が適用され、バッチ全体の終了コードは S4 規約に従う。関係式（M7）が不成立なファイルは個別に fail-fast（非ゼロ）となり、バッチ集計にも反映される。

### 4.2 SHOULD（重要）

- **S1. 変換テンプレートの保存**: 成功したマッピング設定を `save-template` サブコマンドで `templates/<派遣元名>.json` として保存し、次回は `--template <派遣元名>` で即適用。**`convert` サブコマンドによる暗黙保存は行わない**（テンプレート化は必ず明示コマンドで実施）。
  - **採用マッピングの確定方式**（R3: 手動確認手段の明記）:
    - 既定は **非対話モード**。`save-template` はドライランで推論を行い、M3 決定規則（同義語辞書＞編集距離＞左側優先）に従い**自動で最良候補を採用**してテンプレートを書き出す。要確認（0.60〜0.80）は推論結果のまま保存され、`needs_review: true` の列は JSON 内にフラグ付きで記録される。
    - `--interactive` 指定時は **対話式で1列ずつ候補を提示し、担当者がキー入力で確定**（例: `[1] 社員コード → employee_id (0.92) ? y/n/skip`）。要確認閾値に該当する列と未マッピング列のみ確認を要求する。
    - `--mapping-file <yaml>` 指定時は、**事前に用意したYAML/JSONの対応表を正として採用**（CI・バッチ運用向け。対話なし）。
  - 命名規則: **半角英数字＋アンダースコアのスネークケース**（例: `haken_a`、`tokyo_staff_01`）。日本語・全角記号は不可。
  - 重複時: 既存ファイルがあれば `--force` なしでは上書きせずエラー終了。上書きを避けたい場合は `_v2`, `_v3` のサフィックスで派生保存可能。
  - ファイル内にスキーマバージョン（`schema_version: 1`）と生成日時を保持する。
  - CLI例: `python src/main.py save-template --input samples/broken_01.csv --name haken_a`（非対話・既定）
  - CLI例: `python src/main.py save-template --input samples/broken_01.csv --name haken_a --interactive`
  - CLI例: `python src/main.py convert --input next_month.csv --template haken_a`
  - Claude Codeコマンド例: `/csv-convert-with-template next_month.csv haken_a`
- **S2. 差分プレビュー**: 変換前後のサンプル3行を標準出力に並べて表示（Before/Afterセミナー映え用、Before側の個人情報は M9 に従いマスク表示）。
- **S3. ドライラン**: `--dry-run` で実ファイル出力せずサマリ（件数照合＋先頭3行の After プレビュー）だけ確認。
- **S4. 複数ファイル一括処理**: `--input-dir samples/` でディレクトリ内の `*.csv` を一括変換。
  - 失敗時挙動は `--continue-on-error`（既定）/ `--fail-fast` で切替可能。
  - 終了コードは「全成功=0 / それ以外=非ゼロ」が MUST（デモ用途の最低限）。**詳細コード規約（0=全成功 / 2=一部失敗 / 1=致命的または fail-fast）は MAY 扱いで、実装する場合のみこの値域に従う**（R3: 最小実装優先のため）。
- **S5. 進捗ログ**: 1ファイルごとに `[1/5] samples/broken_01.csv → out/broken_01.csv (input=10 output=9 dropped=1 review=1)` の形式で進捗を標準出力に表示する。
- **S6. クリーンアップサブコマンド（MAY）**: `python src/main.py cleanup` で `out/` 配下の整形済CSV・レポートと `samples/tmp_*` を削除。`--dry-run` で削除対象の事前確認も可能。**R3 の指摘により MAY に降格**: デモの核価値（単発変換→レポート→テンプレート再利用）の範囲外であり、README の `rm -rf out/ samples/tmp_*` をフォールバック手順として残すため必須化しない。

### 4.3 MAY（余裕があれば）

- **X1. 列ごとの信頼度スコア表示**: マッピング推論の確信度を% で表示（映え要素）。
- **X2. テンプレート推薦**: 投入CSVのヘッダー指紋から既存テンプレート候補をサジェスト。
- **X3. 簡易バリデーション**: 時給が現実的範囲か、勤務時間が24hを超えないか、等のビジネスルールチェック。
- **X4. HTMLレポート出力**: エラーハイライトをセル単位で色付けしたHTMLレビューを生成。

## 5. 非機能要件

デモ用途のため軽め。

- **処理時間目安**: 10,000行 × 7列（標準スキーマ相当）の勤怠CSVを **5秒以内** に変換。
  - ベンチ条件: macOS（Apple Silicon, メモリ8GB以上）、SSD、UTF-8入力、他プロセス軽負荷状態。
- **対応CSVサイズ上限**: 50MB / 100,000行（これを超える場合は警告表示のみ、処理は続行）。
- **文字コード対応**: UTF-8、UTF-8 with BOM、Shift_JIS（CP932）の自動判定。
  - **判定アルゴリズム（固定順）**:
    1. **BOM検査**: 先頭バイトが `EF BB BF` なら UTF-8 with BOM 確定。
    2. **UTF-8 strict デコード試行**: 全バイトを UTF-8 として `errors='strict'` でデコードし、成功すれば UTF-8 確定。
    3. **CP932 デコード試行**: 2で失敗した場合、CP932 で `errors='strict'` デコードし、成功すれば CP932 確定。
    4. **失敗**: 1〜3いずれも不成立なら非ゼロ終了（ファイル未出力、標準エラーに未対応文字コード旨を表示）。UTF-8/CP932 双方で成功する曖昧ケースは 2 が優先されるため、本順序で一意に決まる。
- **依存関係**: **外部依存なし**（Python 3.10+ の標準ライブラリのみ: `csv`, `datetime`, `re`, `argparse`, `pathlib`, `json`, `unicodedata`, `codecs`）。文字コード自動判定も **手書き判定**で実装し、`chardet` 等のサードパーティ依存は追加しない。**外部サービス/LLM APIへの通信は行わない**（オフラインで動作）。
- **実行環境**: macOS / Linux ターミナル。Claude Code のスラッシュコマンドから起動可能。

## 6. デモシナリオ（セミナー実演の流れ）

想定時間: **約4分**。

1. **[導入 30秒]** 「派遣元A社から月初に届いた勤怠CSVです」とファイルを開いて見せる。ヘッダーが "社員コード / 氏名 / 勤務日（令和表記） / 始業：時刻 / 終業：時刻 / 時給（全角¥）" と崩れていることを強調し、**これを派遣先・CSチームが毎月手作業でクラウドスタッフィングに取り込んでいる** と業務文脈を添える。
2. **[Beforeショック 30秒]** 「これを毎月、月初月末の締めタイミングで手で直してます」と既存Excel手作業の悲惨さを語る。空セル、BOM、Shift_JIS混在も見せ、**確認漏れ・差戻しの温床** であることを強調。
3. **[コマンド実行 20秒]** ターミナルで `/csv-convert samples/timesheet_202604_haken_a.csv` を実行。S5 進捗ログと件数サマリ（input/output/dropped/review）が流れる。
4. **[After披露 60秒]** 生成された `out/timesheet_202604_haken_a.csv` を開き、ヘッダーが標準スキーマに揃い、日付・時給・空白が全部整っていることを見せる。氏名・時給は標準出力ではマスク表示だがファイルでは生値である点を軽く補足。
5. **[エラーハイライト 30秒]** `out/timesheet_202604_haken_a_report.md` を開き、「3行目の勤務日は『令和8年13月』で不正なためハイライトしました」のようなAIの気配りを見せる。**差戻しの前に気づける** 見せ方を意識。
6. **[テンプレート保存 20秒]** `/csv-save-template samples/timesheet_202604_haken_a.csv haken_a` で設定を保存。
7. **[翌月の再利用 30秒]** 「翌月の締めはこう」と別ファイルを `/csv-convert-with-template timesheet_202605_haken_a.csv haken_a` で一発変換。**数秒で終わる** ことを強調。
8. **[締め 20秒]** 「**主要な派遣元パターンに即応、残りはテンプレートで資産化**」と、月次締めの手作業削減・差戻し削減というクラウドスタッフィング運用価値に着地（成功基準の93%目標と整合）。

## 7. 前提・制約

- 本デモはモックであり、実運用グレードのバリデーション・監査ログ・権限制御は実装しない。
- LLM APIは呼ばない（オフライン・完結型）。ヘッダー類似度判定は**同義語辞書＋編集距離＋ルールベース**で擬似的に「AIっぽく」見せる。
- 標準スキーマ（canonical schema）は勤怠CSVを想定して1種類固定（`employee_id`, `name`, `work_date`, `start_time`, `end_time`, `break_minutes`, `hourly_wage`）。請求・社保等への拡張はスコープ外。
- サンプルデータは個人情報を含まないダミーを自動生成。
- Windows対応は保証しない（macOS/Linuxターミナル前提）。

**個人情報・賃金情報の取り扱い（デモ用最低限統制）:**
- 標準出力・進捗ログ・差分プレビュー（S2）は M9 に従いマスク表示。**生値を標準出力・ログに出力しない**。
- 整形済CSV・レポートファイルの既定出力先は **`out/` ディレクトリ配下**（リポジトリ内ローカル、Git管理対象外）。`--output` / `--output-dir` で別ディレクトリを明示指定した場合のみ他の場所に出せる（明示しない限り `out/` の外に生成されない）。
- デモ終了後のクリーンアップは **S6 `cleanup` サブコマンド** 経由を推奨（README の `rm -rf out/ samples/tmp_*` はフォールバック手順として残す）。
- サンプル入力は個人情報を含まないダミーのみを扱う。本番データをこのデモCLIに流さない。

**標準スキーマの必須/任意定義:**

| 列名 | 必須 | デフォルト補完 | 備考 |
|---|---|---|---|
| `employee_id` | 必須 | なし | 空なら要確認行 |
| `name` | 必須 | なし | 空なら要確認行 |
| `work_date` | 必須 | なし | 不正日付は要確認行 |
| `start_time` | 必須 | なし | `HH:MM` に正規化できなければ要確認 |
| `end_time` | 必須 | なし | 同上。`end_time < start_time` は要確認 |
| `break_minutes` | 任意 | `0`（空欄時は 0 分として補完） | 数値変換不可は要確認 |
| `hourly_wage` | 必須 | なし | 数値変換不可は要確認 |

**業界論点の扱い:**
- 本デモは **36協定・同一労働同一賃金・抵触日等の法令判定は行わない**。変換後データを受け取るSaaS側の責務とする。
- 将来拡張の接続点としては、`hourly_wage` の不当低額チェック、`work_date` の抵触日超過フラグ等を想定するが、本プロジェクトでは実装しない。

## 8. 成功基準

- セミナー本番で、**1コマンド実行から5秒以内**（上記ベンチ条件下）に整形済CSVと差分レポートが出力される。
- 観客の目視で「ヘッダーがバラバラのBefore」と「整ったAfter」のコントラストが明確に伝わる。
- テンプレート保存→翌月一発適用の流れが止まらずに流れる（再実行時のエラーゼロ）。
- **ヘッダーマッピング正解率**: サンプルパターンA/B 合計14列（7列×2）中、**13列以上を標準スキーマに正しくマッピング**できる（正解率 93% 以上）。
- **エラー検出率**: サンプルパターンC の想定エラー **4件のうち、3件以上をレポートに検出**できる（R3 整合修正: 7章の必須/任意定義により `break_minutes` 空欄は補完対象なので実質4件）。内訳は 10.4 の `work_date 不正月` / `name 空欄` / `end_time < start_time` / `hourly_wage 数値変換不可` の4件。
- **件数照合**: ポリシー別の関係式（M7 参照）が常に成立する。成立しない場合は **CLIが fail-fast（非ゼロ終了・ファイル未出力）** となり、壊れた結果をそのまま出すことはない。
- 質疑で「他の派遣元CSVでも動きますか？」と聞かれたら、その場で別サンプル（事前用意）を投入して動かせる。

## 9. スコープ外（今回やらない）

- 実LLM API連携（OpenAI/Anthropic等）
- SaaS本体への自動取り込み（出力CSVを生成するところまで）
- 認証・権限・監査ログ
- Web UI / ブラウザ画面（あくまでCLIデモ）
- **請求書CSV・社保届・年末調整等、勤怠以外のフォーマット**（別プロジェクトで扱う想定）
- 36協定・同一労働同一賃金・抵触日等、派遣業界法令関連の判定ロジック
- 大規模データ（100,000行超）のストリーミング処理最適化
- 派遣元への自動フィードバック送信

## 10. 想定入出力例

### 10.1 標準スキーマ（After形）

```csv
employee_id,name,work_date,start_time,end_time,break_minutes,hourly_wage
EMP001,山田 太郎,2026-04-23,09:00,18:00,60,1500
EMP002,佐藤 花子,2026-04-23,10:00,19:00,60,1600
```

### 10.2 崩れCSV パターンA（列順逆転＋ヘッダー揺れ＋令和表記）

投入 `samples/timesheet_202604_haken_a.csv`（派遣元A社が自社Excelから出力した勤怠データ想定。ヘッダー揺れ例: 社員コード / 従業員ID / 従業員番号 / スタッフNo. / スタッフコード 等）:

```csv
時給,終業,始業,勤務日,氏名,社員コード,休憩（分）
"￥1,500",18:00,9:00,令和8年4月23日,山田　太郎,EMP001,60
"￥1,600",19:00,10:00,R8.4.23,佐藤　花子,EMP002,60
```

→ 変換後は 10.1 と同じ。

### 10.3 崩れCSV パターンB（全角半角混在＋BOM＋Shift_JIS）

投入 `samples/timesheet_202604_haken_b.csv`（派遣元B社提出、Shift_JIS, BOMあり ※通常 Shift_JIS では BOM は稀だが、本サンプルは BOM 検出と文字コード判定の両方をデモで見せるため意図的に同居させている）:

```csv
従業員ID,ﾅﾏｴ,日付,開始,終了,休憩,時給
ＥＭＰ００１,山田太郎,2026/4/23,9：00,18：00,1時間,"1,500円"
EMP002, 佐藤花子 ,2026-4-23,10時,19時,60分,￥1600
```

→ 変換後:
```csv
employee_id,name,work_date,start_time,end_time,break_minutes,hourly_wage
EMP001,山田太郎,2026-04-23,09:00,18:00,60,1500
EMP002,佐藤花子,2026-04-23,10:00,19:00,60,1600
```

### 10.4 崩れCSV パターンC（エラー混入）

投入 `samples/timesheet_202604_haken_c.csv`（派遣元C社提出、締め直前の不備混入を想定）:

```csv
社員コード,氏名,勤務日,始業,終業,休憩,時給
EMP001,山田太郎,2026-04-23,09:00,18:00,60,1500
EMP002,佐藤花子,令和8年13月5日,10:00,19:00,60,1600
EMP003,,2026-04-23,09:00,08:00,60,1500
EMP004,鈴木一郎,2026-04-23,09:00,18:00,,abc円
```

→ 変換後 `out/timesheet_202604_haken_c.csv`: 既定 `--error-policy drop` で要確認行を除外し、正常行のみ出力。
→ エラーレポート `out/timesheet_202604_haken_c_report.md`:

```markdown
# 変換レポート: timesheet_202604_haken_c.csv

- input_rows: 4（ヘッダー除く）
- output_rows: 1
- dropped_rows: 3
- review_rows: 3
- 関係式チェック: drop ポリシー `input=output+dropped` → 4 = 1 + 3 ✓（`dropped ≧ review` → 3 ≧ 3 ✓）

（行番号はファイル行番号。ヘッダー行=1 とし、データ1行目=2 から始まる）

| 行 | 列 | 元の値 | 推定理由 |
|---|---|---|---|
| 3 | work_date | 令和8年13月5日 | 月が不正（13月は存在しない） |
| 4 | name | (空) | 必須項目が空欄 |
| 4 | end_time | 08:00 | 終業が始業より前（日またぎは自動補正せず要確認） |
| 5 | hourly_wage | abc円 | 数値に変換できない |
```

注: `break_minutes` は任意項目のため空欄はエラー扱いせず `0` 分として補完される（7章「標準スキーマの必須/任意定義」参照）。上記レポートでも `EMP004` 行は `break_minutes` では要確認にならず、`hourly_wage` のみ要確認となる。

### 10.5 CLI使用例まとめ

```bash
# 単発変換（例: 派遣元A社の2026年4月勤怠CSV）
python src/main.py convert --input samples/timesheet_202604_haken_a.csv --output out/timesheet_202604_haken_a.csv

# テンプレート保存（既定: 非対話、自動採用）
python src/main.py save-template --input samples/timesheet_202604_haken_a.csv --name haken_a

# テンプレート保存（対話式で1列ずつ確認）
python src/main.py save-template --input samples/timesheet_202604_haken_a.csv --name haken_a --interactive

# テンプレート保存（外部マッピングファイルを正として採用）
python src/main.py save-template --input samples/timesheet_202604_haken_a.csv --name haken_a --mapping-file mappings/haken_a.yml

# テンプレート適用（翌月分）
python src/main.py convert --input samples/timesheet_202605_haken_a.csv --template haken_a --output out/timesheet_202605_haken_a.csv

# ディレクトリ一括（既定は --continue-on-error）
python src/main.py convert --input-dir samples/ --output-dir out/

# バッチで1件でも失敗したら即停止したい場合
python src/main.py convert --input-dir samples/ --fail-fast

# ドライラン
python src/main.py convert --input samples/timesheet_202604_haken_b.csv --dry-run

# デモ後クリーンアップ（out/ とsamples/tmp_*を削除）
python src/main.py cleanup
python src/main.py cleanup --dry-run

# エラー行ポリシー指定（既定は drop）
python src/main.py convert --input samples/timesheet_202604_haken_c.csv --error-policy keep
python src/main.py convert --input samples/timesheet_202604_haken_c.csv --error-policy fail

# レポート形式の切替（既定は md）
python src/main.py convert --input samples/timesheet_202604_haken_c.csv --report-format csv
```
