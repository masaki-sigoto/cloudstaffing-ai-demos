### Critical 指摘

- [x] **CSVサンプルがCSVとして壊れている**: 10.2/10.3 の `￥1,500`・`1,500円` が未クォートで列崩れします。**修正方針**: カンマ入り値は必ずダブルクォート、または区切りを `;` に変更。サンプルをパーサで実際に読める形に固定。
- [x] **スコープの自己矛盾**: 1章で「請求・勤怠CSV」と言いながら、7章で「勤怠1種類固定」。さらに「ほぼ全派遣元に適用」と主張。**修正方針**: 対象を「勤怠CSVのみ」に明示統一し、請求は将来拡張として分離。
- [x] **エラー行の扱いが未定義で業務的に危険**: 4.1 M7 と 10.4（正常行のみ出力）が整合していない。欠損行を落とすなら件数照合・再提出導線が必須。**修正方針**: `--error-policy {drop,keep,fail}` を MUST 化し、既定値と監査項目（入力件数/出力件数/除外件数）を定義。
- [x] **個人情報・賃金情報の最低限統制が欠落**: レポート/標準出力に生値を出す設計（M7）があるが、マスキング・保存期間・アクセス前提がない。**修正方針**: デモでも「標準出力はマスク表示」「詳細はローカルファイルのみ」「保存先固定と削除手順」を明文化。

### Major 指摘

- [x] **MUST/SHOULDの検証可能性が弱い**: 「AIっぽく」「汎用性」「数秒」など測定不能語が混在。**修正方針**: 受入条件を数値化（例: ヘッダー推定正解率、処理時間測定条件）。
- [x] **依存方針が曖昧**: 5章で「標準ライブラリ中心」だが `chardet相当` の扱いが不明。**修正方針**: 「外部依存なし」か「`chardet` を唯一許可」のどちらかに固定。
- [x] **行番号定義が不明**: M7 の行番号がヘッダー含む/含まないで運用差異が出る。**修正方針**: 「ファイル実行番号（ヘッダー=1）」等に統一。
- [x] **デモシナリオと要件の結合が弱い**: 進捗ログ表示、テンプレート保存コマンドの入出力仕様が要件側で不足。**修正方針**: シナリオで使う表示を MUST/SHOULD に落とす。
- [x] **3者の実運用責任境界が曖昧**: 誰がCLIを実行し、テンプレートを管理するか不明。**修正方針**: 主体（派遣先/CS）を1つに決め、派遣元は「入力提供者」に整理。
- [x] **業界論点の扱いが薄い**: 36協定・同一労働同一賃金・抵触日への言及がない。**修正方針**: 「本デモでは法令判定しない」明示と、将来拡張の接続点だけ記載。
- [x] **性能要件の測定条件不足**: 10,000行5秒の環境条件が未定。**修正方針**: CPU/メモリ/ファイル特性のベンチ条件を1行で規定。
- [x] **出力仕様の揺れ**: `*_errors.csv` or `*_report.md` が選択制で利用側が困る。**修正方針**: 既定フォーマットを1つに固定し、もう一方は `--report-format` で選択。

### Minor 指摘

- [x] **用語統一**: 「標準形式」「canonical schema」「取り込み形式」が混在。1語に統一。
- [x] **時刻正規化ルールを明示**: `9:0`、`24:00`、日またぎの扱いを明文化。
- [x] **テンプレート命名規則不足**: `<派遣元名>.json` の重複・改名・バージョン管理規約がない。
- [x] **デモ成功基準の演出依存**: 「AIが賢く拾ってくれた感」は主観的。客観KPIに置換推奨。

## ラウンド2 (2026-04-23)

### 総合評価

**B**
前回指摘を反映して具体性は大きく改善しているが、**件数照合ロジックと出力方針に重大な矛盾**が残っている。
このまま実装に入ると、CLIの終了条件とデモの見せ方がぶれて再現性を落とすリスクが高い。
最小実装としては有望であり、定義の整流化でAに届く、との評価。

### Critical 指摘

- [x] **件数照合の定義が自己矛盾**（4.1 M7/M8、8章、10.4）: `入力=出力+除外+要確認` と書かれている一方、`drop` では要確認行は除外に内包され、10.4実例で `4=1+3+3` で破綻。**修正方針**: 指標を `input_rows / output_rows / dropped_rows / review_rows` に分離し、関係式をポリシー別に明記。
- [x] **件数不整合時の挙動が矛盾**（4.1 M7 と 8章）: M7は「件数照合が取れない結果は出力しない」、8章は「ずれた場合はCLIが警告」で衝突。**修正方針**: `fail-fast（非ゼロ終了・ファイル未出力）` に統一。
- [x] **出力先ポリシーが矛盾**（4.1 M1/M6、10.5 と 7章）: M1/10.5は `--output` 指定可能、7章は「out/ 固定」。**修正方針**: 7章を「既定は out/ 配下、`--output`/`--output-dir` で上書き可能」に修正。

### Major 指摘

- [x] **マッピング推論の決定規則が不足**（4.1 M3）: 同点時・低信頼時・複数候補時のルール未定義。**修正方針**: スコア閾値・タイブレーク・未確定時の要確認化を明文化。
- [x] **必須/任意カラム定義が不十分**（M7、10.4）: 必須列一覧が不在。**修正方針**: 標準スキーマ7列の必須・許容欠損・デフォルト補完可否を表で定義。
- [x] **`--error-policy keep` の品質保証が弱い**（M8）: 原値保持のみでは後段取込で落ちるリスク。**修正方針**: `keep` 時に `__needs_review` フラグ列を必須追加。
- [x] **バッチ処理の失敗時仕様が未定義**（S4/S5）: 1ファイル失敗時の挙動・終了コード未定義。**修正方針**: `--continue-on-error`（既定）/`--fail-fast` を追加し終了コード規約を明記。
- [x] **デモ訴求文が過大**（6章 Step8）: 「ほぼ全派遣元にそのまま適用」が8章93%要件と緊張関係。**修正方針**: 「主要パターンに即応、未対応はテンプレートで吸収」にトーンダウン。
- [x] **プライバシー統制が手動依存**（7章）: クリーンアップがREADME手順のみ。**修正方針**: `cleanup` サブコマンドをSHOULDに追加。

### Minor 指摘

- [x] **「ファイル実行番号」表現が曖昧**（M7、10.4）: **修正方針**: 「ファイル行番号（ヘッダー=1）」に統一。
- [x] **実演時間の積算が4分10秒**（6章）: **修正方針**: 想定時間を「約4分」に寄せる。
- [ ] **`Shift_JIS, BOMあり` の例が不自然**（10.3）: 対応見送り。セミナー会場で両方の"崩れ感"を1サンプルで見せる演出上の判断。本文に「※Shift_JIS の BOM は稀だが混在検出のデモ用に同居させている」注記を追記で対応。

### 良かった点

- MUST/SHOULD/MAYの層分けが明確で、実装優先度に落とし込みやすい。
- 派遣元/派遣先/CS運用の役割分離が明文化され、デモ責任範囲が整理されている。
- 法令論点（36協定・同一労働同一賃金・抵触日）をスコープ外として明示し、過剰実装を防げている。
- 標準出力マスキング要件（M9）を先に固定している点は、デモでも実務感がある。

### このラウンドでの対応

- **Critical: 件数照合の自己矛盾** → M7/M8/8章/10.4 を一体修正。指標を `input_rows / output_rows / dropped_rows / review_rows` に整理し、`drop` は `input=output+dropped, dropped≧review`、`keep` は `input=output, dropped=0, review≧0`、`fail` は `review>0 ならファイル未出力で非ゼロ終了` と関係式をポリシー別に明記。10.4の数値も `4=1+3, dropped=3, review=3` に修正。
- **Critical: 件数不整合時の挙動矛盾** → M7 を「件数照合が成立しない場合は fail-fast（非ゼロ終了・ファイル未出力）」に統一、8章の該当条項も「成立しない場合はCLIが非ゼロ終了（ファイル未出力）」に合わせた。
- **Critical: 出力先ポリシー矛盾** → 7章を「既定は out/ 配下、`--output`/`--output-dir` で明示指定した場合のみ他ディレクトリ可」に修正。
- **Major: マッピング決定規則不足** → M3 に類似度閾値（≥0.8 確定 / 0.6〜0.8 候補＋要確認 / <0.6 未確定）、同点タイブレーク（同義語辞書＞編集距離＞左側優先）、複数候補時は要確認化を追記。
- **Major: 必須カラム定義不十分** → 7章 or 10章冒頭に「標準スキーマ必須/任意表」を追加。`employee_id, name, work_date, start_time, end_time` を必須、`break_minutes` を任意（未記入は 0 補完）、`hourly_wage` を必須として整理。
- **Major: keep 時の品質保証** → M8 の `keep` 挙動に `__needs_review` フラグ列の追加を必須化。
- **Major: バッチ失敗時仕様** → S4 に `--continue-on-error`（既定）/`--fail-fast` と終了コード規約（0=全成功, 2=一部失敗, 1=全体失敗/致命的）を追記。
- **Major: デモ訴求文の過大表現** → 6章 Step8 を「主要パターンに即応、残りはテンプレートで資産化」に修正。
- **Major: プライバシー統制の手動依存** → S6 として `cleanup` サブコマンド（`out/` と `samples/tmp_*` を削除）を追加。
- **Minor: 行番号表現** → M7・10.4 を「ファイル行番号（ヘッダー行=1）」に統一。
- **Minor: 実演時間** → 6章の想定時間を「約4分」に修正。積算が4分10秒 → Step6 を20秒に短縮。
- **Minor: Shift_JIS+BOM の例** → 見送り。セミナーの"崩れ感"演出を優先し、注記で補足対応のみ。

## 最終残論点

R3で直しきれなかった／意図的に残置した論点。実装・デモでは下記を前提として運用する。

1. **S4 詳細終了コード規約（0/2/1）を MAY に降格したこと** → 実装者判断で「全成功=0／失敗=非ゼロ」の最小実装でも要件満たす。バッチの詳細運用（CI連携等）は将来拡張で。
2. **S6 cleanup サブコマンドを MAY に降格したこと** → デモ終了後の掃除は README の `rm -rf out/ samples/tmp_*` に戻る。セミナー運用上は許容可能と判断。
3. **Shift_JIS + BOM のサンプル（10.3）** → R2 から継続で注記対応のみ。セミナー演出優先の意図的残置。
4. **`save-template --mapping-file` の YAML/JSON スキーマ詳細** → 本要件ではフォーマットの存在のみ規定し、具体的なキー名やバリデーション規則は実装時に詰める（最小実装前提のため）。
5. **`--interactive` モードの画面設計** → CLI プロンプト文言は実装者裁量。要件としては「列名・候補・信頼度・y/n/skip 選択を出す」レベルの記載に留める。

以上をもって要件定義書は **実装着手可能な状態（承認可能）** と判断する。
