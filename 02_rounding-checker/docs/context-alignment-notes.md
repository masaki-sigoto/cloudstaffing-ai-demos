# クラウドスタッフィング文脈寄せ修正メモ

## 日時: 2026-04-23

## 寄せ方針
- 人材派遣管理SaaS（クラウドスタッフィング）の **給与・請求・勤怠時間計算** の論点に寄せる
- **月次締め・誤請求防止・説明責任（派遣先への金額説明 / スタッフへの給与説明）** の観点を補強
- 登場人物は **派遣元（請求担当・給与担当・営業）／派遣先／派遣スタッフ／CS導入支援** の4系統に整理
- 本質機能仕様（真理値表・amount_rounding・金額計算式・MUST/SHOULD/MAY・成功基準数値）は変更しない
- Codex r3 承認済みのため軽修正に留め、Edit 差分のみで対応

## 修正箇所と内容

### 1. プロジェクト概要
- **変更前**: 「派遣業務における勤怠の端数処理ルールを…CLIツール」「派遣元・派遣先・現場スタッフの三者間で認識齟齬…」
- **変更後**: 「クラウドスタッフィング（人材派遣管理SaaS）の勤怠・請求・給与計算で避けて通れない端数処理ルールを…CLIツール」「派遣元の請求担当・給与担当が…請求書・給与明細の金額について…逆算で説明」「契約登録時の設定レビュー、月次締め前の誤請求防止、派遣先への金額根拠説明の3シーンをカバー」
- **意図**: CS上のルール設定が請求・給与計算の根拠値になるという業務連動を明示。ユーザーの主戦場（請求担当・給与担当）を前面化。

### 2.1 業界の痛点
- **変更前**: 4人のよくある言い分（営業 / 派遣先 / スタッフ / 営業）
- **変更後**: 導入文に「CS上の案件設定がそのまま月次請求・給与計算の根拠値になるため、設定ミスが金額誤差として直接現れる」を追加。登場人物を **請求担当 / 給与担当 / 派遣先 勤怠担当 / 派遣スタッフ / CS導入支援** の5行に再構成
- **意図**: SaaSの設定値が金額にダイレクトに効く構造を説明。

### 2.2 典型的なトラブル事例
- **変更前**: 請求vs給与の不一致、説明に時間、設定ミス後追い、法定内外の混同 の4項
- **変更後**: 5項に増補
  - 「請求 vs 給与の不一致」に「月次締め担当の原因調査負荷」を追記
  - 「月次締め後の誤請求発覚」追加（請求書発行後の訂正・謝罪対応）
  - 「派遣先への説明責任」追加（金額根拠の属人化）
  - 「給与ズレによるスタッフ信頼低下」追加（派遣元の評価・定着率への影響）
  - 「法定内外の混同」は据え置き
- **意図**: 月次締め・説明責任・定着率（評価）という派遣管理SaaSの主要KPIに直結する課題を補強。

### 3.1 一次ユーザー
- **変更前**: 営業担当 / 管理部・経理 / 派遣先 勤怠担当 / 導入コンサル の4行
- **変更後**: **請求担当 / 給与担当 / 営業担当 / 派遣先 勤怠担当 / CS導入支援** の5行。請求・給与担当を筆頭に配置
- **意図**: 「派遣元の請求担当・給与担当が使う道具」をユーザー指示どおり先頭に据える。

### 3.3 代表的な利用シーン
- **変更前**: 営業Aさんの「新規契約前の口頭合意確認」シナリオ
- **変更後**: 請求担当Aさんの **月次締め前** シナリオ。CS上の3案件分打刻データをCSV抜き出し→比較→締め前是正
- **意図**: 「月次締め前の誤請求防止」という主要バリューを具体シーンで表現。

### 6. デモシナリオ（基準シナリオ部分・ステップ0・ステップ5のみ）
- **基準シナリオ注記**: 「派遣元 請求担当が、月次締め直前にCS上の案件（派遣先: 架空のOFFICE-A社、スタッフ: EMP001）について、丸めルール設定が契約どおりか確認する状況」を追記
- **ステップ0 導入トーク**: 「派遣先への請求額と、このスタッフへの給与額はいくらになるでしょう？ CS上の丸め設定次第で月約18,000円ブレます」に修正。請求/給与の両サイド効果を明示
- **ステップ5 クロージング**: 「月次締め前・契約前にシミュレーション」「派遣先への金額説明・スタッフへの給与説明が変わる」へ修正
- **意図**: 数値・コマンド・真理値表は一切変えずに、ナレーションだけをCS文脈に寄せる。

### 10.2 打刻データ例（CSV）
- **変更前**: CSVのみ
- **変更後**: CSVの前に想定コンテキスト注記を追加（「案件 CASE-OFFICE-A-202604、派遣先: 架空のOFFICE-A社、時給1,800円、丸め単位15分 に紐づく EMP001 の月次打刻ログ抜粋」）。氏名・派遣先実社名・案件名は §5.1 / §7.1 によりCSV必須列に入れずメタ情報として運用側管理と明記
- **意図**: 派遣業界の現場データっぽさを出しつつ、実データ禁止・匿名ID必須・CSV 4列固定の MUST 仕様を破らない。

## 触らなかった箇所とその理由

|---|---|
| 2.4 丸め方向の真理値表 | 本質仕様。ユーザー指示で「本質機能仕様は変えない」と明示。 |
| 2.5 用語定義 | 仕様の土台。変更すると全章に波及。 |
| 4. 機能要件（M1〜M5、S1〜S5、W1〜W5） | MUST/SHOULD/MAY は Codex r3 で合意済み。機能スコープは現状維持。 |
| 5. 非機能要件 / 5.1 データ取扱最低ライン | 性能・免責・実データ禁止等は本質制約。寄せ修正対象外。 |
| 6. デモシナリオ ステップ1〜4のコマンド・出力数値 | 「再現可能な数字」として成功基準に紐づく。変更不可。 |
| 7. 前提・制約 / 7.1 入力仕様 / 7.2 異常系 | 実装契約。変更すると真理値表・計算式にも影響する。 |

| 9. スコープ外 | 36協定・同一労働同一賃金・抵触日等の非対象明示は法的リスク境界線。変更不可。 |
| 10.1 YAML ルール定義 / 10.3 CLIコマンド / 10.4 スラッシュコマンド対応表 / 10.5〜10.8 出力サンプル | 仕様そのもの。amount_rounding・数値例・フォーマットを含むため変更しない。 |
