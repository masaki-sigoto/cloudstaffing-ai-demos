# Codex レビュー履歴: システム設計書（02_rounding-checker）

## ラウンド1 (2026-04-23)

### 総合評価
**C**
要件の写経精度は高く、モジュール分割も実装しやすい。
一方で、**金額丸めの実装方針**と**警告/非対応機能の扱い**に、要件逸脱または誤計算リスクが残っていた。
実装に入る前に、デモ数値の信頼性（float 排除）と警告ポリシーの要件整合を修正する必要があった。

### Critical 指摘
- [x] 指摘1: `half_up` を float で実装する設計は誤計算リスク（§7.6 / 付録C.2）
  - 修正方針: `floor(raw + 0.5)` は浮動小数誤差で境界値を誤る可能性あり。整数演算（`(hourly_yen * minutes + 30) // 60` 等）で `floor/half_up/ceil` を厳密実装へ変更。
- [x] 指摘2: `--quiet` で行スキップ警告まで抑制する設計が要件 M2 と衝突（§6.1.1 / §6.3 / §9.4）
  - 修正方針: `--quiet` は免責表示と INFO のみ抑制し、WARN（特に入力スキップ警告）は常時 stderr 出力とする。表・文言を修正。
- [x] 指摘3: `overtime` ブロックの「読み捨て・警告なし」は非対応機能の隠蔽（§5.2 / 付録C.3）
  - 修正方針: `overtime` キー検出時に WARN を必須表示（「このビルドでは `overtime` は未適用」）。SHOULD 機能の存在を可視化する。
- [x] 指摘4: compare 時の YAML `amount_rounding` 無視警告ロジックが要件を満たしていない（§7.5 擬似コード / §4.2）
  - 修正方針: 「任意ルールの YAML 値が有効 `amount_rounding` と異なるなら必ず1回 stderr 警告」に単純化。CLI 指定時も同様に判定。

### Major 指摘
- [x] 指摘5: `explain` の出力形式設計が内部矛盾（§4.3「text 専用」 vs §6.1.1「共通 `--format`」）
  - 修正方針: `explain` は `--format text` のみ許可と CLI 層で明示エラー化する旨を §6.1.4 に明記。
- [x] 指摘6: 入力ソースの排他制約が未定義（§6.1.2）
  - 修正方針: `--punch` と `--punch-file` を argparse の mutually exclusive group で排他化する旨を明記。
- [x] 指摘7: PII 対策が `employee_id` 正規表現依存に偏りすぎ（§9.1）
  - 修正方針: 禁止列名検知（`name` / `employee_name` / `client_name` 等）で WARN または exit 2 を §5.3 と §9.1 に追加。
- [x] 指摘8: `net_min = max(0, ...)` の無警告丸めは異常検知を潰す（§7.2）
  - 修正方針: 休憩過大で 0 化される場合に WARN を出力し、原因追跡可能にする。
- [ ] 指摘9: `validate` サブコマンドが要件外・最小実装方針に対する根拠が弱い（§6.1）
  - 対応状況: 見送り。`validate` は CS オンボーディング支援として §10.4 に利用用途が既に記載されており、実装コストも極めて小さい（YAML 読込＋スキーマ検証のみ）ため残置。Phase4 後送は採用しない。

### Minor 指摘
- [ ] 指摘10: compare 基準ルールの扱いが曖昧（§6.1.3 `--baseline`）
  - 対応状況: 見送り（セミナーデモ範囲では `rules[0]` 既定で十分。将来のラウンドで再検討）。
- [ ] 指摘11: `--out` の cwd 判定を文字列比較に依存（§9.2）
  - 対応状況: 見送り（実装寄りの議論。技術仕様書フェーズで `Path.is_relative_to` 相当の頑健な実装へ切替）。
- [ ] 指摘12: 警告文言の業務影響表現が一部強すぎる（§10.5 相当）
  - 対応状況: 見送り（デモ用途に最適化済み。運用利用は現状スコープ外）。
- [ ] 指摘13: 性能要件の測定設計が不足（§10.4）
  - 対応状況: 見送り（ベンチコマンドは実装後に別途整備。設計書レベルで `100件/1秒` 目標は既に明記済み）。

### 良かった点
- 要件→設計の対応表が明確で、レビュー/実装の起点として使いやすい。
- 責務分離（`punch_parser` / `rounding` / `payroll` / `formatter`）は妥当で、循環依存も起きにくい。
- compare の「共通 `amount_rounding` 適用」という公平性思想は業務説明性に直結していて良い。

### このラウンドでの対応
- Critical 4件すべて対応（金額計算の整数演算化、`--quiet` の警告非抑制、`overtime` WARN 化、compare 警告ロジック単純化）。
- Major 5件のうち4件対応（`explain` の `--format` 制約明記、入力ソース排他、PII 列名検知、休憩過大時 WARN）。残り1件（`validate` 後ろ倒し）は見送り。
- Minor 4件は見送り（セミナーデモ用途・最小実装前提、実装フェーズで再検討）。
- 併せて付録 C.2 / C.3 の決定記録を最新方針に合わせて更新。

## ラウンド2 (2026-04-23)

### 総合評価
**B**（R1 の **C** から1段階改善）
MUSTの実装骨格はほぼ揃い、R1 Critical/Major も反映済み。
一方で境界条件（24:00）・比較仕様（`--break`未指定時）・PII 防御の「allowlist 化」など、Phase4 で解釈が割れるリスクのある点が残っていた。R2 ではこれらを設計レベルで閉じ、実装者の判断余地を最小化した。

### Critical 指摘
- [x] 指摘1: 丸め後時刻の上限未定義で 24:00 境界が破綻しうる（§5.1 / §7.1）
  - 修正方針: 内部レンジを「入力 0..1439 / 丸め結果 0..1440」に拡張。1440 は `"24:00"` 特殊値として扱い、`clock_in` 側の 1440 は exit 2 で拒否。`HH:MM` 再変換ルールも固定。付録 C.7 を追加。
- [x] 指摘2: compare の休憩控除ポリシー未確定（§4.2 / §6.1.3 / §7.5）
  - 修正方針: `--break` 未指定時は **全ルール break=0 を共通適用**と確定。YAML `break_minutes` との不一致時は [WARN] 1回。要§4.1 M5 公平性要件との整合性を付録 C.8 に明記。
- [x] 指摘3: PII 流入防止がブラックリスト依存で不十分（§5.3 / §9.1）
  - 修正方針: 原則を **allowlist（許可列のみ受理）** に反転。`date/employee_id/clock_in/clock_out` 以外は exit 2。`--allow-extra-columns` を例外解除フラグとして追加、ただし禁止列（氏名・連絡先・取引先）は常時拒否で多層防御。付録 C.9 を追加。

### Major 指摘
- [x] 指摘4: `--baseline` が CLI にあるが比較エンジン I/F と疑似コードで未実装（§6.1.3 vs §3.2 / §7.5）
  - 修正方針: `compare_rules(..., baseline)` をシグネチャに追加。name 文字列 / 0始まりインデックス両対応、解決不能時 exit 2。§6.1.3 に解決規則を明記。
- [x] 指摘5: 全行スキップ時の終了条件が曖昧（§4.4 / §8.2）
  - 修正方針: 有効行 0 件は exit 2 に統一。`[ERROR] no valid punches after parsing (all rows skipped)` を明記。exit code 表・§8.2 のマッピングも更新。
- [x] 指摘6: 要件トレーサビリティが片方向で SHOULD/MAY の採否状態が追えない（§1.2）
  - 修正方針: §1.2.2 を新設し、要件ID別に **Implemented / Deferred / Rejected** 状態表を追加。SHOULD (S1〜S4) / MAY (W1/W2/W4/W5) / Rejected (永続ログ、`validate` 後送、`--demo` 全展開) を網羅。
- [x] 指摘7: 観測可能性が将来拡張に弱い（§10.1）
  - 修正方針: §10.1.1 を新設し、`run_id / subcommand / rule_name / input_source / processed_count / skipped_count / exit_code` を共通サマリ項目として設計に追加。単一行サマリフォーマットも明記。
- [x] 指摘8: 入力排他の説明が実装手段として不正確（§6.1.2）
  - 修正方針: argparse の `mutually_exclusive_group` に加え、**STDIN が pipe の場合のランタイム検査**（`sys.stdin.isatty()` 判定）を明記。3ソース未指定時の exit 2 も追加。

### Minor 指摘
- [x] 指摘9: `Punch.date` / `employee_id` が `None` を取りうるのに型が `str`（§5.1）
  - 修正方針: `str | None` に修正。
- [x] 指摘10: `--rules P1,P2` はパス中カンマに弱い（§6.1.3）
  - 修正方針: `--rule PATH` の複数回指定方式へ変更。§6.1.3 表と §4.2 記述を更新。
- [x] 指摘11: compare 時の YAML `amount_rounding` 無視が `[INFO]` は弱い（§7.5）
  - 修正方針: `[INFO]` → `[WARN]` に格上げ。
- [x] 指摘12: `--out` 検証の擬似実装が文字列 prefix 依存（§9.2）
  - 修正方針: `Path.relative_to` を用いた実装へ差し替え。

### 良かった点（R2 時点）
- R1 指摘への反映姿勢が明確。`overtime` の「黙殺しない」設計、純関数化、整数演算による `half_up` 境界固定、stdout/stderr 分離は R2 でも妥当と評価。
- 異常系の分類（致命/警告）と exit code 方針が整理されており、R2 での追加（全行スキップ exit 2、24:00 exit 2、baseline exit 2）も同じ枠組みで一貫整理できた。

### このラウンドでの対応
- Critical 3件すべて対応（24:00 境界確定、compare 休憩共通化、PII allowlist 化）。
- Major 5件すべて対応（baseline I/F、全行スキップ exit 2、トレーサビリティ表、共通ログ項目、STDIN ランタイム検査）。
- Minor 4件すべて対応（型アノテーション、`--rule` 多重指定化、`[WARN]` 格上げ、`Path.relative_to` 化）。
- 併せて付録 C.7 / C.8 / C.9 を新設し、R2 新方針の決定記録を追加。

### 見送り事項
- なし（R2 指摘はすべて対応）。ただし markdown lint の軽微な警告（blanks-around-lists 等）は内容に影響しないため残置。Phase4 の整形時に一括処理する。

**B**（Critical 1件が残存 → R3 で対応、最終形は「承認可能」）

要件トレーサビリティと異常系設計は高水準で、残っていた唯一の Critical（`compare` と `simulate` の支払額計算単位の不整合）を閉じた。Phase4（技術仕様書）へ進める状態に到達。

- [x] 指摘2: `--allow-extra-columns` が「実データ投入禁止」の設計担保を弱める（§5.3 / §9.1）

  - 修正方針: `Punch.clock_out_min` を `0..1439` に統一。24:00（1440）は `RoundedPunch` のみ許容と型コメントに明記。
- [x] 指摘5: 想定外例外の「トレース常時出力」はデモ運用で過多（§8.1 / §8.2）
  - 修正方針: デフォルトは簡潔エラーメッセージ、詳細トレースは `--debug` 指定時のみ。共通オプション表（§6.1.1）に `--debug` 追加、§8.2 の想定外例外行を更新。

## 最終残論点

**R3 最終時点で、本設計書に残す実質的論点**は以下のみ。いずれも Phase4（技術仕様書）で解決すべき実装側の課題であり、設計書を承認する上での障害にはならない。

1. **技術仕様書での必須テストケース**: `simulate` / `compare` の支払額集計で「行単位合算」と「合計1回丸め」で結果が異なる最小ケースを `floor` / `half_up` / `ceil` の3方式で網羅する（C.10 の仕様根拠を検証）。
2. **デモビルド定義の具体化**: `DEMO_BUILD` 環境変数 or ビルド時定数の選定、CI 連動ポリシー、サンプル `samples/` の扱いは Phase4 で確定する（C.11 の実装手段）。
3. **markdown lint 整形**: MD032（blanks-around-lists）等の軽微警告は Phase4 整形時に一括処理する（R2 からの継続事項）。
4. **性能要件の測定設計**: 100件/1秒目標のベンチコマンドは実装後に別途整備（R1 指摘13 の継続）。

以上を踏まえ、**本設計書は Phase4 へ進行可能な「承認可能」状態**と判断する。
