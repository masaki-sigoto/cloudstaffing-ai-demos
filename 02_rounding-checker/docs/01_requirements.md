> **プロジェクトコード**: 02_rounding-checker

> **作成日**: 2026-04-23
> **最終更新**: 2026-04-23
> **対象**: クラウドスタッフィング（派遣管理SaaS）セミナー デモ

## 1. プロジェクト概要

クラウドスタッフィング（人材派遣管理SaaS）の**勤怠・請求・給与計算**で避けて通れない**端数処理ルール**（15分切上げ、30分切捨て、1分単位など）を、CS上のルール設定と打刻データから**契約前・月次締め前にシミュレーション・可視化**するCLIツール。

派遣元の請求担当・給与担当が、派遣先企業・派遣スタッフとの間で認識齟齬が起こりやすい「丸め計算」を、**ルールとして透明化**し、請求書・給与明細の金額について「なぜその結果になったのか」を逆算で説明できるようにする。契約登録時の設定レビュー、月次締め前の誤請求防止、派遣先への金額根拠説明の3シーンをカバーする。

セミナー実演では、基準シナリオ「9:03出勤／18:07退勤・時給1,800円・休憩60分」で、丸めルール設定の違いにより**基準シナリオ条件下で1日あたり最大900円／月20日換算で約18,000円**の差が出ることを**再現可能な数字**で提示し、請求・給与の両サイドにおける**説明コスト削減**インパクトを体感させることが目的。

> **注**: 「最大900円／月18,000円」の差額は上記**基準シナリオ条件下での employee_friendly と company_friendly の差額**であり、汎用的な最大値ではない。打刻時刻・時給・休憩条件が変われば差額も変動する。

### プロダクトのコアバリュー
- **KEY MESSAGE**: **"説明コスト削減"** — ロジックが透明になるだけで、現場の会話がラクになる
- **透明性**: 丸め前後の差分と適用ルールを1画面で可視化
- **事前検知**: 本運用前に「設定ミス」「ルール齟齬」を発見できる
- **逆算説明**: 結果から「なぜこうなった？」をルールで再現可能

## 2. 背景と解決する課題

### 2.1 業界の痛点

派遣業界では、勤怠の端数処理が**派遣元・派遣先・スタッフの3者でバラバラ**に運用されることが少なくない。クラウドスタッフィング上では、案件登録時に設定した丸めルールがそのまま月次請求・給与計算の根拠値になるため、設定ミスが金額誤差として直接現れる。

| 登場人物 | よくある言い分 |
|---|---|
| 派遣元 請求担当 | 「派遣先への請求は1分単位と聞いていたが、CS上の案件設定は15分切捨てで登録されていた」 |
| 派遣元 給与担当 | 「先月分、スタッフ給与の丸め方向が逆で差額支払いが発生。原因調査が半日かかった」 |
| 派遣先 勤怠担当 | 「自社の給与計算は30分単位。派遣スタッフだけ1分単位にするのは大変」 |
| 派遣スタッフ | 「9:03に来て18:07に帰ったのに、8時間しかつかないのはなぜ？」 |
| CS導入支援 | 「オンボーディング時、派遣先の勤怠ルールをCSの設定値に落とし込む際の確認観点が属人化している」 |

### 2.2 典型的なトラブル事例

- **請求 vs 給与の不一致**: 派遣先への請求は1分単位、スタッフへの支払いは15分切捨て → 派遣元がサヤ負け／サヤ得で苦情、派遣元の月次締め担当に原因調査の負荷
- **月次締め後の誤請求発覚**: 締め・請求書発行後に「丸め設定が案件契約と逆向き」判明 → 派遣先への請求訂正・謝罪対応
- **派遣先への説明責任**: 「なぜこの案件の今月請求額がこの金額なのか」を口頭で毎回説明、金額根拠が属人化
- **給与ズレによるスタッフ信頼低下**: 明細と実勤務の差をスタッフに説明できず、派遣元の評価・定着率に悪影響
- **法定内外の混同**: 法定内残業は1分単位、法定外は15分切上げ… という複雑ルールが正しく設定されているか確認困難

### 2.3 本ツールが解決すること

1. **設定ルールをコードで明文化**（YAMLでルール定義）
2. **打刻データを入れるだけで支払対象時間をシミュレーション**
3. **丸め前後の差分を可視化**
4. **逆算チェックで「なぜこの結果か」を説明**
5. **ルール比較で「設定違いによる影響金額」を即時提示**

### 2.4 丸め方向の真理値表（仕様の土台）

本ツールでは「丸め方向 × 出退勤」の増減方向を以下のとおり固定定義する。以降の全セクションはこの表に従う。

| 丸め方向 | 出勤（clock_in）に適用 | 退勤（clock_out）に適用 |
|---|---|---|
| `ceil`（切上げ） | 時刻が後ろにずれる → **労働時間 減少方向** | 時刻が後ろにずれる → **労働時間 増加方向** |
| `floor`（切捨て） | 時刻が前にずれる → **労働時間 増加方向** | 時刻が前にずれる → **労働時間 減少方向** |
| `round`（四捨五入, half-up） | 時刻は前後どちらにもずれる → **中立（注1）** | 時刻は前後どちらにもずれる → **中立（注1）** |

組み合わせによる方向性（代表例）:

| 出勤 | 退勤 | 総合方向 | ラベル |
|---|---|---|---|
| `floor` | `ceil` | **労働時間 増加方向** | "スタッフ有利" / employee-friendly |
| `ceil` | `floor` | **労働時間 減少方向** | "会社有利" / company-friendly |
| `ceil` | `ceil` | 出勤は減・退勤は増（ケースにより中立〜やや増） | mixed |
| `floor` | `floor` | 出勤は増・退勤は減（ケースにより中立〜やや減） | mixed |

> 注: 「スタッフ有利/会社有利」は現場で使われる口語を併記したものであり、正式な仕様用語は**「労働時間 増加方向／減少方向」**とする。
>
> 注1（`round=中立`）: 単発の打刻では増加方向・減少方向のどちらにも振れる。「中立」は多数の打刻を長期平均した際に期待値が0に近づくという統計的な意味。個別日次では必ず±の偏りが発生する点に留意。

### 2.5 用語定義（本書内の用語統一）

| 用語 | 定義 |
|---|---|
| **打刻時刻（生データ）** | スタッフが実際に打刻した時刻（丸め前） |
| **丸め後時刻** | 設定ルールに基づき丸められた後の出退勤時刻 |
| **Gross労働時間** | 丸め後の (退勤 − 出勤)。休憩控除**前**。 |
| **支払対象時間（Net労働時間）** | Gross労働時間 − 休憩控除。本書で「実働時間」「労働時間」と書いたときはこれを指す。 |
| **差分** | 丸め前後の分差（±分）。符号は丸め後 − 丸め前。 |

## 4. 機能要件

### 4.1 MUST（デモに必須）

#### M1. ルール設定入力（YAML）
- 単位選択: `1min` / `5min` / `15min` / `30min` / `60min`
- 丸め方向: `ceil`（切上げ）/ `floor`（切捨て）/ `round`（四捨五入, **half-up = 0.5は切上げ**）
- 出勤/退勤で別ルールを許容（例: 出勤は切上げ、退勤は切捨て）
- 休憩時間の自動差引（固定分のみ。打刻ベース休憩はSHOULD）
- **金額丸め（`amount_rounding`）**: 時給×時間が整数円にならない場合の円未満処理を固定。許容値 `floor`（切捨て・デフォルト）/ `half_up`（四捨五入）/ `ceil`（切上げ）。YAMLルート直下に任意キーとして記述可。未指定時は `floor`。
- 設定ファイル例は `samples/rules/` に複数パターン同梱（MUST機能のみを含む基本セット）
- 法定内外の別ルール（`overtime` ブロック）はSHOULDスコープ。MUST版YAMLには含めない。
- **MUSTはYAMLのみ**。JSON入力はMAY（将来拡張）に降格。

#### M2. 打刻パターン入力
- CLI引数直接入力（`--punch 9:03,18:07` 形式。ゼロ埋め `09:03` も許容）
- CSVファイル読み込み（`--punch-file samples/punches.csv`、列: `date,employee_id,clock_in,clock_out`）
- 標準入力（パイプ対応、CSVと同形式。1行目をヘッダとして扱う）
- 複数日まとめて処理可能
- 不正値（退勤<出勤、時刻フォーマット違反、日付不正、必須列欠損）は**1行ごとに警告を出して当該行のみスキップ**。全体実行は停止しない。

#### M3. シミュレーション結果表示
- 打刻時刻（生データ）
- 丸め後時刻
- 実働時間（時:分）
- 休憩控除後の労働時間
- 丸めによる差分（±分）
- 適用ルール名

#### M4. 逆算チェック（**KEY機能**）
- 結果の時刻に対し「なぜこうなったか」をルール単位で説明
- 例: `18:07 → 18:00（ルール: 退勤/15min/floor → 15分単位で切り捨て、7分切り捨て）`
- 計算過程を1行ずつトレース形式で表示
- **警告機能（発火条件は以下のいずれかに該当時、常時表示）**:
  - (a) `clock_in.direction == ceil` かつ `clock_out.direction == floor`（労働時間 減少方向）
  - (b) `clock_in.direction == floor` かつ `clock_out.direction == ceil` かつ `unit_minutes >= 15`（労働時間 増加方向・スタッフ過大計上）
  - 上記に該当しない場合は警告を出さない。曖昧な「強く偏っている」判定は廃止。
- **`--demo` オプション**: 7分デモ枠向けに出力を短縮（7ステップ → 3ステップ: 「丸め結果」「控除」「最終値」のみ）

#### M5. ルール比較モード（**セミナー映え機能**）
- 複数ルール設定を並べて比較（例: `15min_employee_friendly` vs `1min` vs `15min_company_friendly`）
- 時給を与えて**支払額差分**まで計算
- **比較基準**: 全ルールで共通の休憩（固定60分）を控除した**支払対象時間（Net）** で比較する。Grossでの比較は誤解を招くためデフォルト無効。`--show-gross` 指定時のみ追加列として表示。
- **支払額の計算**: `支払額 = round_by(時給 × 支払対象時間(分) / 60, amount_rounding)`。デフォルト `floor` = 円未満切捨て。円未満端数の処理方式を出力脚注に明記する。
- **compare時の amount_rounding 扱い（公平性担保）**: compareでは**全ルールに共通の金額丸め方式**を適用し、ルール差ではなく「丸め方式差」による誤差が混入しないようにする。優先順位は以下:
  1. CLI `--amount-rounding {floor|half_up|ceil}` 指定時はそれを全ルールに適用
  2. 未指定時はデフォルト `floor` を全ルールに適用
  3. 各ルールYAMLの `amount_rounding` 値は**compare時のみ無効化**し、不一致があれば stderr に「compareでは共通丸めを適用（YAML値は無視）」の警告を1回表示
- テーブル形式で視覚的に提示

### 4.2 SHOULD（あれば嬉しい）

- S1. 法定内/法定外残業の別ルール適用（8時間超分だけ別丸め）※ `overtime` ブロックを持つYAMLは `samples/rules/advanced/` に分離
- S2. 深夜帯（22時以降）の自動検知と割増対応
- S3. 月次集計（打刻を日別→月計で合算し、丸め誤差の累積を可視化）
- S4. 打刻ベース休憩（休憩打刻を差引）
- S5. `samples/punches.csv` に典型パターン10件程度を同梱（定時、早出、残業、深夜、またぎ）

### 4.3 MAY（将来拡張）

- W1. Web UI化（Streamlit など）
- W2. 既存勤怠システムCSVフォーマット取り込み（KING OF TIME等）
- W3. 派遣契約書PDFからルール自動抽出（LLM活用）
- W4. ルール変更時の**影響額シミュレーション**（過去3ヶ月分を遡って再計算）
- W5. 派遣元・派遣先で別ルール設定し、請求額・給与額の**両方**を同時出力

| **免責表示（MUST）** | `--format text`（デフォルト）の**標準出力先頭**に `※本ツールは設定ルール通り計算するだけの補助ツールです。36協定・同一労働同一賃金・抵触日等の法令適合判定は行いません。` を固定表示。`--format json` / `--format csv` の機械可読出力時は**標準出力には免責文を出さず stderr に1行出力**（自動 `--quiet` 相当）。`--quiet` 明示時は text/json/csv いずれも抑制可 |
| **色付き出力の互換性** | ANSIエスケープは TTY 出力時のみ有効。非TTY（パイプ/リダイレクト）では自動で色無効。`--no-color` で明示無効化も可能 |

### 5.1 データ取扱最低ライン（デモ用途でも遵守）

- **実データ禁止**: 実在スタッフ氏名・社員番号・派遣先社名をサンプルに使用しない。必ず匿名ID（`EMP001` 等）・架空企業名のみ。
- **匿名ID必須**: CSVの `employee_id` 列は匿名ID限定。氏名列は非対応（誤って実名を入れられない設計）。
- **出力保存先（ツール内蔵 `--out` 限定）**: ツール内蔵の `--out <path>` オプションで指定する保存先は、**正規化後の絶対パスが `cwd` 配下**であることを検証し、範囲外なら exit code 2 で拒否する。シェルリダイレクト（`>` / `>>` / tee など）はOSレベルの機能であり**CLI側で制御不可能**なため本制約の対象外とする（運用上の注意事項として本書に明記）。
- **ログ保持**: 本ツールは入力打刻データを永続ログに残さない（標準エラー出力の警告のみ。デフォルト `--log-file` なし）。
- **スクリーンショット**: セミナー時は本書 §10 の例示データ・匿名IDのみ画面表示する。

## 6. デモシナリオ（セミナー実演の流れ・所要7分）

> **基準シナリオ**: 派遣元 請求担当が、**月次締め直前**にクラウドスタッフィング上の案件（派遣先：架空のOFFICE-A社、スタッフ：EMP001）について、丸めルール設定が契約どおりか確認する状況を想定。打刻 `9:03 / 18:07`・時給 `1,800円`・休憩固定 `60分`。
> 全ステップ・全数値はこの前提で再現可能。

### ステップ0: 導入トーク（30秒）
> 「皆さん、派遣スタッフが9:03に打刻して18:07に退勤した。派遣先への請求額と、このスタッフへの給与額はいくらになるでしょう？ CS上の丸め設定次第で、実は月約18,000円ブレます」
> → 会場から「9時間」「8時間半」など多様な回答

### ステップ1: 素の打刻で実行（1分）
```bash
$ python src/main.py simulate --punch 9:03,18:07 --rule samples/rules/1min.yml
```
> Gross: **9時間04分**（1分単位） / 支払対象時間（Net, 休憩60分控除後）: **8時間04分**

### ステップ2: 「スタッフ有利」ルールに変更（1分）
```bash
$ python src/main.py simulate --punch 9:03,18:07 --rule samples/rules/15min_employee_friendly.yml
```
> Gross: **9時間15分**（出勤floor＋退勤ceil = **労働時間 増加方向**） / Net: **8時間15分**

### ステップ3: ルール比較で衝撃演出（2分）
```bash
$ python src/main.py compare --punch 9:03,18:07 --hourly 1800 --break 60 \
    --rules samples/rules/1min.yml,samples/rules/15min_employee_friendly.yml,samples/rules/15min_company_friendly.yml
```
> 出力例（テーブル、支払対象時間=Net基準。支払額は `amount_rounding: floor` ＝円未満切捨て）:

| ルール | 丸め後出勤 | 丸め後退勤 | 支払対象時間 | 支払額(時給1,800円) | 基準との差 |
|---|---|---|---|---|---|
| 1分単位（フェア） | 9:03 | 18:07 | 8:04 | 14,520円 | 基準 |
| 15min employee_friendly（増加方向） | 9:00 | 18:15 | 8:15 | 14,850円 | **+330円** |
| 15min company_friendly（減少方向） | 9:15 | 18:00 | 7:45 | 13,950円 | **-570円** |

> 「employee_friendly と company_friendly の差は、**この基準シナリオ条件下で1日あたり最大900円**。20日換算で **約18,000円** 違います」

## 7. 前提・制約

### 前提
- セミナー実演用モックであり、**本番データ接続・認証は想定外**
- 日本の勤怠慣習（分単位、日本時刻JST）に特化
- 打刻データはHH:MM形式の文字列で与えられる（ゼロ埋め `09:03` も許容）
- ルール定義はYAML管理者が記述可能（非エンジニアは `samples/` から選ぶ）

### 7.1 入力仕様詳細

| 項目 | 仕様 |
|---|---|
| 時刻フォーマット | `H:MM` または `HH:MM`（ゼロ埋め可）。例: `9:03` / `09:03` はいずれも有効 |
| 秒の扱い | **MUSTでは `HH:MM:SS` 形式を不許可**とし、当該行スキップ＋stderr警告（§7.2）。丸め計算は分単位に統一し、`round/half-up` の挙動と入力粒度の乖離を排除する。秒対応はSHOULD扱い（将来拡張時は丸め計算を秒まで含めて再定義） |
| 日付フォーマット | ISO8601 `YYYY-MM-DD` のみ（CSV `date` 列） |
| 文字エンコード | UTF-8（BOM有無どちらも可） |
| 標準入力 | CSVと同形式（1行目ヘッダ）。パイプ経由で渡す |
| CSV必須列 | `date`, `employee_id`, `clock_in`, `clock_out` の4列（順不同可、列名一致で判定） |
| `employee_id` | 匿名ID限定。許容パターン `^[A-Z]{2,4}\d{3,6}$`（例: `EMP001`, `STAFF0001`）。違反時は当該行スキップ＋stderr警告（§7.2準拠）。氏名の直接記入は禁止（§5.1） |
| `--break` と `break.minutes` の優先順位 | **CLI `--break` 指定時はそちらが優先**。未指定時のみYAML `break.minutes` を使用。両方無指定なら `0` 分（休憩控除なし） |

### 7.2 異常系の最小定義

| 異常 | 挙動 |
|---|---|
| 時刻フォーマット違反（例: `25:77`, `abc`） | 当該行スキップ、stderrに警告（行番号・内容） |
| `HH:MM:SS`（秒付き）入力 | 当該行スキップ、stderrに警告（MUSTでは秒入力非対応、§7.1準拠） |
| `clock_out <= clock_in`（同時刻含む） | 当該行スキップ、警告。MUSTでは日またぎ判定を行わず、単純に不正として扱う（日またぎ対応はSHOULD） |
| `clock_in` または `clock_out` が空欄 | 当該行スキップ、警告 |
| `date` フォーマット不正 | 当該行スキップ、警告 |
| `employee_id` 匿名IDパターン違反 | 当該行スキップ、警告（§7.1許容パターン準拠） |
| CSV必須列欠損 | 実行中断、stderrにエラーメッセージを出し exit code 2 |
| YAMLパースエラー | 実行中断、exit code 2 |
| 不明な `direction`（ceil/floor/round 以外） | 実行中断、exit code 2 |
| 不明な `amount_rounding`（floor/half_up/ceil 以外） | 実行中断、exit code 2 |

### 制約
- 日またぎシフト（深夜0時超え）は**SHOULD相当**とし、MUSTは当日内完結シフトのみ
- 複数回休憩（午前・午後で別休憩）はSHOULD扱い。MUSTは固定分（例: 60分）の一括控除のみ
- 有給・遅刻早退扱いは本ツールの対象外
- 労基法の法解釈は扱わない（あくまで設定ルールどおり計算するだけ）

## 8. 成功基準

### 定量
- セミナー実演が**技術トラブルなし**で完遂できる
- 打刻100件を**1秒以内**で処理（条件: Apple M1以上、CSV入力、`simulate` 単体モード）
- デモシナリオ（§6）を**7分±1分**で回せる（`--demo` 短縮モード利用を前提）

### 定性
- 会場参加者から「うちもこれで揉めてる」という反応が出る（アンケートで自社該当と回答した人が30%以上）
- ステップ3の比較テーブル（**基準シナリオ条件下での日次最大差額900円／月約18,000円**）に対し、アンケートで「差額インパクトが理解できた」と回答した人が70%以上
- 逆算チェックで「説明可能性が向上する」と回答した人が60%以上
- セミナー後のアンケートで「端数処理チェッカー」への興味が3項目中2位以内

## 9. スコープ外

- 実在の派遣会社・派遣先の勤怠データ連携
- 認証・ログイン・マルチテナント管理
- 法改正への自動追随
- 給与計算ソフト・勤怠管理システムへの書き戻し
- 打刻機デバイスとの連携
- 労基署への提出書類生成
- 税金・社会保険料計算
- スマートフォン向けUI
- **36協定の上限時間判定**（月45h / 年360h 等の閾値チェックは本ツール非対象）
- **同一労働同一賃金の判定**（派遣元と派遣先の処遇比較は対象外）
- **派遣抵触日の管理**（3年ルール等の期間管理は対象外）
- 法定内外・深夜割増のMUST対応（これらはSHOULD。MUSTは単純な丸めと固定休憩控除のみ）

### 10.1 ルール定義ファイル例（YAML）

> MUSTスコープのサンプルは `overtime` ブロックを含めない。法定内外別ルール（SHOULD）は `samples/rules/advanced/` に分離。

**samples/rules/15min_company_friendly.yml**（労働時間 減少方向 / 通称「会社有利」）
```yaml
name: "15分丸め（減少方向 / 会社有利）"
description: "出勤ceil（切上げ）×退勤floor（切捨て）。労働時間が減少方向に偏る。"
unit_minutes: 15
clock_in:
  direction: ceil  # 切上げ → 労働時間 減少方向
clock_out:
  direction: floor  # 切捨て → 労働時間 減少方向
break:
  type: fixed
  minutes: 60  # 1時間休憩を自動控除
amount_rounding: floor  # 円未満切捨て（デフォルト）
```

**samples/rules/15min_employee_friendly.yml**（労働時間 増加方向 / 通称「スタッフ有利」）
```yaml
name: "15分丸め（増加方向 / スタッフ有利）"
description: "出勤floor（切捨て）×退勤ceil（切上げ）。労働時間が増加方向に偏る。"
unit_minutes: 15
clock_in:
  direction: floor  # 切捨て → 労働時間 増加方向
clock_out:
  direction: ceil  # 切上げ → 労働時間 増加方向
break:
  type: fixed
  minutes: 60
```

**samples/rules/1min.yml**（フェア）
```yaml
name: "1分単位（フェア）"
description: "丸めなし相当。端数処理の基準比較用。"
unit_minutes: 1
clock_in:
  direction: round  # half-up（0.5は切上げ）。1分単位なら実質変化なし
clock_out:
  direction: round
break:
  type: fixed
  minutes: 60
```

**samples/rules/advanced/15min_with_overtime.yml**（SHOULD対応・参考）
```yaml
# SHOULDスコープ。MUSTデモでは使用しない。

description: "法定内は切捨て、法定外（8h超）は切上げ"
unit_minutes: 15
clock_in:
  direction: ceil
clock_out:
  direction: floor

  type: fixed
  minutes: 60

  legal_hours_per_day: 8
  inside_legal:
    unit_minutes: 15
    direction: floor
  outside_legal:
    unit_minutes: 15
```
